### 1. 3.1 如何利用 知识图谱（KG）进行上下文增强？

- 主标签：RAG与向量检索
- 来源条数：2
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 检索增强生成(RAG) 优化策略篇 / 未知](https://articles.zsxq.com/id_gu4p7gszsh82.html)

### 基础知识补充

- 实体抽取与链接：将用户查询映射到图谱实体节点
- 子图检索：通过多跳查询获取实体间的关系路径
- 图文融合：将结构化三元组转化为自然语言补充上下文

### 详细解答

利用知识图谱（GraphRAG）可以有效弥补纯向量检索在复杂逻辑推理和全局信息感知上的不足。结论上，KG通过提供结构化关系路径，显著增强了LLM的推理能力。原理上，首先利用NER模型或LLM从Query中提取实体，并在图谱中进行实体链接；接着通过Cypher等查询语言或图神经网络（GNN）召回相关的多跳子图；最后将子图的三元组信息序列化为文本，与向量检索的文档片段拼接后输入大模型。对比纯向量检索，KG能准确回答“A的子公司的创始人是谁”这类多跳问题。工程权衡上，图谱构建成本极高，且实体对齐困难，通常建议采用轻量级图谱与向量数据库混合的架构。

### 案例模拟

业务案例模拟：在医疗问诊系统中，患者提问“吃阿司匹林能喝西柚汁吗？”。纯向量检索可能只召回两种药物的说明书，而GraphRAG通过知识图谱直接检索到“阿司匹林-[相互作用]-西柚汁-[导致]-出血风险增加”的子图路径。系统将该三元组转化为提示词，使大模型生成极其准确的禁忌警告。

### 2. 1. 为什么 需要 FLARE ？

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型外挂知识库优化——如何利用大模型辅助召回？ / 未知](https://articles.zsxq.com/id_oznm6qixjw61.html)

### 基础知识补充

- 解决传统单次检索无法应对复杂长文本生成的局限
- 基于模型生成置信度动态触发检索的前向主动策略
- 缓解大模型在长程推理和多步生成中的幻觉累积问题

### 详细解答

FLARE（Forward-Looking Active REtrieval）是一种主动式、前瞻性的检索增强生成框架。结论是：它解决了传统 RAG “一次检索、一次生成”在面对复杂长篇问答时的信息匮乏和幻觉问题。原理上，传统 RAG 仅在生成前根据 Query 检索一次，如果生成过程较长，后续内容很容易脱离检索到的知识而产生幻觉。FLARE 提出在生成过程中持续监控大模型的输出置信度（如 Token 的概率分布）。当模型对即将生成的词汇置信度较低时，说明其内部知识不足，此时 FLARE 会主动暂停生成，利用已生成的上下文或预测的未来句子作为 Query 去触发新的检索，获取外部知识后再继续生成。这种动态权衡极大提升了长文本生成的准确性。

### 案例模拟

面试官追问：“FLARE 怎么判断什么时候该触发检索？” 回答：“FLARE 主要通过监控生成 Token 的概率值来实现。如果当前生成的句子中某些关键 Token 的概率低于预设的阈值，系统就会认为模型处于‘不确定’状态，从而触发检索。在工程实现上，为了避免频繁检索带来的巨大延迟，我们通常会设置一个滑动窗口或按句子粒度进行置信度评估，平衡生成质量与响应速度。”

### 3. 1. 介绍一下 HYDE 思路？

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型外挂知识库优化——如何利用大模型辅助召回？ / 未知](https://articles.zsxq.com/id_oznm6qixjw61.html)

### 基础知识补充

- 基于大语言模型生成假设性文档的检索增强方法
- 将查询向量空间转化为文档向量空间的对齐策略
- 利用无监督对比学习模型进行稠密向量编码与检索

### 详细解答

HyDE（Hypothetical Document Embeddings）是一种创新的零样本检索增强思路。结论是：它通过“先生成后检索”的方式，巧妙解决了 Query 和 Document 之间的语义空间不一致问题。原理上，传统的稠密检索直接计算简短 Query 和长 Document 的向量相似度，两者往往存在特征鸿沟。HyDE 首先让大模型根据用户的 Query 生成一篇“假设性文档”（即伪答案），这篇文档虽然可能包含事实错误（幻觉），但其文本模式、词汇分布和语义结构与目标真实文档高度相似。随后，系统对这篇假设性文档进行向量化，并在知识库中检索最相似的真实文档。工程权衡上，HyDE 极大提升了零样本场景下的召回率，但牺牲了推理速度。

### 案例模拟

面试官追问：“HyDE 生成的假设性文档如果有严重幻觉，会影响检索吗？” 回答：“通常不会产生致命影响。因为 HyDE 的核心假设是：大模型生成的伪答案在‘语义模式’和‘关键词分布’上更接近真实文档。即使包含错误事实，其底层的上下文特征依然能帮助向量检索器在向量空间中定位到相关领域的高质量文档。不过，如果幻觉偏离主题太远，确实会导致召回噪音。”

### 4. 1.1 RAG 工作流程

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 检索增强生成(RAG) 优化策略篇 / 未知](https://articles.zsxq.com/id_gu4p7gszsh82.html)

### 基础知识补充

- 数据准备：文档解析、文本分块（Chunking）、向量化。
- 检索阶段：用户Query向量化，在向量库中计算相似度召回。
- 生成阶段：组装Prompt，将查询与召回上下文送入大模型。

### 详细解答

RAG的标准工作流程可以清晰地划分为离线数据处理和在线检索生成两个阶段。结论是，每一个环节的输出质量都直接决定了下游环节的上限。在离线阶段，首先需要对多格式文档（PDF、Word等）进行解析和清洗，接着采用合适的策略将长文本切分为固定长度的Chunk，并利用Embedding模型将其转化为稠密向量，存入向量数据库（如Milvus或FAISS）。在在线阶段，用户的Query首先被同一个Embedding模型向量化，系统在向量库中执行近似最近邻（ANN）搜索，召回最相关的Top-K个Chunk。最后，系统将这些Chunk作为背景知识，结合用户的原始Query填入预设的Prompt模板中，交由大语言模型进行推理，生成最终的自然语言回答。

### 案例模拟

面试官追问：“在文本分块（Chunking）环节，如果Chunk设置得过大或过小，会有什么影响？” 回答：“如果Chunk过小，虽然检索的精准度可能提升，但会丢失重要的上下文语境，导致大模型生成时缺乏足够信息；如果Chunk过大，不仅会引入大量无关的噪声信息，降低检索的信噪比，还可能超出大模型的上下文窗口限制，增加Token消耗和推理延迟。工程上通常采用固定大小加重叠（Overlap）的策略，或基于语义段落进行切分。”

### 5. 1.1 为什么 大模型 需要 外挂(向量)知识库？

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 基于LLM+向量库的文档对话 经验面 / 未知](https://articles.zsxq.com/id_xk58m8ok2sob.html)

### 基础知识补充

- 知识时效性：大模型参数冻结，无法直接获取训练之后的最新资讯。
- 领域专业性：通用模型缺乏企业内部私有数据和垂直领域的深度知识。
- 幻觉问题：外挂知识库提供事实依据，有效降低模型生成虚假信息的概率。

### 详细解答

结论：大模型需要外挂向量知识库主要是为了解决知识更新滞后、缺乏私有领域数据以及容易产生幻觉这三大痛点。 原理：大模型的知识内化在神经网络的权重中，更新成本极高（需重新微调或预训练）。通过外挂知识库（RAG架构），系统可以在生成回答前，先将用户问题向量化，从知识库中检索出最相关的文档片段，然后将这些片段作为上下文与问题一起输入给大模型。 工程权衡：相比于全量微调（Fine-tuning），外挂知识库具有成本低、知识可实时更新、数据权限易于控制等优势。但它也引入了检索链路的延迟，且最终回答的质量高度依赖于文档切分策略和向量检索的准确率。

### 案例模拟

面试官追问：什么时候应该选择微调，什么时候选择外挂知识库？ 回答：在业务落地中，如果是为了让模型掌握特定的语言风格、输出格式或特定的推理逻辑，我会选择微调；如果是为了让模型回答基于最新业务文档、企业规章制度等事实性问题，我会优先选择外挂知识库（RAG）。实际项目中往往是两者结合，先通过微调优化模型的指令遵循能力，再通过RAG注入动态的私有知识。

### 6. 1.1 介绍一下 内容缺失问题？

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / RAG 关键痛点及对应解决方案 / 未知](https://articles.zsxq.com/id_1bmbedojsj0t.html)

### 基础知识补充

- 知识库盲区：底层文档库未包含回答问题所需的信息
- 幻觉诱导：大模型在缺乏上下文时强行编造错误答案
- 边界判定失效：系统未能识别出“无法回答”的场景

### 详细解答

内容缺失（Missing Content）是RAG系统中最基础的失败模式之一。结论上，当知识库本身不包含用户提问的答案时，RAG系统不仅无法提供帮助，还极易诱发大模型的幻觉。原理上，RAG的假设是“答案存在于检索到的文档中”，如果底层语料库更新不及时、覆盖面不足，或者用户提出了超纲问题，检索模块召回的将是无关文档。此时，如果Prompt中没有严格的约束，大模型会利用其预训练权重中的过时或错误知识强行生成回答。对比其他检索错误，内容缺失是数据源层面的硬伤。工程权衡上，扩充知识库会增加存储和检索成本，而让模型学会“拒绝回答”则可能降低用户体验，需要根据业务容错率进行调整。

### 案例模拟

业务案例模拟：在企业内部规章制度问答系统中，员工询问“2024年新增的陪产假有几天？”。由于知识库仅更新到2023年，系统召回了旧版休假制度。大模型基于旧文档或自身幻觉回答了“3天”，导致员工获取错误信息。这说明在内容缺失时，系统缺乏有效的数据时效性管理和拒答机制。

### 7. 1.1.1 Annoy 介绍

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 向量检索常见面试篇 / 未知](https://articles.zsxq.com/id_dnq0o4aicjso.html)

### 基础知识补充

- 核心原理：基于随机投影树构建的近似最近邻搜索算法。
- 内存优化：支持将索引文件映射到内存，实现多进程共享。
- 适用场景：适合高维稀疏向量检索，常用于推荐系统召回。

### 详细解答

Annoy（Approximate Nearest Neighbors Oh Yeah）是Spotify开源的一种高效的近似最近邻（ANN）搜索算法库。其核心原理是通过构建多棵随机投影树（Random Projection Trees）来划分向量空间。在构建阶段，算法随机选择两个点并用超平面将空间一分为二，递归进行直到子空间节点数满足阈值。查询时，通过遍历这些树快速定位到目标向量所在的叶子节点，从而大幅缩小搜索范围。Annoy的工程优势在于其极小的内存占用和便捷的部署能力。它支持将构建好的索引保存为静态文件，并通过mmap（内存映射）技术加载，这意味着多个进程可以共享同一份索引内存，非常适合在内存受限的生产环境中进行分布式部署。不过，Annoy不支持动态增删向量，且在超大规模稠密向量检索上，召回率可能略逊于HNSW。

### 案例模拟

面试官追问：Annoy在构建索引后不支持动态更新，在实际业务中如何解决增量向量的问题？ 回答：在实际推荐系统或RAG业务中，由于Annoy不支持动态增删，我们通常采用“全量+增量”的双路检索架构。全量数据每天定时离线构建Annoy索引并推送到线上；对于实时新增的少量向量，则存放在内存中的精确检索库（如Faiss的Flat模式）或支持动态更新的HNSW库中。查询时合并两路结果。

### 8. 1.2 如何 解决 内容缺失问题？

- 来源条数：1

### 基础知识补充

- 知识库扩充与动态更新：接入实时API或增量同步数据
- 拒答机制设计：在Prompt中强制要求无信息时回复不知道
- 意图识别与兜底策略：通过分类器拦截超纲的闲聊查询

### 详细解答

解决内容缺失问题需要从数据源保障和模型行为约束两方面入手。结论上，既要尽可能丰富知识库，又要赋予系统“知之为知之，不知为不知”的能力。原理上，首先建立自动化的数据管道，定期爬取或同步最新业务文档，确保知识库的时效性和覆盖率；其次，在Prompt工程中加入严格指令，例如“仅根据提供的上下文回答，若上下文中没有答案，请直接回复‘知识库暂无相关信息’”，以此抑制幻觉；最后，可引入NLI（自然语言推理）模型，在生成后校验答案是否蕴含于参考文档中。工程权衡上，严格的拒答机制会降低系统的召回率（Recall），导致部分本可回答的问题被拦截，需通过A/B测试找到最佳的Prompt阈值。

### 案例模拟

面试官追问：“如果Prompt约束不住大模型，它还是会产生幻觉怎么办？”回答示例：“我会引入后置校验模块（Post-Retrieval Verification）。在LLM生成答案后，使用一个轻量级的判别模型（如基于DeBERTa微调的蕴含分析模型），计算答案与检索文档的逻辑一致性得分。如果得分低于阈值，则强制将输出替换为标准兜底话术。”

### 9. 1.2. 基于LLM+向量库的文档对话 思路是怎么样？

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 基于LLM+向量库的文档对话 经验面 / 未知](https://articles.zsxq.com/id_xk58m8ok2sob.html)

### 基础知识补充

- 数据处理：文档解析、文本分块（Chunking）与向量化嵌入。
- 检索阶段：用户提问向量化，计算余弦相似度召回Top-K文档。
- 生成阶段：将召回文档作为上下文拼接进Prompt，由LLM生成回答。

### 详细解答

结论：基于LLM+向量库的文档对话（RAG）主要分为离线数据构建和在线检索生成两个核心阶段。 原理：离线阶段，首先将各类格式的文档（PDF、Word等）解析为纯文本，按照一定规则（如按段落或固定长度）切分为文本块（Chunk），然后利用Embedding模型将文本块转化为高维向量，并存入向量数据库。在线阶段，当用户发起提问时，系统使用相同的Embedding模型将问题向量化，在向量库中进行相似度检索，召回最相关的几个文本块。最后，将这些文本块与用户问题组装成特定的Prompt模板，交由大模型进行阅读理解并生成最终答案。 工程权衡：该思路的难点在于文本切分粒度的把握。切分过大可能包含过多噪声，导致LLM抓不到重点；切分过小则可能丢失上下文语义。通常需要结合文档结构进行语义切分。

### 案例模拟

面试官追问：如果用户的问题需要跨越多个文档的综合信息才能回答，目前的思路有什么缺陷？ 回答：基础的RAG思路往往只能召回局部片段，难以处理需要全局总结或跨文档推理的问题。在项目中，我引入了多路召回和重排（Rerank）机制，并结合了知识图谱（Graph RAG）来补充实体间的关联信息。对于全局总结类问题，还可以采用Map-Reduce策略，先对各个文档块分别生成摘要，再进行最终汇总。

### 10. 1.3. 基于LLM+向量库的文档对话 核心技术是什么？

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 基于LLM+向量库的文档对话 经验面 / 未知](https://articles.zsxq.com/id_xk58m8ok2sob.html)

### 基础知识补充

- 文本切分技术：基于规则或语义的文本切分策略，保留上下文连贯性。
- 向量嵌入模型：将文本映射为高维稠密向量，捕捉深层语义特征。

### 20. 2.2.2 如何通过后检索处理提升检索结果？

- 来源：[LLMs_interview_notes / RAG（Retrieval-Augmented Generation）面 / 未知](https://articles.zsxq.com/id_xk58m8ok2sob.html)

### 基础知识补充

- 采用交叉编码器进行重排以提升相关文档的排序精度。
- 通过文档压缩技术去除冗余信息以节省大模型上下文窗口。
- 利用多样性过滤策略减少重复片段并提升信息覆盖面。

### 详细解答

后检索处理（Post-Retrieval Processing）是连接检索器和生成器的关键桥梁，其结论是：通过重排、压缩和过滤，可以显著提升输入给大模型的上下文质量，降低噪声并节省 Token 成本。 原理上，初步检索（如向量召回）通常追求高召回率，但返回的文档可能包含大量冗余或弱相关内容。工程上，第一步是重排（Reranking），使用 Cross-Encoder 模型对 Query 和 Document 进行深度交互打分，将最相关的文档排在前面。第二步是上下文压缩（Context Compression），利用小模型或规则提取文档中的关键句子，剔除无关段落，从而在有限的上下文窗口内塞入更多有效信息。第三步是去重与多样性控制，避免输入多个内容高度同质化的文本块。这种权衡在于，虽然增加了后处理的计算延迟，但大幅降低了 LLM 的推理成本并减少了幻觉产生的概率。

### 案例模拟

面试官追问：“重排（Rerank）模型和双塔召回模型有什么区别？为什么不直接用重排做检索？” 回答：“双塔模型（Bi-Encoder）将 Query 和 Document 分别独立编码成向量，计算余弦相似度，速度极快，适合在海量库中做粗筛召回。而重排模型（Cross-Encoder）将 Query 和 Document 拼接后一起输入模型进行深度自注意力交互，准确率高但计算复杂度呈平方级增长。如果直接用重排模型对百万级文档进行打分，延迟是不可接受的。因此工程上采用‘双塔召回+交叉重排’的级联架构来平衡效率与精度。”

### 详细解答

“脱离上下文/整合策略限制”（Not in Context / Integration Limits）是指即使正确文档被召回并输入给LLM，模型依然无法生成正确答案的现象。结论上，这暴露了大模型在长文本理解和多文档信息聚合方面的能力瓶颈。原理上，最典型的是“Lost in the Middle”现象，即大模型对输入文本的开头和结尾注意力集中，但极易忽略中间部分的信息。此外，当检索召回的Top-K文档中夹杂大量噪声（无关片段）时，模型的推理链路会被干扰。面对需要跨文档总结的复杂问题，模型往往只能提取局部信息，无法完成全局整合。工程权衡上，虽然可以通过扩大模型的上下文窗口（如支持128K）来输入更多文档，但这不仅成本高昂，且窗口越大，信息提取的信噪比往往越低。

### 案例模拟

业务案例模拟：在金融研报分析系统中，用户提问“对比A公司和B公司2023年的净利润率”。系统成功召回了包含A公司数据的文档（排第1）和B公司数据的文档（排第15）。由于合并后的上下文长达8000 Token，B公司的数据处于文本中部。大模型在生成时出现了“迷失在中间”效应，仅回答了A公司的数据，忽略了B公司。

### 基础知识补充

- MinHash通过多个哈希函数的最小哈希值生成文档签名。
- 两个文档MinHash签名相等的概率等于其Jaccard相似度。
- LSH通过将签名分段映射到桶中实现亚线性时间的检索。

### 详细解答

结论：在海量预训练数据去重中，MinHash结合LSH（局部敏感哈希）是解决大规模文档模糊去重的工业级标准方案。原理：直接计算两两文档的Jaccard相似度时间复杂度为$O(N^2)$，在海量数据下不可接受。MinHash通过将文档转化为N-gram集合，并应用多个独立的哈希函数求取最小值，构建出固定长度的签名向量。数学上可以证明，两个集合MinHash值相同的概率严格等于它们的Jaccard相似度。为了进一步加速，LSH将签名向量划分为多个Band，只要两个文档在任意一个Band上的哈希值完全相同，就被映射到同一个桶中作为候选对。对比与工程权衡：这种方法将全局对比转化为局部桶内对比，极大降低了计算量。通过调整Band的数量和大小，可以灵活权衡去重的召回率与精确率，是典型的用空间换时间、用近似解换取极致效率的工程设计。

### 案例模拟

面试官追问：“LSH中Band的数量（b）和每个Band的行数（r）如何影响去重效果？”回答：“b和r的乘积等于MinHash签名的总长度。如果增大b（即Band变多，每个Band变短），文档在某个Band上碰撞的概率就会增加，这会导致召回率上升，但可能会引入更多假阳性，增加后续精确计算的开销；反之，如果增大r（Band变少但变长），碰撞条件变得苛刻，精确率提升但可能漏掉一些相似文档。实际工程中需要根据业务容忍度通过公式 $1 - (1 - s^r)^b$ 绘制S型曲线来选择最优参数。”

- nn.Embedding：充当查找表，将离散的整型索引映射为稠密向量。
- One-hot等价性：Embedding在数学上等价于One-hot向量乘权重。

### 详细解答

结论：nn.Linear用于连续特征空间的线性映射，而nn.Embedding专门用于将离散的类别或词汇索引转换为连续的稠密向量表示。 原理与对比：nn.Linear接受浮点型张量，通过矩阵乘法计算输出，适用于全连接层。nn.Embedding接受整型索引，直接从权重矩阵中查表提取对应的行向量，不涉及乘法运算。虽然将离散索引转为One-hot向量后再通过nn.Linear也能得到相同结果，但nn.Embedding直接查表避免了构建稀疏One-hot矩阵的巨大内存开销和无效的零乘法计算。 工程权衡：在NLP或推荐系统中，词表或特征基数极大，使用nn.Embedding是唯一可行的方案。此外，nn.Embedding支持sparse=True参数，在更新极大规模词表时可使用稀疏梯度更新，进一步节省显存和加速训练。

### 案例模拟

面试官追问：在推荐系统中，如果用户ID的数量达到上亿级别，直接使用nn.Embedding会有什么问题？如何解决？ 回答示例：上亿级别的ID会导致Embedding表极其庞大，单卡显存无法容纳（OOM）。在工程实践中，通常有几种解决方案：一是使用Hash Trick将ID映射到较小的固定空间，但这会引入哈希冲突；二是采用分布式Embedding，将参数切分到多台机器的内存或显存中；三是使用多重哈希或特征交叉组合来替代单一的庞大ID表，从而大幅降低参数量。

- Patch Embedding是Vision Transformer(ViT)处理图像的核心步骤。
- 使用步长等于卷积核大小的二维卷积可高效实现图像分块。
- 卷积输出需经过展平(flatten)和转置(transpose)以适配Transformer。

### 详细解答

在PyTorch中实现图像到Patch Embedding的过程，结论是利用二维卷积（nn.Conv2d）是最高效且优雅的工程实现方式。原理上，ViT需要将输入图像划分为不重叠的Patch，并将其映射为固定维度的向量序列。通过设置卷积核大小（kernel_size）和步长（stride）均等于Patch的大小（如16），卷积操作能够一次性完成图像的无重叠切分与线性投影，输出形状为(B, embed_dim, H/P, W/P)。随后，通过flatten(2, 3)将空间维度合并，再通过transpose(1, 2)调整维度顺序，最终得到形状为(B, num_patches, embed_dim)的序列数据，完美契合Transformer的输入要求。对比手动切片再进行线性映射，卷积实现不仅代码更简洁，还能充分利用底层cuDNN的加速优化，显著提升计算效率。

### 案例模拟

面试官追问：为什么在展平之后需要进行transpose(1, 2)操作？ 回答：卷积层的输出格式是(Batch, Channels, Height, Width)，经过flatten(2, 3)后，形状变为(Batch, embed_dim, num_patches)。然而，标准的Transformer编码器期望的输入序列格式通常是(Batch, Sequence_Length, Embedding_Dimension)（在batch_first=True时）。因此，必须使用transpose(1, 2)将特征维度和序列长度维度进行交换，使得每个Patch的特征向量处于最后一个维度，从而满足自注意力机制的计算要求。

Graph RAG 是一种将知识图谱（Knowledge Graph）与检索增强生成（RAG）深度融合的先进架构。它的核心思想是用图结构来组织和检索外部知识，而非传统的纯向量数据库。具体而言，它首先利用大模型或NLP工具从海量文档中抽取实体（如人名、机构）和关系，构建出结构化的知识图谱；在检索阶段，当用户提出问题时，系统会识别问题中的关键实体，并在图谱中进行子图匹配或多跳遍历，召回相关的关系链路和节点属性；最后，将这些结构化的图谱知识转化为自然语言上下文，与用户问题一起输入给大模型生成答案。相比纯向量RAG，它具备更强的逻辑推理能力和全局信息感知度。

- 核心流程包含文档切分、向量化、检索匹配与提示词构建。
- 有效缓解大模型幻觉问题并支持私有知识库的动态更新。

### 详细解答

RAG（Retrieval-Augmented Generation）即检索增强生成，是一种结合外部知识库检索与大语言模型生成能力的技术架构。其核心结论是：通过在生成回答前引入相关外部知识，RAG 能够显著降低大模型的幻觉，并提供时效性强、来源可溯的回答。 原理上，RAG 系统通常分为索引、检索和生成三个阶段。首先将私有文档切分为文本块，通过嵌入模型转化为向量并存入向量数据库。当用户发起查询时，系统将查询转化为向量，在数据库中检索出最相似的文本块。最后，将这些检索到的上下文与用户查询拼接成提示词，输入给大语言模型生成最终回答。相比于微调，RAG 的工程权衡在于无需重新训练模型即可实现知识更新，成本更低且灵活性更高，但受限于检索质量和上下文窗口长度。

### 59. 五、说一下 RAG-Fusion 工作流程？

- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）RAG 优化策略 —— RAG-Fusion篇 / 未知](https://articles.zsxq.com/id_4ce04xwvic1z.html)

### 基础知识补充

- 接收用户原始查询，调用大模型生成多个视角的衍生查询语句。
- 将所有查询并行输入向量数据库，获取多组Top-K文档列表。
- 应用RRF算法对多组文档进行融合重排，截取最终上下文生成。

### 详细解答

RAG-Fusion的工作流程是一个典型的“发散-收敛-生成”的漏斗结构。第一步是“发散”，系统接收到用户的原始Query后，通过Prompt引导大模型生成N个（通常为3-5个）语义相关但表述不同的衍生查询。第二步是“检索”，将原始查询和衍生查询并发地送入向量数据库，各自召回Top-K个相关文档块，形成多组检索结果。第三步是“收敛”，系统采用倒数秩融合（RRF）算法对这些文档列表进行交叉合并与重新打分，剔除重复项，并根据RRF得分选出最终的全局Top-N文档。最后一步是“生成”，将经过融合重排的高质量文档作为上下文，与原始问题一起拼接成最终的Prompt，交由大模型进行阅读理解并生成最终答案。在工程实现上，这一流程通过并发处理有效控制了延迟。

### 案例模拟

面试官追问：“RAG-Fusion的流程中，生成多个查询会增加耗时，如何优化？” 回答：“在工程实践中，我们主要通过两种方式优化耗时：一是采用较小且速度极快的模型（如专门微调过的开源小模型）来专门执行Query改写任务，而不是依赖GPT-4级别的大模型；二是将多路检索设计为严格的异步并发执行。此外，如果业务场景对延迟要求极高，我们会限制衍生查询的数量（例如最多生成2个），在召回率和响应时间之间取得最佳平衡。”

结论：PDF解析面临的核心问题集中在版面还原困难、特殊元素提取失真以及非标准文档的鲁棒性差。 原理与工程权衡：首先是版面理解问题，PDF缺乏结构标签，多栏排版、图文混排、跨页段落极易导致提取文本时语义断裂。其次是表格与公式提取，无边框表格的行列对齐非常困难，数学公式在常规OCR下常被识别为乱码，导致RAG系统丢失关键逻辑。最后是文档质量问题，扫描件倾斜、模糊、水印干扰会大幅降低OCR准确率。在工程实践中，解决这些问题往往需要堆叠多个深度学习模型（如版面检测、表格结构识别、公式识别），这不仅增加了系统的复杂度和维护成本，也使得解析耗时成倍增加，难以满足实时性要求。

### 65. 四、说一下 RAG-Fusion 核心技术？

- 主标签：RAG与向量检索
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）RAG 优化策略 —— RAG-Fusion篇 / 未知](https://articles.zsxq.com/id_4ce04xwvic1z.html)

### 基础知识补充

- 查询生成与扩展：利用大模型将原始问题改写为多个相似查询。
- 多路并发检索：基于生成的多个查询在向量库中分别进行检索。
- 倒数秩融合算法：通过RRF对多路检索结果进行交叉打分重排。

### 详细解答

RAG-Fusion的核心技术主要由查询扩展（Query Generation）和倒数秩融合（Reciprocal Rank Fusion, RRF）两部分构成。首先，系统利用大模型强大的指令遵循能力，针对用户的原始输入生成多个具有不同表达方式或侧重点的衍生查询。接着，这些查询会并行地输入到向量数据库中进行多路检索，获取多组候选文档。最关键的步骤是RRF算法，它不需要依赖额外的相关性分数，而是纯粹基于文档在不同检索列表中的排名位置进行打分（公式通常为 $Score = \sum 1 / (k + Rank)$，k为平滑常数）。对比传统的单一打分机制，RRF能够有效平衡不同查询带来的偏差，将那些在多个查询中都排名靠前的文档置顶。这种工程设计既巧妙又高效，无需训练专门的重排模型即可实现高质量的文档整合。

### 案例模拟

面试官追问：“RRF算法中的常数k有什么作用？一般怎么设置？” 回答：“在RRF公式中，常数k（通常默认设为60）主要用于平滑排名权重，防止排名极度靠前（如第一名）的文档获得过大的绝对优势。如果k设置得太小，排名第一的文档得分会远超第二名，导致融合结果被某一路检索严重主导；如果k较大，则能更好地综合多路检索的整体表现。在实际工程调优中，我们会根据召回文档的多样性需求对k值进行微调测试。”
