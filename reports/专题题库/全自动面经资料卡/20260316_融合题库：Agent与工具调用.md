### 1. 2.1 LangChain 中 Components and Chains 是什么？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）langchain 面 / 未知](https://articles.zsxq.com/id_ve2dgaiqrjzv.html)

### 基础知识补充

- Components是执行单一特定任务的模块化基础构建块。
- Chains是将多个组件按特定顺序组合而成的执行流水线。
- LLMChain是最基础的链，组合了提示词模板与语言模型。

### 详细解答

在 LangChain 中，Components（组件）和 Chains（链）是构建复杂应用的基础架构。结论是：组件是提供单一功能的积木，而链是将积木拼装成有业务价值的工作流。 Components 包含了 LLM 包装器、Prompt 模板、输出解析器等，它们被设计为高度模块化和可复用的单元。然而，单独的组件往往无法完成复杂的业务逻辑。Chains 的作用就是将这些组件串联起来。例如，最常用的 LLMChain 接收用户输入，将其格式化为 Prompt，传递给 LLM，然后解析输出。更复杂的链（如 SequentialChain）可以将前一个链的输出作为后一个链的输入。工程权衡上，使用 Chains 可以极大地提高代码的可读性和复用性，但在处理高度动态的逻辑时，过于复杂的嵌套链可能会导致状态管理困难，此时通常需要转向 Agent 架构。

### 案例模拟

项目案例模拟：在开发一个自动撰写技术博客的工具时，我们使用了 SequentialChain。第一步是一个 LLMChain，根据用户提供的主题生成文章大纲；第二步是另一个 LLMChain，它接收第一步生成的大纲作为输入，逐个章节生成详细内容；第三步使用一个语法检查的 Chain 对全文进行润色。通过将大任务拆解为多个串联的 Chain，显著提升了最终生成文章的质量和逻辑连贯性。

### 2. 2.1 介绍一下 规划（planning）？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 任务分解：将复杂宏观目标拆解为多个可执行的子任务。
- 路径规划：确定子任务的执行顺序与依赖关系。
- 自我反省：在执行过程中评估结果并动态调整后续计划。

### 详细解答

规划（Planning）是Agent处理复杂任务的核心能力，主要包含任务分解和自我反省两个维度。结论上，规划使Agent从单步问答进化为多步问题解决者。原理方面，任务分解通常借助思维链或思维树等提示工程技术，引导大模型将宏大目标拆解为逻辑连贯的子任务序列。自我反省则是让Agent在执行子任务后，评估当前结果是否符合预期，若发现错误或偏离目标，则动态调整后续规划。在工程权衡中，过于精细的规划会增加推理延迟和Token消耗，而过于粗略则容易导致执行失败。因此，实际应用中常采用启发式规则或预设标准作业程序来辅助LLM进行高效规划。

### 案例模拟

面试官追问：“如果Agent在规划时陷入死循环怎么办？” 回答：“这在ReAct框架中很常见。我们通常采取三种策略：一是设置最大迭代步数强制熔断；二是引入外部监督机制，当检测到重复执行相同动作时，注入特定的纠偏Prompt；三是优化自我反省模块，让Agent在每次行动前显式评估当前状态与历史动作的差异，从而主动跳出循环。”

- 提示工程引导：通过思维链或特定指令让模型逐步思考。

- 人机协同输入：在关键节点允许人类介入确认或修改子任务。

### 详细解答

拆解子目标和任务分解主要通过精心设计的提示工程和框架引导大模型完成。结论上，有效的拆解需要结合任务上下文和模型的推理能力。原理上，最基础的方法是使用逐步思考等简单指令触发模型的思维链。对于更复杂的场景，可以采用少样本提示，提供几个成功的任务分解示例供模型学习。在工程实践中，常将任务分解抽象为一个独立的LLM调用节点，输入总目标和可用工具列表，输出结构化的子任务数组及依赖关系。为了提高稳定性，还可以引入领域特定的标准作业程序，将大模型的自由发挥限制在预设的业务流程框架内。权衡来看，动态拆解灵活性高但易出错，静态流程稳定性好但泛化性差，两者结合是当前主流。

### 案例模拟

业务案例模拟：“在开发一个数据分析Agent时，用户输入‘分析上季度销售下滑原因’。Agent首先通过任务分解模块，将目标拆解为：1.调用SQL工具获取上季度各品类销量；2.调用API获取市场大盘数据；3.对比分析并生成图表；4.撰写总结报告。通过将拆解结果结构化输出，系统能按顺序调度不同工具，确保复杂任务有序执行。”

### 详细解答

任务分解的方法从简单的线性提示到复杂的图结构搜索不断演进。结论上，选择哪种方法取决于任务的复杂度和对实时性的要求。最基础的是思维链，通过线性步骤引导模型拆解，适合逻辑直接的任务。进阶方法如思维树或思维图，允许模型在拆解时探索多个分支，并在每步进行评估和回溯，适用于需要多重试错的复杂推理。此外，还有基于特定框架的方法，如利用模型描述作为上下文进行任务分配；或者结合传统AI的符号规划器，LLM负责将自然语言转化为规划语言，由外部求解器生成严格的执行步骤。工程上，思维树等方法会成倍增加Token消耗，需在效果与成本间权衡。

### 5. 2.1.2.1 如何进行 模型自我反省？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 结果评估：对比执行结果与预期目标，判断是否成功。
- 错误归因：分析失败原因，如工具参数错误或逻辑漏洞。
- 策略调整：基于错误分析生成新的执行计划或修正动作。

### 详细解答

模型自我反省是通过让Agent审视自身过去的行动和结果，以优化未来决策的过程。结论上，它是提升Agent在复杂环境中鲁棒性的关键。原理上，自我反省通常作为一个独立的评估节点存在。当Agent执行完一个子任务后，系统会将当前目标、采取的行动和环境反馈输入给LLM，要求其判断行动是否有效。如果无效，LLM需要生成一段反思文本，指出错误原因，并据此更新下一步的规划。在工程实现中，反省机制可以显著降低幻觉和工具调用失败率，但代价是增加了交互轮次和延迟。因此，通常只在关键节点或捕获到明确异常时才触发反省机制。

### 案例模拟

业务案例模拟：“在代码生成Agent中，如果生成的代码在沙箱中运行报错，Agent会触发自我反省。它会将报错堆栈信息和原始代码作为输入，分析出‘变量未初始化导致空指针异常’。随后，Agent在记忆中记录这一反思，并在下一次迭代中修正代码。这种闭环反省机制使代码通过率从百分之四十提升到了百分之七十五。”

### 6. 2.1.2.2 模型自我反省 有哪些方法？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 反应框架：交替进行推理与行动，实时根据环境反馈调整。
- 语言强化：通过语言反馈强化学习，维护长期反思记忆。
- 经验回溯：利用历史成败经验指导当前决策，优化输出。

### 详细解答

模型自我反省的方法主要围绕如何获取反馈并将其转化为改进策略。结论上，不同方法在反馈粒度和记忆持久性上有所区别。最经典的是反应框架，它在每步行动后直接观察结果并进行下一步推理，属于即时、局部的反省。进阶框架则更进一步，它不仅在任务失败后生成语言反馈，还会将这些反思存储在长期记忆中，使得Agent在未来遇到类似任务时能避免重复犯错。此外，还有自我优化方法，主要用于文本或代码生成，模型通过多轮自我批评和修改来提升输出质量。工程权衡上，维护复杂的记忆结构适合长周期任务；而简单反应框架实现轻量，更适合短平快的工具调用场景。

### 案例模拟

面试官追问：“基于语言反馈的反思和传统的强化学习有什么区别？” 回答：“传统强化学习依赖标量奖励信号来更新模型权重，这在自然语言任务中很难设计且训练成本高。基于语言的反思可以看作是一种轻量级的强化学习，它不更新模型权重，而是将反思文本作为上下文提示输入给模型。这种方式可解释性强，且非常适合利用大模型强大的上下文学习能力。”

### 7. 2.2 LangChain 中 Prompt Templates and Values 是什么？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）langchain 面 / 未知](https://articles.zsxq.com/id_ve2dgaiqrjzv.html)

### 基础知识补充

- Prompt Templates是生成动态提示词的可复用参数化模板。
- Prompt Values是模板填充后传递给模型的最终实例化对象。
- 支持Few-Shot模板以注入示例提升模型特定任务表现。

### 详细解答

Prompt Templates（提示词模板）和 Prompt Values（提示词值）是 LangChain 中管理模型输入的标准化机制。结论是：模板负责定义输入的结构与变量，而值是变量被具体数据填充后的最终产物。 Prompt Templates 允许开发者预先定义一段包含占位符的文本（如“请将以下{language}翻译成中文：{text}”）。在运行时，传入具体的变量字典，模板就会将其渲染为 Prompt Values。Prompt Values 是一个抽象接口，它可以被转换为纯文本（供普通 LLM 使用）或消息列表（供 Chat Model 使用，如包含 SystemMessage、HumanMessage）。工程权衡上，将 Prompt 模板化不仅实现了提示词与业务代码的解耦，方便统一管理和版本控制，还能轻松集成 Few-Shot（少样本）示例，显著提升模型在特定格式输出或垂直领域任务中的稳定性和准确率。

### 案例模拟

面试官追问：“在处理 Chat Model 时，Prompt Template 有什么特殊之处？” 回答：“对于传统的 LLM，Prompt Template 输出的是单一的字符串。但对于 Chat Model（如 GPT-4），输入是结构化的消息序列。因此 LangChain 提供了 ChatPromptTemplate，它允许我们分别定义 System Message（设定角色和规则）、Human Message（用户输入）和 AI Message（历史回复）的模板。这种结构化的模板能更好地利用 Chat Model 的指令遵循能力。”

### 8. 2.2 介绍一下 记忆（Memory）？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 短期记忆：存储当前对话上下文，受限于模型上下文窗口。
- 长期记忆：利用向量数据库持久化存储海量历史信息。
- 记忆检索：通过相似度计算将相关记忆召回至当前上下文中。

### 详细解答

记忆模块赋予了Agent在时间跨度上保持状态和积累经验的能力。结论上，记忆机制解决了大模型无状态和上下文窗口受限的问题。原理上，记忆分为短期和长期。短期记忆通常指当前任务的上下文或多轮对话历史，直接受限于模型的上下文窗口大小。长期记忆则将历史对话、外部文档或反思经验转化为向量，存储在向量数据库中。当Agent需要决策时，通过语义检索召回相关记忆，作为提示词的一部分输入给模型。工程实践中，记忆的难点在于遗忘与更新机制，即如何清理冗余信息、解决记忆冲突，以保证检索的准确性和效率。

### 案例模拟

业务案例模拟：“在开发一个个人助理Agent时，用户今天说‘我喜欢喝拿铁’，系统将其存入长期记忆。一个月后用户说‘帮我点杯咖啡’，Agent通过向量检索召回了‘喜欢拿铁’的记忆，并自动推荐了拿铁。为了防止记忆膨胀，我们还引入了记忆衰减和摘要机制，定期将零散的短期对话压缩为结构化的用户画像标签。”

### 9. 2.3 LangChain 中 Example Selectors 是什么？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：近似题匹配：DeepLearing-Interview-Awesome-2024_手撕代码专题:CodeAnything/Reference.md
- 来源：[LLMs_interview_notes / 大模型（LLMs）langchain 面 / 未知](https://articles.zsxq.com/id_ve2dgaiqrjzv.html)

### 基础知识补充

- Example Selectors用于在少样本提示中动态选择最相关的示例。
- 可基于文本相似度、长度或最大边际相关性进行示例筛选。
- 核心目的是在上下文窗口限制内最大化提示词的指导效果。

### 详细解答

Example Selectors（示例选择器）是 LangChain 中用于优化 Few-Shot Prompting（少样本提示）的高级组件。结论是：它通过动态筛选最合适的示例注入到提示词中，以提升模型表现并节省 Token 成本。 在构建复杂应用时，我们可能拥有一个包含数百个示例的庞大样本库。由于 LLM 的上下文窗口有限且 Token 计费，将所有示例都放入 Prompt 是不现实的。Example Selectors 能够根据用户的当前输入，动态挑选出最相关的几个示例。常见的选择策略包括：基于向量数据库的语义相似度选择（SemanticSimilarityExampleSelector）、基于输入长度的动态截断选择（LengthBasedExampleSelector）以及兼顾相关性与多样性的最大边际相关性选择（MMR）。工程权衡上，动态选择示例能显著提高模型在特定任务上的泛化能力，但引入向量检索也会增加系统的延迟和架构复杂度。

### 案例模拟

业务案例模拟：在开发 Text-to-SQL（自然语言转SQL）应用时，我们积累了上千条“自然语言-SQL”的映射对。当用户提问时，我们使用 SemanticSimilarityExampleSelector。它会先将用户的自然语言问题向量化，去向量库中检索出最相似的 3 个历史问答对，然后将这 3 个示例作为 Few-Shot 注入到 Prompt 中。这使得大模型能参考类似表结构的查询逻辑，大幅降低了 SQL 生成的语法错误和幻觉。

### 10. 2.3 介绍一下 工具使用（tool use）？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 能力扩展：弥补大模型在实时信息和精确计算上的不足。
- 接口定义：通过标准化格式描述工具功能与参数要求。
- 函数调用：模型输出结构化参数，由系统执行并返回结果。

### 详细解答

工具使用是Agent与外部物理或数字世界交互的桥梁。结论上，它极大地拓展了大模型的能力边界，使其从文本生成器变为任务执行者。原理上，开发者首先需要以标准化格式向模型描述可用工具的名称、功能和参数要求。当用户输入复杂请求时，模型通过推理决定调用哪个工具，并生成符合格式的结构化参数。系统拦截这些参数，在本地或云端执行实际的接口调用，然后将执行结果作为观察返回给模型，供其生成最终回答。工程权衡中，工具数量过多会导致模型选择困难和幻觉，通常需要通过检索增强技术动态召回相关工具，或对模型进行专门的工具调用微调。

### 案例模拟

面试官追问：“在实际业务中，这四个模块哪个最容易成为性能瓶颈？” 回答：“通常是规划和工具使用的结合处。当任务复杂时，LLM可能会规划出错误的工具调用顺序，或者生成不符合API格式的参数。我们在项目中通过引入中间层的校验机制和Few-shot Prompting来规范工具调用，同时利用短期记忆缓存中间结果，显著降低了API调用失败率。”

### 详细解答

结论：Function Call微调的核心难点在于数据构建的复杂性、格式输出的稳定性以及模型通用能力的保持。 原理与对比：首先，高质量的Function Call数据极难获取，需要覆盖单步调用、多步调用、参数缺失追问以及无需调用等多种复杂场景，通常需要借助强模型（如GPT-4）进行合成并辅以人工校验。其次，LLM本质上是概率生成模型，极易在生成JSON参数时遗漏引号、括号或捏造不存在的参数，导致解析失败。最后，过度微调Function Call任务会导致模型在日常闲聊或逻辑推理上的能力退化（灾难性遗忘）。 工程权衡：为了解决这些难点，工程上通常采用混合训练策略（如保持30%的通用数据），并在推理端引入基于有限状态机（FSM）的结构化解码技术（如Outlines库），强制模型输出100%合法的JSON格式，从而降低对模型原生格式化能力的苛刻要求。

### 详细解答

开源模型实现Function Call主要有三种方案。第一种是指令微调（Fine-tuning），代表如Gorilla、ToolLLaMA，通过构建大量“指令-API调用”的问答对，对开源基座模型进行微调，使其原生具备输出结构化JSON的能力，准确率高但训练成本大。第二种是提示词工程（Prompt Engineering），如ReAct框架，通过在System Prompt中详细描述工具格式，引导模型按特定模板（如Thought/Action/Observation）输出，无需训练但占用上下文且稳定性较差。第三种是推理期约束（Constrained Decoding），如使用Guidance或Outlines库，在生成阶段基于正则表达式或有限状态机（FSM）强制模型只能输出合法的JSON格式，这种方案工程落地性强，能有效避免语法错误。
