### 1. 2.1 LangChain 中 Components and Chains 是什么？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）langchain 面 / 未知](https://articles.zsxq.com/id_ve2dgaiqrjzv.html)

### 基础知识补充

- Components 是构建大语言模型应用的基础模块化组件。
- Chains 是将多个组件按特定顺序组合执行的端到端工作流。
- 常见组件包括提示词模板、语言模型接口和输出解析器等。

### 详细解答

在 LangChain 中，Components（组件）和 Chains（链）是构建 LLM 应用的核心抽象。结论上，Components 是提供单一功能的积木，而 Chains 是将积木拼接以完成特定任务的流水线。原理上，Components 涵盖与大模型交互的独立环节，如 Models（封装 LLM API）、Prompts（格式化提示词）和 Memory（上下文记忆），它们高度解耦且可复用。Chains 则是对组件的编排，例如基础的 LLMChain 将 PromptTemplate、LLM 和 OutputParser 串联，实现“输入变量->生成提示词->模型推理->解析输出”的自动化流程。工程权衡方面，直接调用 Components 适合简单任务或极致自定义，但代码冗余；使用 Chains 能大幅降低复杂任务的开发门槛。当前，传统 Chains 正逐渐被更灵活的 LCEL 替代，以更好地支持动态路由和流式输出。

### 案例模拟

面试官追问：“在实际项目中，如果现有的 Chains 无法满足复杂的业务逻辑，你会怎么做？” 回答示例：“如果内置的 Chains（如 RetrievalQAChain）不够灵活，我通常有两种方案。第一种是继承 Chain 基类自定义，重写 _call 方法来硬编码业务逻辑。第二种，也是目前推荐的做法，是使用 LCEL（LangChain表达式语言）。通过管道符将各个 Components 组合起来，不仅代码更简洁，还能原生支持异步调用、流式输出和 Fallback 机制，极大提升了工程鲁棒性。”

### 2. 2.1 介绍一下 规划（planning）？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 任务分解将宏大目标拆解为可执行的原子级子任务。
- 逻辑推理通过思维链等方式推导任务执行的先后顺序。
- 动态调整根据环境反馈实时修改后续的执行计划。

### 详细解答

结论：规划是Agent将复杂目标转化为有序执行步骤的核心能力，包含任务分解和动态调整。原理：面对用户提出的复杂需求，Agent首先需要进行宏观规划。它利用LLM的推理能力，将大目标拆解为多个相互关联的子目标。在执行过程中，规划不仅是静态的步骤列表，还包含动态的自我反思。当某个子任务执行失败或获得新信息时，Agent能够评估当前状态，调整后续计划，甚至重新规划路径。工程权衡：强大的规划能力依赖于LLM的参数规模和推理深度。使用如Tree of Thoughts等复杂规划算法能显著提升任务成功率，但会成倍增加Token消耗和推理延迟。在工程实践中，通常对简单任务使用单步规划，对复杂任务才启用多步树状规划。

### 案例模拟

面试官追问：如果Agent在规划时陷入了死循环，一直重复相同的错误步骤，如何解决？回答：这种“规划死循环”在Agent开发中很常见。我们的解决方案是引入外部干预和状态监控机制。首先，在Prompt中强制要求Agent输出反思日志，对比当前状态与历史状态；其次，在代码层设置最大迭代次数和重复动作检测。一旦检测到连续执行相同动作且结果未变，系统会强制中断当前规划，注入错误提示，迫使Agent尝试备用路径或直接请求人类协助。

### 3. 2.1.1.1 如何进行 拆解子目标和任务分解？

### 基础知识补充

- 提示词工程通过特定指令引导模型输出分步计划。
- 领域知识注入利用专家经验指导特定场景的任务拆解。
- 依赖关系图构建明确各个子任务之间的先后执行顺序。

### 详细解答

结论：任务分解主要通过精心设计的Prompt引导LLM，结合领域知识将复杂任务转化为有向无环图或线性步骤。原理：进行任务分解时，通常向LLM提供包含“思考过程”的系统提示词。例如，要求模型以“第一步、第二步”的格式输出，或者生成结构化的JSON格式计划。高级的分解方法会要求模型不仅列出任务，还要明确任务间的输入输出依赖关系，形成执行图。此外，对于特定垂直领域，可以向模型注入标准工作流（SOP）作为上下文，指导其进行更专业的拆解。工程权衡：细粒度的拆解有助于提高单步执行的成功率，但会导致整体流程过长，增加累积误差和延迟。粗粒度的拆解虽然高效，但对单步执行工具的能力要求极高。因此，拆解粒度需要根据底层LLM的能力和可用工具的复杂度进行动态调整。

### 案例模拟

面试官追问：如何评估Agent生成的任务分解计划是否合理？回答：在业务中，我们通常采用“双模型校验”机制。主Agent生成任务分解计划后，我们会调用另一个轻量级评估模型（或基于规则的校验器）来检查该计划。评估标准包括：步骤是否完整、依赖关系是否冲突、是否调用了不存在的工具等。如果校验不通过，会将错误信息反馈给主Agent要求重写。这种机制虽然增加了少量延迟，但大幅提升了复杂任务的执行成功率。

### 4. 2.1.1.2 拆解子目标和任务分解 有哪些方法？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 思维链引导大语言模型逐步写出中间的逻辑推理过程。
- 思维树在每一步探索多个分支并进行启发式路径评估。
- 任务特定提示基于标准操作流程进行结构化任务拆解。

### 详细解答

结论：常见的任务分解方法包括思维链（CoT）、思维树（ToT）、Plan-and-Solve以及基于SOP的结构化拆解。原理：1. CoT：最基础的方法，通过“Let's think step by step”促使模型线性拆解任务。2. ToT：将拆解过程视为树状搜索，在每一步生成多个可能的子任务分支，并评估其可行性，选择最优路径。3. Plan-and-Solve：先让模型生成完整的计划列表，然后严格按照列表逐一执行。4. 基于SOP的拆解：在Prompt中提供标准操作流程，模型只需将当前任务映射到SOP的各个环节中。工程权衡：CoT实现简单、延迟低，适合常规任务；ToT逻辑严密、成功率高，但需要多次模型调用，成本和延迟极高，仅适用于高价值的复杂推理任务。实际工程中常采用Plan-and-Solve结合动态修正的折中方案。

### 案例模拟

面试官追问：在实际项目中，你会优先选择哪种任务分解方法？回答：我会根据任务的确定性来选择。对于确定性高、流程固定的业务（如数据报表生成），我会优先使用基于SOP的拆解，通过Few-shot Prompt让模型直接输出结构化步骤，稳定且高效。对于开放性强、探索性高的任务（如复杂Bug排查），我会采用Plan-and-Solve框架，先生成初步计划，但在执行每一步后加入状态评估，允许模型动态修改后续计划，以兼顾效率和灵活性。

### 5. 2.1.2.1 如何进行 模型自我反省？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 状态对比分析当前执行结果与预期目标之间的差异。
- 错误归因通过分析报错信息找出执行失败的根本原因。
- 策略修正基于反思结果生成新的执行路径或调用参数。

### 详细解答

结论：模型自我反省通过将执行结果作为反馈输入给LLM，促使其评估偏差并调整后续策略。原理：自我反省是Agent实现闭环控制的关键。当Agent执行一个动作（如调用API）后，系统会捕获该动作的输出（如成功数据、报错信息或环境状态变化）。将这些反馈信息与原始目标一起重新构建为Prompt，要求LLM回答三个问题：1. 当前结果是否满足预期？2. 如果失败，原因是什么？3. 下一步应该如何调整？模型基于这些反思生成新的动作指令。工程权衡：自我反省显著提升了Agent的鲁棒性和容错率，使其能从错误中恢复。但频繁的反思会消耗大量Token，并可能导致上下文窗口溢出。因此，工程上通常只在关键节点或动作执行失败时才触发深度反省机制。

### 案例模拟

面试官追问：如果API返回的错误信息非常长且包含大量无用堆栈，模型反省时抓不住重点怎么办？回答：这是一个典型的工程痛点。直接将冗长的报错喂给LLM不仅浪费Token，还容易引发注意力分散。我们的做法是在反馈给LLM之前，先经过一层轻量级的“信息清洗”。使用正则表达式提取核心错误码和关键异常信息，或者使用一个较小的模型专门做错误摘要。只有经过提炼的、高信噪比的错误信息才会进入主Agent的反思Prompt中，从而提高反省的准确率。

### 6. 2.1.2.2 模型自我反省 有哪些方法？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- ReAct框架交替进行推理思考与动作执行以实现反思。
- Reflexion通过维护反思日志来指导未来的决策过程。
- 批评者模型引入独立大模型专门负责评估和指出错误。

### 详细解答

结论：主流的自我反省方法包括ReAct、Reflexion以及引入外部Critic模型的评估机制。原理：1. ReAct：将推理和执行交织，模型在每次行动前先输出一段思考，评估上一步结果并决定下一步，是最基础的反思形式。2. Reflexion：在ReAct基础上引入了长期记忆，模型在任务失败后会生成一段文字反思，并将其存储起来，在下次尝试时作为上下文提示，避免重复犯错。3. Critic机制：使用一个独立的、专门微调过的LLM作为“批评者”，专门审查主Agent的输出和计划，指出逻辑漏洞或工具使用错误，主Agent根据批评意见进行修改。工程权衡：ReAct轻量且易于实现，是目前工业界的标配。Reflexion适合需要反复试错的复杂环境，但记忆管理复杂。Critic机制效果最好，能有效缓解主模型的“盲目自信”，但会增加一倍的推理成本。

### 案例模拟

面试官追问：在多Agent系统中，如何利用反省机制提升整体表现？回答：在多Agent协作场景中，我们会将反省机制转化为“辩论与同行评审”。例如在代码生成任务中，Coder Agent负责写代码，Tester Agent负责运行测试并生成报错，Reviewer Agent负责分析报错并给出修改建议（即反省过程）。这种职责分离不仅降低了单个模型的认知负担，还能通过多视角的交叉验证，显著提升最终代码的质量和系统的稳定性。

### 7. 2.2 LangChain 中 Prompt Templates and Values 是什么？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）langchain 面 / 未知](https://articles.zsxq.com/id_ve2dgaiqrjzv.html)

### 基础知识补充

- Prompt Template 是用于动态生成提示词的参数化模板类。
- Prompt Value 是模板实例化后的对象，可转为字符串或消息。
- 包含 FewShot 等模板类型，满足复杂工程提示词构建需求。

### 详细解答

在 LangChain 中，Prompt Templates（提示词模板）和 Prompt Values（提示词值）是构建和管理大模型输入的两个核心概念。结论是：Templates 是定义提示词结构的“模具”，而 Values 是注入具体数据后生成的“成品”，负责适配不同类型的大模型接口。 原理与对比：Prompt Template 允许开发者通过占位符定义提示词的骨架。它不仅支持简单的字符串替换，还支持针对聊天模型的结构化消息模板（ChatPromptTemplate），如区分 System、Human 等角色。当向 Template 传入具体参数后，就会生成一个 Prompt Value。Prompt Value 的核心作用是解耦：因为不同大模型 API 接收的输入格式不同（纯文本或消息列表），Prompt Value 提供统一接口（如 to_string 和 to_messages），在底层自动完成格式转换。 工程权衡：硬编码提示词会导致代码难以维护。使用 Templates 可实现提示词的版本控制、复用和动态组装，极大地提升了应用的可扩展性和跨模型兼容性。

### 案例模拟

面试官追问：在构建 RAG（检索增强生成）应用时，你会如何使用 Prompt Templates？ 回答示例：在 RAG 项目中，我会使用 ChatPromptTemplate 来构建输入。首先定义 System Message 模板规定 AI 人设；然后定义 Human Message 模板，包含 {context}（检索文档）和 {question}（用户问题）占位符。运行时，检索器返回的文档动态填充到 {context}，生成最终的 Prompt Value 传给 LLM，既保证结构清晰，又能缓解提示词注入风险。

### 8. 2.2 介绍一下 记忆（Memory）？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 短期记忆通常指当前对话轮次中的上下文历史信息。
- 长期记忆利用向量数据库存储海量历史交互与知识。
- 记忆检索通过相似度计算提取与当前任务相关的经验。

### 详细解答

结论：Agent的记忆模块分为短期记忆和长期记忆，用于维持上下文连贯性和积累历史经验。原理：短期记忆受限于LLM的上下文窗口大小，通常通过滑动窗口或摘要机制保存当前任务的近期交互历史，确保Agent理解当前语境。长期记忆则突破了窗口限制，将历史对话、成功经验或外部知识库转化为向量表示，存储在向量数据库中。当Agent面临新任务时，通过语义检索召回相关的长期记忆，作为增强上下文注入到Prompt中。工程权衡：短期记忆响应快但容量有限；长期记忆容量无限但存在检索延迟和召回准确率问题。优秀的Agent系统需要设计高效的记忆读写策略，如定期将短期记忆总结归档为长期记忆，并结合关键词和向量混合检索提升召回质量。

### 案例模拟

面试官追问：随着交互增加，短期记忆越来越长，快要超出模型的Context Window时怎么处理？回答：在工程实践中，我们通常采用“分层压缩”策略。首先，保留最近N轮的完整对话以维持即时语境；对于更早的对话，调用LLM生成精简的摘要替换原文；如果摘要依然过长，则将最旧的摘要转化为向量存入长期记忆库，并从上下文中移除。这样既保证了Token消耗的可控，又不会丢失关键的历史信息。

### 详细解答

在 LangChain 中，Indexes（索引）和 Retrievers（检索器）是构建 RAG（检索增强生成）系统的两大核心组件，分别负责数据的“存”与“取”。 Indexes 是一个数据处理管道，包含文档加载（Document Loaders）、文本切分（Text Splitters）、向量化（Embeddings）以及存储到向量数据库（VectorStores）的过程，目的是将非结构化数据转化为大模型可高效检索的结构化格式。Retrievers 则是独立于存储的检索接口，它接收用户的自然语言查询，并返回最相关的文档列表。 在工程实践中，单纯的向量相似度往往不够。因此 LangChain 提供了高级检索器，如 MultiQueryRetriever（多查询扩展提高召回率）、ContextualCompressionRetriever（上下文压缩减少冗余 token 消耗）以及 EnsembleRetriever（混合检索）。实际开发中，需根据延迟要求、成本预算和召回率目标，权衡选择合适的索引策略与高级检索器组合。

结论：开源模型的Function Call方案主要分为提示词工程、模型微调和推理时约束三种路径。 原理与对比：1. 提示词工程（如ReAct）：通过System Prompt详细描述工具格式，依赖模型自身的涌现能力，实现简单但稳定性较差。2. 模型微调（如ToolLLaMA、Qwen-Agent）：使用大量工具调用轨迹数据对开源模型进行SFT，使其原生支持特定格式的函数调用，准确率高但训练成本大。3. 推理时约束（如Guidance、Outlines）：在解码阶段通过正则表达式或有限状态机强制模型输出合法的JSON参数，能彻底解决格式错误问题，但可能影响推理速度。工程上常结合微调与推理约束。

### 案例模拟

面试官追问：你们项目中选择了哪种方案？为什么？ 回答：我们采用了模型微调结合推理时约束的混合方案。业务场景对工具调用的准确率要求极高，单纯的提示词工程经常出现JSON格式错误或参数遗漏。我们先用开源的Qwen模型，因为它本身经过了工具调用微调，然后在推理端接入了Outlines库，强制约束输出符合JSON Schema，彻底杜绝了解析失败的问题。
