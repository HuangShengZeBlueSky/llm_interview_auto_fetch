## 大模型（LLMs）agent 面

### 1. 一、什么是 大模型（LLMs）agent？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- Agent是以大语言模型为大脑的智能体系统。
- 核心组件包括规划、记忆、工具调用和执行模块。
- 能够自主感知环境、拆解任务并执行复杂操作。

### 详细解答

结论：大模型 Agent（智能体）是一种以大型语言模型（LLM）为核心计算引擎，具备自主规划、记忆管理和工具调用能力的系统，旨在完成复杂的、多步骤的现实世界任务。 原理：传统 LLM 只是被动回答问题的文本生成器，而 Agent 赋予了模型“行动力”。其架构通常包含四个核心模块：1. 规划（Planning）：将复杂目标拆解为可执行的子任务，并具备自我反思（Reflection）能力以纠正错误；2. 记忆（Memory）：包括短期记忆（上下文）和长期记忆（向量数据库），用于维持多轮交互的连贯性；3. 工具（Tools）：允许模型调用外部 API（如搜索、计算器、代码解释器）以获取实时信息或执行动作；4. 执行（Action）：根据规划和工具返回的结果，决定下一步操作。 工程权衡：Agent 极大地拓展了 LLM 的应用边界，但其稳定性和成功率高度依赖于底层模型的推理能力（如 Prompt 遵循和逻辑推理）。多步执行容易产生错误累积，且工具调用的延迟较高。

### 案例模拟

业务案例模拟：在开发一个“智能数据分析 Agent”时，用户输入“分析上个月的销售数据并生成报告”。Agent 首先进行规划：1. 调用 SQL 工具从数据库提取数据；2. 调用 Python 代码解释器进行数据清洗和趋势计算；3. 生成可视化图表；4. 撰写总结报告。在执行过程中，如果 SQL 查询报错，Agent 的反思机制会读取错误日志，自动修改 SQL 语句并重新查询，最终自主完成整个数据分析闭环。

### 2. 二、大模型（LLMs）agent 有哪些部分组成？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 核心大脑：大语言模型负责理解、推理和决策生成。
- 规划模块：将复杂任务拆解为可执行的子任务序列。
- 记忆与工具：短期长期记忆结合外部API调用扩展能力。

### 详细解答

大模型Agent通常由四个核心部分组成：大脑（LLM）、规划（Planning）、记忆（Memory）和工具使用（Tool Use）。大脑是Agent的控制中心，负责自然语言理解、逻辑推理和生成指令。规划模块使Agent能够将复杂的宏大目标分解为易于管理的子任务，并制定执行步骤，同时包含自我反省机制以纠正错误。记忆模块分为短期记忆（如上下文对话历史）和长期记忆（如通过向量数据库存储的外部知识），帮助Agent在长时间交互中保持状态。工具使用则赋予了Agent与外部世界交互的能力，例如调用搜索引擎、计算器、代码解释器或特定业务API，从而弥补大模型在实时信息获取和精确计算上的短板。工程上，这四者的协同决定了Agent的智能上限。

### 案例模拟

面试官追问：“在实际业务中，这四个模块哪个最容易成为性能瓶颈？” 回答：“通常是规划和工具使用的结合处。当任务复杂时，LLM可能会规划出错误的工具调用顺序，或者生成不符合API格式的参数。我们在项目中通过引入中间层的校验机制和Few-shot Prompting来规范工具调用，同时利用短期记忆缓存中间结果，显著降低了API调用失败率。”

### 3. 2.1 介绍一下 规划（planning）？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 任务分解：将复杂宏观目标拆解为多个可执行的子任务。
- 路径规划：确定子任务的执行顺序与依赖关系。
- 自我反省：在执行过程中评估结果并动态调整后续计划。

### 详细解答

规划（Planning）是Agent处理复杂任务的核心能力，主要包含任务分解和自我反省两个维度。结论上，规划使Agent从单步问答进化为多步问题解决者。原理方面，任务分解通常借助思维链或思维树等提示工程技术，引导大模型将宏大目标拆解为逻辑连贯的子任务序列。自我反省则是让Agent在执行子任务后，评估当前结果是否符合预期，若发现错误或偏离目标，则动态调整后续规划。在工程权衡中，过于精细的规划会增加推理延迟和Token消耗，而过于粗略则容易导致执行失败。因此，实际应用中常采用启发式规则或预设标准作业程序来辅助LLM进行高效规划。

### 案例模拟

面试官追问：“如果Agent在规划时陷入死循环怎么办？” 回答：“这在ReAct框架中很常见。我们通常采取三种策略：一是设置最大迭代步数强制熔断；二是引入外部监督机制，当检测到重复执行相同动作时，注入特定的纠偏Prompt；三是优化自我反省模块，让Agent在每次行动前显式评估当前状态与历史动作的差异，从而主动跳出循环。”

### 详细解答

拆解子目标和任务分解主要通过精心设计的提示工程和框架引导大模型完成。结论上，有效的拆解需要结合任务上下文和模型的推理能力。原理上，最基础的方法是使用逐步思考等简单指令触发模型的思维链。对于更复杂的场景，可以采用少样本提示，提供几个成功的任务分解示例供模型学习。在工程实践中，常将任务分解抽象为一个独立的LLM调用节点，输入总目标和可用工具列表，输出结构化的子任务数组及依赖关系。为了提高稳定性，还可以引入领域特定的标准作业程序，将大模型的自由发挥限制在预设的业务流程框架内。权衡来看，动态拆解灵活性高但易出错，静态流程稳定性好但泛化性差，两者结合是当前主流。

### 案例模拟

业务案例模拟：“在开发一个数据分析Agent时，用户输入‘分析上季度销售下滑原因’。Agent首先通过任务分解模块，将目标拆解为：1.调用SQL工具获取上季度各品类销量；2.调用API获取市场大盘数据；3.对比分析并生成图表；4.撰写总结报告。通过将拆解结果结构化输出，系统能按顺序调度不同工具，确保复杂任务有序执行。”

### 6. 2.1.2.1 如何进行 模型自我反省？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 结果评估：对比执行结果与预期目标，判断是否成功。
- 错误归因：分析失败原因，如工具参数错误或逻辑漏洞。
- 策略调整：基于错误分析生成新的执行计划或修正动作。

### 详细解答

模型自我反省是通过让Agent审视自身过去的行动和结果，以优化未来决策的过程。结论上，它是提升Agent在复杂环境中鲁棒性的关键。原理上，自我反省通常作为一个独立的评估节点存在。当Agent执行完一个子任务后，系统会将当前目标、采取的行动和环境反馈输入给LLM，要求其判断行动是否有效。如果无效，LLM需要生成一段反思文本，指出错误原因，并据此更新下一步的规划。在工程实现中，反省机制可以显著降低幻觉和工具调用失败率，但代价是增加了交互轮次和延迟。因此，通常只在关键节点或捕获到明确异常时才触发反省机制。

### 案例模拟

业务案例模拟：“在代码生成Agent中，如果生成的代码在沙箱中运行报错，Agent会触发自我反省。它会将报错堆栈信息和原始代码作为输入，分析出‘变量未初始化导致空指针异常’。随后，Agent在记忆中记录这一反思，并在下一次迭代中修正代码。这种闭环反省机制使代码通过率从百分之四十提升到了百分之七十五。”

### 7. 2.1.2.2 模型自我反省 有哪些方法？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 反应框架：交替进行推理与行动，实时根据环境反馈调整。
- 语言强化：通过语言反馈强化学习，维护长期反思记忆。
- 经验回溯：利用历史成败经验指导当前决策，优化输出。

### 详细解答

模型自我反省的方法主要围绕如何获取反馈并将其转化为改进策略。结论上，不同方法在反馈粒度和记忆持久性上有所区别。最经典的是反应框架，它在每步行动后直接观察结果并进行下一步推理，属于即时、局部的反省。进阶框架则更进一步，它不仅在任务失败后生成语言反馈，还会将这些反思存储在长期记忆中，使得Agent在未来遇到类似任务时能避免重复犯错。此外，还有自我优化方法，主要用于文本或代码生成，模型通过多轮自我批评和修改来提升输出质量。工程权衡上，维护复杂的记忆结构适合长周期任务；而简单反应框架实现轻量，更适合短平快的工具调用场景。

### 案例模拟

面试官追问：“基于语言反馈的反思和传统的强化学习有什么区别？” 回答：“传统强化学习依赖标量奖励信号来更新模型权重，这在自然语言任务中很难设计且训练成本高。基于语言的反思可以看作是一种轻量级的强化学习，它不更新模型权重，而是将反思文本作为上下文提示输入给模型。这种方式可解释性强，且非常适合利用大模型强大的上下文学习能力。”

### 8. 2.2 介绍一下 记忆（Memory）？

### 基础知识补充

- 短期记忆：存储当前对话上下文，受限于模型上下文窗口。

- 记忆检索：通过相似度计算将相关记忆召回至当前上下文中。

### 详细解答

记忆模块赋予了Agent在时间跨度上保持状态和积累经验的能力。结论上，记忆机制解决了大模型无状态和上下文窗口受限的问题。原理上，记忆分为短期和长期。短期记忆通常指当前任务的上下文或多轮对话历史，直接受限于模型的上下文窗口大小。长期记忆则将历史对话、外部文档或反思经验转化为向量，存储在向量数据库中。当Agent需要决策时，通过语义检索召回相关记忆，作为提示词的一部分输入给模型。工程实践中，记忆的难点在于遗忘与更新机制，即如何清理冗余信息、解决记忆冲突，以保证检索的准确性和效率。

### 案例模拟

业务案例模拟：“在开发一个个人助理Agent时，用户今天说‘我喜欢喝拿铁’，系统将其存入长期记忆。一个月后用户说‘帮我点杯咖啡’，Agent通过向量检索召回了‘喜欢拿铁’的记忆，并自动推荐了拿铁。为了防止记忆膨胀，我们还引入了记忆衰减和摘要机制，定期将零散的短期对话压缩为结构化的用户画像标签。”

### 详细解答

工具使用是Agent与外部物理或数字世界交互的桥梁。结论上，它极大地拓展了大模型的能力边界，使其从文本生成器变为任务执行者。原理上，开发者首先需要以标准化格式向模型描述可用工具的名称、功能和参数要求。当用户输入复杂请求时，模型通过推理决定调用哪个工具，并生成符合格式的结构化参数。系统拦截这些参数，在本地或云端执行实际的接口调用，然后将执行结果作为观察返回给模型，供其生成最终回答。工程权衡中，工具数量过多会导致模型选择困难和幻觉，通常需要通过检索增强技术动态召回相关工具，或对模型进行专门的工具调用微调。

### 案例模拟

面试官追问：“如果大模型生成的工具参数不符合格式，或者缺少必填参数怎么办？” 回答：“这在工程中很常见。我们通常采用多层防御机制：首先，使用支持强约束输出的模型；其次，在代码层捕获解析异常，如果失败，则将错误信息和原始要求重新拼接成提示词，让模型自我修正；最后，如果缺少必填参数，系统会中断调用，主动向用户发起多轮对话以补全缺失信息。”

### 10. 三、大模型（LLMs）agent 主要 利用了 大模型 哪些能力？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）agent 面 / 未知](https://articles.zsxq.com/id_le02luntesap.html)

### 基础知识补充

- 意图理解与任务拆解能力，将复杂目标转化为步骤。
- 上下文记忆与逻辑推理能力，支持多轮交互与反思。
- 格式化输出与工具调用能力，实现与外部环境交互。

### 详细解答

大模型Agent主要利用了LLM的推理规划、记忆管理和工具调用三大核心能力。首先，推理规划能力（如CoT、ReAct）使模型能够理解复杂指令，将其拆解为可执行的子任务，并在执行过程中进行自我反思与纠错。其次，记忆能力包括短期上下文窗口和长期向量数据库检索，帮助Agent在多轮交互中保持状态连贯性。最后，工具调用能力（Function Calling）允许模型生成符合特定格式（如JSON）的参数，从而触发外部API、搜索引擎或代码解释器。在工程实践中，这三者的结合使得LLM从单纯的文本生成器进化为能够自主感知环境并采取行动的智能体，极大地拓展了业务落地场景。

### 案例模拟

面试官追问：“如果Agent在执行多步任务时陷入死循环，如何利用大模型能力解决？” 回答：“可以引入反思机制（Reflection）。在每次执行工具后，将结果反馈给大模型，并要求其评估当前进度是否偏离目标。如果检测到重复调用或无效输出，强制触发预设的纠错Prompt，或者在系统层面对同一工具的连续失败调用设置最大重试次数，超过阈值则让人类介入。”

### 11. 一、为什么需要 函数调用(function call)？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：LLMs_interview_notes / 函数调用 Function Call 篇 / 未知

### 基础知识补充

- 弥补大模型知识滞后性，实时获取外部最新数据。
- 解决大模型数学计算和逻辑执行能力薄弱的问题。
- 打通大模型与企业内部私有系统及API的交互壁垒。

### 详细解答

引入函数调用（Function Call）主要是为了打破大模型“只能输出文本”的封闭状态，赋予其与外部世界交互的能力。首先，大模型的训练数据存在截止日期，无法回答实时信息（如天气、股价），通过调用外部API可以弥补知识滞后。其次，大模型本质是概率预测模型，在精确数学计算或复杂逻辑执行上容易产生幻觉，交由外部代码执行器处理更为可靠。最后，在企业级应用中，大模型需要操作私有数据库或执行特定业务流程（如发邮件、查订单），函数调用提供了一种结构化、确定性的接口标准，使得大模型能够作为大脑精准调度各类工程组件，实现从“聊天机器人”到“生产力工具”的跨越。

### 案例模拟

业务案例模拟：在开发智能客服系统时，用户询问“帮我查一下昨天买的鞋子发货没”。如果没有Function Call，模型只能回答通用话术。引入后，模型会解析出意图并生成调用get_order_status(date="yesterday", item="shoes")的JSON参数，后端执行该函数查询数据库，将真实物流状态返回给模型，最终模型生成“您的鞋子已发货，预计明天送达”的准确回复。

### 12. 二、什么是 函数调用(function call)？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：LLMs_interview_notes / 函数调用 Function Call 篇 / 未知

### 基础知识补充

- 允许大模型根据用户输入生成结构化函数参数。
- 包含意图识别、参数提取和外部执行三个核心环节。
- 本质是将自然语言转化为机器可读的API调用指令。

### 详细解答

函数调用（Function Call）是一种让大语言模型能够连接外部工具和API的技术机制。其核心流程是：开发者在请求大模型时，除了提供用户Prompt，还会附带一组可用函数的描述（包括函数名、功能说明及参数的JSON Schema）。大模型在推理时，会判断是否需要调用这些函数来回答用户问题。如果需要，模型不会直接输出自然语言回复，而是输出一个符合预设格式的JSON对象，包含目标函数名及其所需参数。随后，应用程序拦截该输出，在本地或服务端实际执行对应函数，并将执行结果作为新的上下文再次输入给大模型，最终由模型综合结果生成对用户的自然语言回答。

### 案例模拟

面试官追问：“Function Call和传统的意图识别（NLU）有什么区别？” 回答：“传统NLU通常依赖预定义的意图分类和槽位填充模型，泛化能力弱，新增工具需重新训练。而Function Call利用大模型的零样本理解能力，只需提供自然语言的函数描述即可动态适配新工具。此外，大模型能处理更复杂的上下文依赖和多轮参数收集，甚至能一次性规划多个函数的并行调用。”

### 13. 开源模型 Function Call 方案有哪些？

- 主标签：Agent与工具调用
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：LLMs_interview_notes / 函数调用 Function Call 篇 / 未知

### 基础知识补充

- 基于微调的方案，如ToolLLaMA和Gorilla模型。
- 基于提示词工程的方案，如ReAct和System Prompt。
- 格式强制约束方案，如使用JSON Schema或语法树。

### 详细解答

开源模型实现Function Call主要有三种方案。第一种是指令微调（Fine-tuning），代表如Gorilla、ToolLLaMA，通过构建大量“指令-API调用”的问答对，对开源基座模型进行微调，使其原生具备输出结构化JSON的能力，准确率高但训练成本大。第二种是提示词工程（Prompt Engineering），如ReAct框架，通过在System Prompt中详细描述工具格式，引导模型按特定模板（如Thought/Action/Observation）输出，无需训练但占用上下文且稳定性较差。第三种是推理期约束（Constrained Decoding），如使用Guidance或Outlines库，在生成阶段基于正则表达式或有限状态机（FSM）强制模型只能输出合法的JSON格式，这种方案工程落地性强，能有效避免语法错误。

### 案例模拟

面试官追问：“在实际业务中，你会如何选择开源模型的Function Call方案？” 回答：“我会采用组合策略。首先使用具有较强基础能力的开源模型（如Qwen或Llama3），它们已经经过了一定程度的工具调用微调。其次，在推理端引入Outlines等格式约束工具，确保输出的JSON 100%合法，避免解析崩溃。如果特定业务API极其复杂，我还会收集线上真实调用数据，使用LoRA对模型进行轻量级微调，以提升参数提取的准确率。”
