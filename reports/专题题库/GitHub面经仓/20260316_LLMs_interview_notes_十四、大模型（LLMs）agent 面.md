## 专题总结

## 本专题考什么
- **Agent 核心概念与架构**：考察候选人是否真正理解 Agent 的本质（以大模型为大脑的智能体），以及能否清晰拆解 Agent 的标准组成部分（规划、记忆、工具）。
- **Function Calling（函数调用）机制**：考察对大模型与外部世界交互底层原理的掌握，包括为什么需要它（解决大模型固有缺陷）以及它的完整执行闭环。
- **工程落地认知**：评估候选人是否具备将大模型从“文本生成器”升级为“任务执行系统”的系统性思维。

## 答题主线
- **宏观定义入题**：用“大模型是大脑，工具是手脚”的生动比喻切入，明确 Agent 的核心定位。
- **架构拆解支撑**：按照“感知 -> 规划 -> 记忆 -> 行动”的逻辑链路，结构化说明 Agent 的运转机制。
- **痛点引出技术**：通过指出纯大模型的局限性（幻觉、无实时数据、无法执行动作），自然过渡到 Function Calling 的必要性。
- **原理解析收尾**：清晰描述 Function Calling 的交互流程（定义工具 -> 模型输出参数 -> 本地执行 -> 结果返还模型），展现扎实的工程底色。

## 代表题答法

### 问题1：什么是大模型（LLMs）agent？
**答题要点：**
大模型 Agent（智能体）是一种以 LLM 作为核心大脑，能够自主感知环境、理解意图、进行逻辑推理与规划，并调用外部工具来执行复杂任务的智能系统。
如果说传统大模型是“被动回答问题”的百科全书，那么 Agent 就是“主动解决问题”的数字员工。它的本质公式可以概括为：**Agent = LLM（大脑） + 规划（思考） + 记忆（经验） + 工具（手脚）**。

### 问题2：大模型（LLMs）agent 有哪些部分组成？
**答题要点：**
业界通常参考经典的 Agent 架构，将其分为四个核心部分：
1. **大脑（Brain）**：即 LLM 本身，负责自然语言理解、逻辑推理和决策。
2. **规划（Planning）**：
   - **任务分解**：将复杂大任务拆解为可执行的子任务（如 CoT 思维链、ToT 思维树）。
   - **反思与纠错**：在执行失败时，能够根据反馈调整计划（如 ReAct 框架）。
3. **记忆（Memory）**：
   - **短期记忆**：当前对话的上下文（受限于 Context Window）。
   - **长期记忆**：历史交互和知识库，通常借助向量数据库（Vector DB）进行存储和检索。
4. **工具/行动（Tools/Action）**：Agent 与外部世界交互的接口，如网络搜索、代码解释器、数据库查询或各类第三方 API。

### 问题3：什么是函数调用（Function Call）？
**答题要点：**
Function Calling 是大模型连接外部工具的核心机制。它允许开发者在向大模型提问时，附带提供一组“函数签名（包含函数名、功能描述和参数格式）”。
**工作流程：**
1. 大模型根据用户输入，判断是否需要调用这些函数。
2. 如果需要，模型不会直接生成自然语言回复，而是输出一个符合要求的结构化数据（通常是 JSON），包含要调用的**函数名**及**参数**。
3. 开发者在本地代码中解析该 JSON，实际执行对应的函数。
4. 开发者将函数的执行结果再次输入给大模型，大模型基于该结果生成最终的自然语言回答。

### 问题4：为什么需要函数调用（Function Call）？
**答题要点：**
引入 Function Calling 主要是为了解决纯大模型固有的四大痛点：
1. **突破信息时效性**：大模型的知识停留在训练截止日，通过调用“搜索”或“天气”API，可以获取实时信息。
2. **弥补计算能力短板**：大模型本质是概率预测，不擅长精准数学计算，通过调用“计算器”或“代码执行器”可保证结果绝对准确。
3. **消除幻觉并获取私有数据**：通过调用企业内部数据库（SQL/API），让模型基于真实业务数据回答，而非凭空捏造。
4. **实现对物理/数字世界的控制**：让模型从“只说不做”变成“能说能做”，例如通过 API 发送邮件、预定机票、控制智能家居等。此外，它能强制模型输出稳定的 JSON 格式，极大降低了工程解析的难度。
