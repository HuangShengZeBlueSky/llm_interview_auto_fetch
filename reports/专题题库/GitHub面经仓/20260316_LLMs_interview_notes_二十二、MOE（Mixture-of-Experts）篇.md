## 专题总结

## 本专题考什么
- **核心动机与 Scaling Law**：考察候选人是否理解在算力瓶颈下，如何通过稀疏化打破模型参数量与计算量（FLOPs）的强绑定关系。
- **网络架构与核心组件**：考察对 MOE 内部结构的掌握，包括 Router（路由/门控网络）和 Experts（专家网络）的设计。
- **工程与算法的 Trade-off**：考察对 MOE 优缺点的认知，如显存占用、通信开销（All-to-All）、负载均衡（Load Balancing）以及训练稳定性问题。

## 答题主线
- **痛点引入**：从 Dense（稠密）模型的算力墙切入，说明参数量与计算量成正比的困境。
- **核心解法**：点出 MOE 的本质——**“稀疏激活”（Sparse Activation）**，即“大参数量 + 小计算量”。
- **技术实现**：拆解 MOE 结构，按“输入 -> Router 算分 -> Top-K 路由 -> 专家前向计算 -> 加权求和”的流水线进行阐述。
- **关键挑战**：补充说明 MOE 训练中的常见坑点（如专家旱涝不均）及解决手段（辅助损失函数）。

## 代表题答法

### 问题1：一、为什么需要 MOE（Mixture-of-Experts）？
**答题要点：**
需要 MOE 的根本原因是为了**在有限算力下追求更大的模型容量（Scaling Law）**。可以从以下三个维度展开：
1. **打破 Dense 模型的算力瓶颈**：传统的稠密模型（如 LLaMA-2）在推理和训练时，每个 Token 都需要经过网络中的每一个参数。当模型规模扩大到万亿级别时，计算延迟和算力成本将变得不可接受。
2. **解耦参数量与计算量**：MOE 引入了稀疏激活机制。它允许我们在不显著增加推理计算量（FLOPs）的前提下，大幅增加模型的总参数量。例如，一个 8x7B 的 MOE 模型，总参数量接近 50B，但每个 Token 激活的参数量可能只有 14B，从而兼顾了“大模型的性能”与“小模型的速度”。
3. **多任务/多领域能力的提升**：虽然不绝对，但 MOE 架构中的不同专家在训练过程中往往会自发地产生一定程度的“专业化”分工，有助于模型在处理复杂多领域任务时减少知识冲突（Negative Transfer）。

### 问题2：二、MOE（Mixture-of-Experts）的思路是什么样的？
**答题要点：**
MOE 的核心思路是**“分而治之”与“稀疏激活”**。在主流的大语言模型（如 Mixtral 8x7B, DeepSeek-V2）中，通常是将 Transformer Block 中的 FFN（前馈神经网络）层替换为 MOE 层。具体包含以下几个关键步骤：
1. **专家网络（Experts）**：将原本单个庞大的 FFN 拆分成 $N$ 个结构相同但参数独立的小 FFN（即专家）。
2. **路由网络（Router/Gating）**：引入一个轻量级的门控网络（通常是一个线性层 + Softmax）。当一个 Token 进来时，Router 会计算该 Token 与各个专家的匹配得分。
3. **Top-K 稀疏路由**：为了保持计算效率，系统不会激活所有专家，而是只选取 Router 得分最高的 Top-$K$ 个专家（通常 $K=1$ 或 $2$）。该 Token 只被送入这 $K$ 个专家进行前向计算。
4. **加权聚合**：将这 $K$ 个专家的输出结果，按照 Router 给出的概率得分进行加权求和，作为该 MOE 层的最终输出。
5. **补充细节（高分项）**：为了防止 Router 偷懒只把 Token 发给少数几个“强者专家”（导致马太效应和专家坍塌），在训练思路上**必须引入负载均衡损失（Load Balancing Loss）**，强制要求 Token 尽可能均匀地分配给所有专家，以保证整体参数的利用率。
