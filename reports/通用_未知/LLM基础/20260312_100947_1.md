# 题目1：请介绍 Transformer 的结构组成及各部分作用

**题目描述**：阐述 Transformer 架构的核心组件（Encoder与Decoder）及其具体功能。

**前置基础知识**：Seq2Seq 模型架构、自注意力机制（Self-Attention）、残差连接（Residual Connection）与层归一化（Layer Normalization）。

**核心解答**：
Transformer 整体采用 Encoder-Decoder 架构：
1. **输入部分**：
   - **Embedding 层**：将离散的 Token 转化为连续的稠密向量。
   - **Positional Encoding**：由于 Transformer 是并行处理数据，缺乏序列位置信息，需通过绝对位置编码（如正余弦编码）或相对位置编码（如 RoPE、ALiBi）注入位置信息。
2. **Encoder（编码器）**：由多个相同的 Layer 堆叠而成，主要用于提取输入序列的丰富特征表示。每个 Layer 包含：
   - **多头自注意力机制（Multi-Head Self-Attention）**：捕捉序列内部各个 Token 之间的全局依赖关系。
   - **前馈神经网络（FFN）**：通常是两层全连接层（中间加激活函数，如 ReLU/GELU），对注意力机制输出的特征进行非线性映射，增强模型的表达能力。
   - **Add & Norm**：残差连接（Add）缓解深层网络梯度消失问题；LayerNorm（Norm）稳定数据分布，加速收敛。
3. **Decoder（解码器）**：同样由多个 Layer 堆叠，用于自回归生成输出序列。除了包含 Encoder 的组件外，还有：
   - **掩码多头自注意力（Masked Multi-Head Self-Attention）**：加入下三角 Mask，确保在生成当前 Token 时，只能看到前面的 Token，防止“信息泄露”（看到未来信息）。
   - **交叉注意力（Cross-Attention）**：Query 来自 Decoder 的上一层输出，而 Key 和 Value 来自 Encoder 的最终输出，实现生成内容与输入上下文的对齐。
4. **输出部分（Linear + Softmax）**：将 Decoder 的输出隐向量映射到词表维度，通过 Softmax 转化为生成各个 Token 的概率分布。

**引申思考/面试追问**：
- *追问*：为什么 Transformer 中使用 LayerNorm 而不是 BatchNorm？
  - *思路*：NLP 任务中句子长度（Sequence Length）差异很大，BatchNorm 在 Batch 维度进行归一化，会导致较长序列后面部分计算得到的均值方差极度不准确（存在大量 Padding）。LayerNorm 在 Feature（Hidden Size）维度归一化，不受序列长度和 Batch Size 影响。

# 题目2：如何降低 Transformer 的计算复杂度？常见的稀疏注意力变体有哪些？

**题目描述**：分析降低 Transformer 中 Self-Attention 二次复杂度的策略及主流的稀疏注意力机制。

**前置基础知识**：标准 Self-Attention 的时间复杂度和空间复杂度均为 $\mathcal{O}(N^2 \cdot d)$，其中 $N$ 为序列长度，$d$ 为隐藏层维度。当上下文窗口急剧增大时，$N^2$ 成为计算和显存的绝对瓶颈。

**核心解答**：
降低计算复杂度的方向主要分为**系统级优化**、**架构级优化**和**算法级优化（稀疏/线性注意力）**：
1. **系统级/IO优化**：如 **FlashAttention**。不改变数学等价性，通过 Tiling（分块计算）和 Recomputation（重计算），减少 GPU SRAM 与 HBM 之间的内存读写（Memory Bound），大幅提升计算速度并降低峰值显存。
2. **架构级优化**：如 **MQA（Multi-Query Attention）** 和 **GQA（Grouped-Query Attention）**，通过共享 Key 和 Value 矩阵，极大减少 KV Cache 的显存占用，从而提高推理时的 Batch Size 和吞吐量。
3. **算法级优化（常见稀疏注意力变体）**，通过打破全连接图，将复杂度降至 $\mathcal{O}(N \log N)$ 或 $\mathcal{O}(N)$：
   - **局部注意力（Local/Sliding Window Attention）**：每个 Token 只与周围固定窗口大小的 Token 计算注意力（如 Longformer, Mistral 的 SWA）。
   - **块稀疏注意力（BlockSparse Attention）**：将注意力矩阵划分为多个块，只计算部分关键块（如 BigBird 结合了局部、随机和全局 Token）。
   - **步幅/扩张注意力（Dilated/Stride Attention）**：类似于空洞卷积，按一定步长跳跃计算注意力（如 Sparse Transformer）。
   - **动态/哈希注意力**：通过 LSH（局部敏感哈希）将相似的 Query 和 Key 聚类到同一个桶中，只在桶内计算（如 Reformer）。
   - **线性注意力（Linear Attention）**：利用核函数（Kernel）或泰勒展开改变矩阵乘法顺序，将 $\mathcal{O}(N^2)$ 降为 $\mathcal{O}(N)$（如 Performer, RWKV, Mamba 等 SSM 架构本质上也解决了二次复杂度问题）。

**引申思考/面试追问**：
- *追问*：FlashAttention 既然不改变理论复杂度 $\mathcal{O}(N^2)$，为什么还能让长文本训练变得可行？
  - *思路*：GPU 计算 Attention 时瓶颈不在算力（FLOPs），而在访存（显存带宽）。FlashAttention 通过硬件感知优化，将数据加载到高速 SRAM 中分块计算，避免了庞大的中间注意力矩阵 $S$ 和 $P$ 写入低速 HBM，从而突破了显存墙。

# 题目3：LoRA 微调的原理是什么？秩 r 的选择会对模型表现产生什么影响？

**题目描述**：解释 LoRA（Low-Rank Adaptation）的数学原理及超参数秩 $r$ 对模型效果和资源消耗的影响。

**前置基础知识**：预训练模型的“内在秩（Intrinsic Dimension）”假说（模型在特定任务上的优化其实发生在低维子空间中）；矩阵低秩分解；全参数微调（SFT）显存占用过高的痛点。

**核心解答**：
**LoRA 原理**：
LoRA 假设模型在微调过程中的权重更新矩阵 $\Delta W$ 具有极低的“内在秩”。对于预训练权重矩阵 $W_0 \in \mathbb{R}^{d \times k}$，LoRA 冻结 $W_0$，通过引入两个可训练的低秩矩阵 $A \in \mathbb{R}^{r \times k}$ 和 $B \in \mathbb{R}^{d \times r}$（其中 $r \ll \min(d, k)$）来近似 $\Delta W$。
前向传播公式变为：$h = W_0 x + \Delta W x = W_0 x + B A x$。
初始化时，矩阵 $A$ 采用随机高斯初始化，矩阵 $B$ 全零初始化，确保初始状态下 $BA = 0$，即微调初始模型等价于原模型。推理时，可以直接将 $BA$ 提前加到 $W_0$ 中，实现**零额外推理延迟**。

**秩 $r$ 的选择及影响**：
- **模型表现**：$r$ 决定了更新矩阵的表达能力（参数量）。$r$ 过小（如 $r=1$）可能导致欠拟合，无法学习到复杂任务的特征；$r$ 适中（如 $8, 16, 64$）通常能达到与全参微调媲美的效果；$r$ 过大会导致边际收益递减，且容易在小数据集上引发过拟合。
- **资源消耗**：参数量与 $r$ 呈线性关系。$r$ 越大，微调阶段需要保存的优化器状态（如 Adam 的动量和方差）和梯度就越多，占用的 GPU 显存和训练时间也会相应增加。

**引申思考/面试追问**：
- *追问*：LoRA 的 A 和 B 矩阵初始化方式可以互换吗？或者都用高斯初始化？
  - *思路*：不能互换，也不能都用高斯。必须保证 $B$ 为零，这样微调开始时的旁路输出为 0，模型表现与预训练模型完全一致，保证训练的稳定性。如果都非零，初始的随机噪声会破坏预训练模型的输出，导致训练初期 Loss 震荡。

# 题目4：KV Cache 是什么？为什么能极大地提升推理速度？

**题目描述**：解释 KV Cache 机制及其在 LLM 自回归推理中加速的原理。

**前置基础知识**：LLM 的自回归生成（Autoregressive Generation）机制（即根据 $Token_{1...t-1}$ 预测 $Token_t$）；Self-Attention 计算公式。

**核心解答**：
**KV Cache 是什么**：
在 Transformer Decoder 的自回归推理阶段，每生成一个新的 Token，都需要计算当前 Token 与之前所有历史 Token 的注意力。KV Cache 是一种空间换时间的优化技术：它将历史 Token 计算过的 Key (K) 和 Value (V) 张量缓存到 GPU 显存中。

**为什么能极大提升推理速度**：
在没有 KV Cache 时，生成第 $t$ 个 Token 需要重新计算前 $t-1$ 个 Token 的 Q、K、V，并执行完整的矩阵乘法，这导致大量重复计算，时间复杂度为 $\mathcal{O}(N^2)$。
引入 KV Cache 后，在生成第 $t$ 个 Token 时：
1. 只需要通过当前（第 $t$ 个）Token 计算出 $Q_t$、$K_t$、$V_t$。
2. 将 $K_t$、$V_t$ 拼接到缓存的历史 $K_{1:t-1}$ 和 $V_{1:t-1}$ 后面。
3. 用 $Q_t$ 直接与拼接后的全量 K 和 V 计算 Attention。
这样，每一步生成的计算量只与当前单步相关，避免了历史 Token KV 投影的重复计算，将生成阶段（Decode Phase）的时间复杂度从 $\mathcal{O}(N^2)$ 降到了 $\mathcal{O}(N)$，极大地降低了计算延迟（Latency）。

**引申思考/面试追问**：
- *追问*：KV Cache 虽然降低了计算量，但也带来了什么新瓶颈？业界是怎么解决的？
  - *思路*：带来了严重的显存占用瓶颈和显存碎片化问题（Memory Bound）。解决方法：算法层采用 MQA/GQA 减少 KV Head 数量；系统层采用 PagedAttention（如 vLLM 框架），像操作系统管理虚拟内存一样对 KV Cache 进行分页管理，消除显存碎片，极大提升并发 Batch Size。

# 题目5：RAG 完整流程，构建向量检索库时如何处理时间衰减对召回的影响？

**题目描述**：梳理 RAG 全链路，并提出解决知识时效性（时间衰减）在向量检索中问题的方案。

**前置基础知识**：RAG（Retrieval-Augmented Generation）基础架构；向量数据库（Vector DB）；Embedding 模型。

**核心解答**：
**RAG 完整流程**：
1. **数据处理与入库（Offline）**：文档解析（PDF/Word/HTML提取） $\rightarrow$ 文本切分（Chunking） $\rightarrow$ 向量化（Embedding） $\rightarrow$ 存入向量数据库（建立 HNSW 等索引，并附加 Metadata）。
2. **检索与生成（Online）**：用户 Query $\rightarrow$ Query改写/扩展 $\rightarrow$ Query 向量化 $\rightarrow$ 向量数据库检索（Top-K 相似度召回） $\rightarrow$ 召回内容重排（Rerank） $\rightarrow$ 将上下文拼接到 Prompt $\rightarrow$ LLM 生成回答。

**处理时间衰减对召回影响的策略**：
知识具有时效性，旧文档的向量相似度可能很高，但内容已过期。处理方案如下：
1. **时间加权的混合检索**：在向量检索时，不仅计算余弦相似度，还引入时间衰减函数（如高斯衰减、指数衰减）。最终 Score = $\alpha \cdot \text{VectorScore} + \beta \cdot \text{TimeDecayFunction}(T_{now} - T_{doc})$。例如 Elasticsearch 的 `function_score` 结合衰减函数。
2. **Metadata 硬过滤与动态路由**：将文档的创建/更新时间作为 Metadata 存入向量库。通过 LLM 意图识别（如识别到“最新”、“2024年”），在检索时加入时间范围的硬过滤条件（Pre-filtering）。
3. **向量与时间特征融合**：在 Embedding 阶段，将时间戳信息通过 Time2Vec 等方式编码为向量，与文本 Embedding 拼接，让模型在向量空间层面感知时间距离。
4. **知识库生命周期管理（CRUD）**：建立定期的知识更新机制，对于同一实体的旧知识，打上“已过期”标签降权，或直接进行 Upsert（更新覆盖）和 Delete 操作。

**引申思考/面试追问**：
- *追问*：如果用户问“对比一下2022年和2023年的财报数据”，时间衰减策略会导致2022年的召回不出来，怎么解决？
  - *思路*：此时不能用简单的全局时间衰减。需要通过 Query 理解（Self-Query 机制）提取出时间实体（2022, 2023），将时间衰减策略转换为 Metadata 的精确条件匹配（`year in [2022, 2023]`），结合多路召回解决。

# 题目6：微调时的训练数据是怎么构建的？如何保证样本多样性和质量？

**题目描述**：阐述 SFT（Supervised Fine-Tuning）阶段数据构建的流程以及质量与多样性控制策略。

**前置基础知识**：SFT 数据格式（Instruction, Input, Output）；“LIMA: Less Is More for Alignment” 思想（少量极高质量数据胜过海量低质量数据）。

**核心解答**：
**训练数据构建流程**：
1. **任务定义与种子收集**：确定微调目标（如问答、摘要、角色扮演），收集少量的优质人工标注数据（Seed Data）。
2. **数据合成（Self-Instruct / Evol-Instruct）**：利用强基座模型（如 GPT-4 / Claude）基于种子数据进行指令扩写、难度演化（如增加约束、多跳推理）生成大规模数据。
3. **业务日志清洗**：从真实用户的业务交互日志中提取 Query，经过脱敏后，由人工或强模型标注标准答案。

**保证样本多样性**：
1. **指令分布控制**：使用动词/名词词频分析（如画出指令分布的 Sunburst 图），确保指令类型（总结、翻译、写代码、推理等）分布均匀。
2. **长度与难度控制**：在 Prompt 中强制要求生成不同长度的回复；引入 Evol-Instruct 提升数据复杂度（从简单问答演化为需要逻辑推理的复杂问答）。
3. **聚类去重**：对数据进行 Embedding 后使用 K-Means 聚类，或使用 MinHash LSH 算法剔除语义高度相似的样本，确保模型见识到更广的特征空间。

**保证样本质量**：
1. **基于规则的启发式过滤**：剔除含有乱码、过短/过长、格式损坏、包含有害词汇的样本。
2. **模型裁判（LLM-as-a-Judge）**：使用 GPT-4 或专门的 Reward Model 对合成数据进行多维度打分（准确性、相关性、流畅度、无幻觉），设定阈值过滤低分样本。
3. **人工抽检与对齐**：对过滤后的数据进行随机抽样，人工审核是否符合人类价值观（HHH原则：Helpful, Honest, Harmless）。
4. **防止数据污染**：确保训练数据中不包含测试集/Benchmark（如 MMLU, GSM8K）的内容，使用 N-gram 重合度检测进行剔除。

**引申思考/面试追问**：
- *追问*：微调后发现模型出现了“灾难性遗忘”（丧失了预训练时的通用能力），在数据层面如何解决？
  - *思路*：在 SFT 数据集中混入一定比例（如 10%-20%）的高质量通用预训练数据或通用指令数据（如 Flan 数据集），作为正则化项，保持模型原有的能力分布。

# 题目7：在 RAG+知识图谱的 Agent 系统中，知识图谱更新的机制是怎样的？如何保证实时性？

**题目描述**：设计 GraphRAG/Agent 系统中知识图谱的动态更新机制及实时性保障方案。

**前置基础知识**：知识图谱（实体-关系-实体 三元组）；图数据库（Neo4j）；流式数据处理；GraphRAG 的工作原理。

**核心解答**：
**知识图谱更新机制**：
1. **增量信息抽取（IE）**：当有新文档输入时，Agent 调用 LLM 或专门的小模型（如 GLiNER, REBEL）进行增量的实体识别（NER）和关系抽取（RE）。
2. **实体消歧与对齐（Entity Resolution）**：新抽取的实体（如“马斯克”）需要与图谱中已有的实体（如“埃隆·马斯克”）进行比对融合，避免图谱爆炸。通常计算实体 Embedding 的相似度或交由 LLM 判断。
3. **图谱合并与 Upsert**：
   - **新增**：发现新实体或新关系，直接在图数据库中创建节点和边。
   - **更新/冲突解决**：发现属性变化，采取“追加边”（引入时序图谱概念，给边加上 `valid_from` 和 `valid_to` 时间戳）而不是直接覆盖，保留知识的演变历史。

**保证实时性的方案**：
1. **事件驱动的流式处理架构**：新知识产生时，将事件发送到消息队列（如 Kafka/RabbitMQ），由后台 Worker（如 Flink 或 Celery）异步消费并触发抽取和入库流程，不阻塞 Agent 的主链路。
2. **读写分离与多级缓存**：
   - 更新操作写入主库（Master）。
   - Agent 查询时，先查 Redis 缓存。对于刚刚更新的高频热点实体，图谱更新服务在写入图数据库的同时主动失效或更新 Redis 缓存（Cache Aside 模式）。
3. **局部子图更新（Sub-graph Update）**：避免全量重新构建图谱。只对受新文档影响的局部子图进行社区聚类（Community Detection）和总结（Summary）的更新。
4. **小步快跑的 LLM 抽取**：使用微调后的百亿参数级小模型（如 Qwen-7B-IE）替代 GPT-4 进行信息抽取，大幅降低抽取延迟，提升更新的吞吐率。

**引申思考/面试追问**：
- *追问*：相比于纯向量检索，引入知识图谱（GraphRAG）主要解决 Agent 的什么痛点？
  - *思路*：解决跨文档的多跳推理（Multi-hop reasoning）问题和全局信息总结（Global context）问题。向量检索容易丢失实体间的隐式关系，而图谱通过图游走（Graph Traversal）能精准找到分散在不同文档中的实体关联。

# 题目8：训练 LoRA 模型时，你是如何选择冻结层的？依据是什么？

**题目描述**：阐述 LoRA 微调时应用目标模块（Target Modules）的选择策略及背后的理论依据。

**前置基础知识**：Transformer 内部权重矩阵的分布（Attention 层的 $W_q, W_k, W_v, W_o$；FFN 层的 $W_{up}, W_{down}, W_{gate}$）。

**核心解答**：
**选择策略**：
1. **默认基础配置**：仅对 Attention 层的 $W_q$ 和 $W_v$ 注入 LoRA。
2. **进阶/推荐配置**：对全线性层（All Linear Layers）注入 LoRA，即 $W_q, W_k, W_v, W_o$ 以及 FFN 的所有权重矩阵。

**选择依据**：
1. **特征学习的敏感度**：$W_q$ 和 $W_v$ 分别负责特征的查询和值的映射，对任务的适配最为敏感。早期 LoRA 论文证明，在参数预算有限时，调整 $W_q$ 和 $W_v$ 带来的收益最大。
2. **参数预算与效果的权衡（经验法则）**：LoRA 原论文指出，在保持总参数量（参数预算）不变的前提下，**将较低的秩 $r$ 分散到更多的权重矩阵上，比将极高的秩 $r$ 集中在单一矩阵上效果更好**。例如，全层应用 $r=8$ 往往优于仅在 $W_q, W_v$ 应用 $r=32$。
3. **任务的迁移跨度**：
   - 如果是**领域内微调**（如语气对齐、简单的 QA 格式化），模型所需知识已经存在于预训练权重中，仅更新 Attention 层（调整信息路由机制）通常足够。
   - 如果是**跨领域微调**（如注入新语言能力、垂直行业知识、代码/数学能力），必须将 LoRA 注入到 **FFN 层**。因为研究表明 Transformer 的 FFN 层承担了类似“键值记忆库（Key-Value Memories）”的作用，存储了大量的具体事实知识。

**引申思考/面试追问**：
- *追问*：Embedding 层和 LM Head 层需要加 LoRA 吗？
  - *思路*：一般不需要。但如果微调任务涉及**扩充词表**（如增加特定医学专业词汇，或增强中文能力），则必须对 Embedding 层和输出的 LM Head 进行全参数训练，或者对它们单独应用 LoRA，否则模型无法正确映射新 Token。

# 题目9：在高并发查询 Agent 系统中，你会如何优化召回和生成阶段的延迟？

**题目描述**：针对大流量 Agent 场景，提出检索召回与 LLM 生成的全链路降延迟（Latency Optimization）方案。

**前置基础知识**：向量数据库索引机制（HNSW）；LLM 推理引擎优化（vLLM, TensorRT-LLM）；Agent 的 Tool Call 流程。

**核心解答**：
**召回阶段优化（Retrieval Latency）**：
1. **语义缓存（Semantic Cache）**：在 Agent 接入层引入 Redis + 向量轻量比对。当高并发 Query 到来时，若与缓存中历史 Query 的语义相似度超过阈值，直接返回历史答案，彻底跳过召回和生成。
2. **向量索引与降维**：使用 HNSW（分层导航小世界）或 IVF-PQ（倒排乘积量化）索引代替暴力检索；对 Embedding 向量使用标量量化（如 INT8）或采用降维模型（如 BGE-m3），减少内存带宽占用，加速距离计算。
3. **并发多路召回**：将向量检索、BM25 关键词检索、图谱检索通过协程（如 Python `asyncio`）并行发起，取最慢的一路作为总体延迟，而不是串行执行。

**生成阶段优化（Generation Latency）**：
1. **高性能推理引擎**：使用 **vLLM**（基于 PagedAttention 解决 KV Cache 显存碎片，极大提升并发 Batch Size）或 **TensorRT-LLM**（算子融合、In-flight Batching）。
2. **投机解码（Speculative Decoding）**：使用一个极小且快的草稿模型（Draft Model）一次性生成多个 Token，再由大模型（Target Model）并行验证。在不损失精度的前提下，可提速 1.5x - 3x，极大降低 TPOT（每个 Token 生成时间）。
3. **模型量化**：采用 AWQ、GPTQ 或 W8A8 量化，将模型权重从 FP16 降至 INT8/INT4。生成阶段是 Memory-Bound（访存密集型），降低显存读取量能直接降低延迟。
4. **Agent 链路优化**：
   - **Prompt 压缩**：使用 LLMLingua 等技术剔除冗余的检索上下文，减少 TTFT（首字延迟）。
   - **减少思考轮数**：通过微调让 Agent 模型学会将多个 Tool Call 合并为单次并行调用（Parallel Tool Calling），减少 LLM 的多轮自回归开销。
   - **流式输出（Streaming）**：在系统架构上采用 SSE/WebSocket 协议，大模型吐出一个 Token 就推给前端，消除用户的等待焦虑。

**引申思考/面试追问**：
- *追问*：Semantic Cache 的相似度阈值如果设置过低会导致什么问题？如何缓解？
  - *思路*：会导致“假阳性”，命中率高但回答文不对题。缓解方案：结合大模型的意图识别，或者将 Cache 分级（精确匹配层 + 语义匹配层），对于高风险场景（如医疗、金融）提高阈值或关闭语义缓存。

# 题目10：大规模 Agent 系统在多线程/多进程场景下的资源调度策略如何设计？

**题目描述**：设计分布式 Agent 系统中的计算资源（CPU/GPU）与任务的并发调度架构。

**前置基础知识**：并发编程模型（协程、线程池、进程池）；分布式任务调度（Celery, Ray）；计算密集型 vs IO 密集型任务。

**核心解答**：
大规模 Agent 系统具有混合特征：既有大量 **IO 密集型任务**（网络搜索、API 调用、数据库读写），又有极度消耗显存的 **计算密集型任务**（LLM 推理、Embedding 向量化）。因此，资源调度的核心在于**解耦与池化**：

1. **架构分离与计算解耦**：
   - 绝对不能在 Agent 进程内部直接加载大模型权重。
   - 将 LLM 推理、Embedding、Rerank 剥离为独立的**无状态微服务**（部署在 GPU 集群上，使用 vLLM/TGI 提供 HTTP/gRPC 接口）。
   - Agent 核心逻辑层（状态机、Tool 调用、业务串联）变为纯 CPU/IO 任务。
2. **Agent 层的并发调度（CPU 侧）**：
   - 使用 **异步协程（Asyncio）** 处理 Agent 的网络 I/O 等待，单进程即可支撑海量并发。
   - 对于消耗 CPU 的操作（如复杂的文档解析、大 JSON 序列化），投递到 **进程池（ProcessPoolExecutor）** 中，避免阻塞 Python 的 GIL（全局解释器锁）。
3. **任务队列与分布式路由**：
   - 引入消息队列（如 Kafka/RabbitMQ）和分布式计算框架（如 **Ray** 或 Celery）。
   - 根据 Agent 任务的类型和优先级进行路由。例如：将需要长上下文处理的 Agent 路由到显存充足的特定 GPU 节点；将短问答 Agent 路由到高并发节点。
4. **GPU 资源的全局调度（GPU 侧）**：
   - 利用推理引擎的 **Continuous Batching（动态批处理）** 机制。当成百上千个 Agent 并发请求 LLM 时，推理引擎能在单次前向传播中动态插入新请求、踢出已完成请求，最大化 GPU 的吞吐量（Throughput）。
   - 基于 K8s HPA（水平 Pod 自动扩缩容），监控消息队列的积压深度（Lag）和 GPU 利用率，动态拉起或销毁推理 Worker 节点。

**引申思考/面试追问**：
- *追问*：如果多个 Agent 需要共享同一段超长历史对话上下文，如何在分布式调度中优化？
  - *思路*：利用 **KV Cache 复用机制（如 vLLM 的 Prefix Caching）**。调度器采用一致性哈希（Consistent Hashing），将拥有相同 System Prompt 或历史 Context 的 Agent 请求路由到同一个 GPU 节点，直接命中显存中的 KV Cache，大幅降低计算开销。

# 题目11：如果在 GPU 资源有限的条件下同时提供推理和微调服务，如何做资源分配和任务调度以保证时延和吞吐？

**题目描述**：在单机或有限 GPU 集群上混合部署 LLM 推理（在线、延迟敏感）与微调（离线、吞吐敏感）任务的调度方案。

**前置基础知识**：GPU 显存管理；在线服务与离线任务的特征差异；NVIDIA MPS（多进程服务）/ MIG（多实例 GPU）；ZeRO-Offload。

**核心解答**：
核心原则：**推理服务是高优先级的在线任务，需严格保证时延（Latency）；微调服务是低优先级的离线任务，追求吞吐（Throughput），可容忍中断。**

1. **物理隔离：显存的空分复用（Spatial Multiplexing）**
   - 如果卡足够大（如 A100 80G），可以使用 NVIDIA **MIG（Multi-Instance GPU）** 或 **MPS** 将物理 GPU 硬件级划分为多个实例。
   - 例如，固定分配 70% 的显存和算力给推理服务（锁定 vLLM 的 `gpu_memory_utilization` 避免 OOM），剩余 30% 分配给微调任务。
2. **时间隔离：抢占式调度与时分复用（Time Multiplexing）**
   - 实现一个基于优先级的调度器。在业务低谷期（如夜间），拉起微调任务，吃满 GPU 资源。
   - 当推理请求突增时，调度器向微调任务发送中断信号。微调框架（如 Deepspeed）保存当前的 Checkpoint，挂起进程并释放显存，将算力全面让渡给推理服务。高峰期过后再恢复微调。
3. **架构级复用：LoRA 权重复用（针对同源模型）**
   - 如果微调任务和推理任务基于同一个 Base Model。可以部署如 **LoRAX / Punica / vLLM (multi-LoRA)** 这样的框架。
   - 显存中只保留**一份 Base Model 权重**。推理和微调共用这部分显存。微调时只在前向/反向传播中计算和更新极小的 LoRA 权重，极大节省显存。
4. **极致降显存：微调状态卸载（Offloading）**
   - 限制微调任务的显存占用，为推理让路。使用 **ZeRO-Offload** 技术，将微调时的优化器状态（Optimizer States）和梯度（Gradients）卸载到 CPU 内存进行计算。虽然牺牲了微调的吞吐速度，但保证了 GPU 显存足以支撑推理服务的并发。

**引申思考/面试追问**：
- *追问*：在同一个 GPU 上同时跑推理和微调进程，如果不做隔离，最容易发生什么问题？
  - *思路*：最容易发生 OOM（显存溢出）。因为 LLM 推理（尤其是 KV Cache）和微调（激活值和梯度）都会动态抢占显存，一旦发生峰值叠加，进程会直接崩溃。此外，微调的密集矩阵乘法会抢占 CUDA Core，导致推理延迟出现严重的长尾效应（P99 飙升）。

# 题目12：LeetCode 15 三数之和

**题目描述**：算法题：给定一个整数数组 `nums`，判断是否存在三元组 `[nums[i], nums[j], nums[k]]` 满足 `i != j`、`i != k` 且 `j != k`，同时这三个元素之和为 0。请返回所有和为 0 且不重复的三元组。

**前置基础知识**：排序算法；双指针（对撞指针）思想；剪枝与去重逻辑。

**核心解答**：
该题的核心在于**排序 + 双指针**，能在 $\mathcal{O}(N^2)$ 的时间复杂度内解决，并方便地处理去重问题。

**解题步骤**：
1. **特判与排序**：如果数组长度小于 3，直接返回空。对数组进行升序排序，时间复杂度 $\mathcal{O}(N \log N)$。
2. **固定基准数，双指针扫描**：遍历数组，固定当前元素 `nums[i]` 作为第一个数。
   - **剪枝**：如果 `nums[i] > 0`，因为数组已排序，后面的数必然大于 0，三数之和不可能为 0，直接 `break`。
   - **去重 1**：如果 `nums[i] == nums[i-1]`，说明当前基准数已被计算过，直接 `continue` 跳过。
3. **双指针寻找另外两数**：设左指针 `L = i + 1`，右指针 `R = n - 1`。
   - 计算 `sum = nums[i] + nums[L] + nums[R]`。
   - 如果 `sum == 0`：找到一个结果加入答案。接着左指针右移，右指针左移。**此时必须进行去重**：跳过所有连续相等的 `nums[L]` 和 `nums[R]`，防止产生重复的三元组。
   - 如果 `sum < 0`：说明数小了，`L` 右移。
   - 如果 `sum > 0`：说明数大了，`R` 左移。

**Python 核心代码实现**：
```python
def threeSum(nums):
    nums.sort()
    res = []
    n = len(nums)
    
    for i in range(n - 2):
        # 剪枝：最小的数大于0，不可能和为0
        if nums[i] > 0:
            break
        # 去重：跳过重复的基准数
        if i > 0 and nums[i] == nums[i-1]:
            continue
            
        L, R = i + 1, n - 1
        while L < R:
            total = nums[i] + nums[L] + nums[R]
            if total == 0:
                res.append([nums[i], nums[L], nums[R]])
                # 去重：跳过重复的L和R
                while L < R and nums[L] == nums[L+1]:
                    L += 1
                while L < R and nums[R] == nums[R-1]:
                    R -= 1
                L += 1
                R -= 1
            elif total < 0:
                L += 1
            else:
                R -= 1
                
    return res
```

**引申思考/面试追问**：
- *追问*：如果题目改为“寻找最接近目标值 `target` 的三数之和”（LeetCode 16），代码整体框架需要怎么改？
  - *思路*：整体框架（排序+双指针）完全不变。不需要精确判断 `sum == 0`，而是维护一个全局变量 `closest_sum` 和最小差值 `min_diff`。每次计算 `sum` 时更新 `closest_sum`，然后根据 `sum` 和 `target` 的大小关系移动左右指针即可。
