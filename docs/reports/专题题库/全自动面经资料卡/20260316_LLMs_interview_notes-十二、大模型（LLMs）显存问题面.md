## 大模型（LLMs）显存问题面

### 1. 大模型大概有多大，模型文件有多大?

- 主标签：训练并行与系统
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）显存问题面 / 未知](https://articles.zsxq.com/id_jhiocx89p3su.html)

### 基础知识补充

- 参数量决定模型大小，1B参数约等于10亿个参数。
- 半精度FP16/BF16下，1B参数占用约2GB存储空间。
- 模型文件大小还受词表大小、优化器状态等因素影响。

### 详细解答

大模型的规模通常以参数量（如7B、13B、65B）来衡量，模型文件的大小直接取决于参数量和存储精度。结论是：在常用的半精度（FP16或BF16）格式下，10亿（1B）参数大约占用2GB的磁盘空间。例如，一个7B的模型文件大约为14GB，65B的模型大约为130GB。如果采用单精度（FP32），则大小翻倍；若采用量化技术（如INT8或INT4），则分别降至约1GB和0.5GB每1B参数。在工程实践中，除了模型权重本身，训练时还需要保存优化器状态（如Adam的动量和方差）、梯度等，这会使检查点（Checkpoint）文件的大小达到纯权重的数倍。因此，在部署和训练时，必须提前规划好充足的磁盘和显存空间。

### 案例模拟

面试官追问：“下载了一个7B的开源模型，为什么实际占用空间比14GB大？” 回答：“这通常是因为除了模型权重（如.bin或.safetensors文件），还包含了词表文件、配置文件、分词器模型等。此外，有些模型发布时可能保存了FP32格式的权重，导致体积翻倍。在实际业务中，我们会优先使用safetensors格式，它不仅加载速度更快，还能避免反序列化带来的安全风险，并且支持内存映射（mmap）技术。”

### 2. 能否用4 * v100 32G训练vicuna 65b？

- 主标签：训练并行与系统
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）显存问题面 / 未知](https://articles.zsxq.com/id_jhiocx89p3su.html)

### 基础知识补充

- 65B模型在FP16下仅权重就需130GB显存，超出总显存。
- 训练还需存储梯度、优化器状态和激活值，显存需求更大。
- 4张32G显卡总显存128G，无法直接进行常规全参数训练。

### 详细解答

结论是：使用4张32G的V100显卡无法对Vicuna 65B进行常规的全参数微调或预训练。原理上，65B模型在半精度（FP16）下，仅模型权重就需要约130GB显存，这已经超过了4张32G显卡的总显存（128GB）。在训练过程中，还需要额外存储梯度（约130GB）、Adam优化器状态（约260GB）以及前向传播的激活值，总显存需求通常在1TB以上。即使采用LoRA等参数高效微调（PEFT）方法，虽然优化器状态和梯度的显存大幅减少，但基础模型权重的130GB依然无法装入128GB的显存中。工程权衡上，如果必须在这套硬件上运行，只能考虑极端的量化技术（如QLoRA将底座量化为4-bit）结合ZeRO-Offload将部分计算卸载到CPU内存，但这会导致训练速度极慢。

### 案例模拟

面试官追问：“如果非要在这4张卡上微调65B模型，你会采取什么具体方案？” 回答：“我会采用QLoRA方案。首先将65B基础模型加载为4-bit量化格式，这样权重显存占用降至约35GB，可以分布在多张卡上或单卡加载。然后注入低秩矩阵（LoRA）进行训练，仅更新极少量参数。同时开启Gradient Checkpointing以空间换时间减少激活值显存，并结合DeepSpeed ZeRO-3 Offload，将优化器状态卸载到CPU内存。虽然通信和CPU计算会成为瓶颈，但能勉强跑通。”

### 3. 如果就是想要试试65b模型，但是显存不多怎么办？

- 主标签：训练并行与系统
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）显存问题面 / 未知](https://articles.zsxq.com/id_jhiocx89p3su.html)

### 基础知识补充

- 采用模型量化技术，如INT8或INT4降低显存占用。
- 使用参数高效微调方法，如LoRA或QLoRA减少训练显存。
- 利用CPU Offload技术将部分数据卸载到系统内存。

### 详细解答

如果显存有限但希望尝试65B模型，结论是：必须结合量化、参数高效微调（PEFT）以及内存卸载技术来突破显存瓶颈。原理上，显存主要被模型权重、梯度、优化器状态和激活值占据。首先，可以通过QLoRA技术将65B的底座模型量化为4-bit，这能将权重显存从130GB（FP16）大幅压缩至约35GB。其次，使用LoRA只训练极少量的附加参数，从而极大地减少了梯度和优化器状态的显存需求。工程上，还可以开启Gradient Checkpointing（梯度检查点）技术，通过重新计算丢弃的激活值来节省显存。如果显存依然告急，可以引入DeepSpeed的ZeRO-Offload或Accelerate库，将优化器状态甚至部分模型权重卸载到廉价的CPU内存中。这种方案以牺牲一定的计算速度为代价，换取了在有限显存下运行大模型的可能性。

### 案例模拟

面试官追问：“使用CPU Offload技术会有什么明显的副作用？” 回答：“最明显的副作用是训练速度会大幅下降。因为GPU和CPU之间的数据传输带宽（PCIe）远低于GPU内部的显存带宽，频繁地将张量在CPU和GPU之间来回搬运会造成严重的通信瓶颈。在实际项目中，如果不是显存绝对不够，我们尽量避免使用Offload，或者只Offload更新频率较低的优化器状态，而不Offload模型权重或激活值。”

### 4. nB模型推理需要多少显存？

- 主标签：推理优化与部署
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）显存问题面 / 未知](https://articles.zsxq.com/id_jhiocx89p3su.html)

### 基础知识补充

- 推理显存主要由模型权重和KV Cache两部分组成。
- FP16精度下，nB模型权重占用约2n GB显存。
- KV Cache随序列长度和批次大小线性增长，需额外预留。

### 详细解答

计算nB（n十亿参数）模型推理所需的显存，结论是：在半精度（FP16/BF16）下，基础显存约为2n GB，但实际部署时需额外预留20%-50%用于KV Cache和上下文。原理上，推理过程不需要保存梯度和优化器状态，显存占用主要分为静态和动态两部分。静态部分是模型权重，1B参数在FP16下占2GB，因此nB模型占2n GB。动态部分主要是KV Cache（键值缓存），用于加速自回归生成，其大小与Batch Size、序列长度、模型层数和隐藏层维度成正比。工程权衡上，如果显存不足，可以采用INT8（约n GB）或INT4（约0.5n GB）量化推理，或者使用PagedAttention（如vLLM）来优化KV Cache的显存碎片，从而在有限显存下支持更大的并发量和更长的上下文。

### 案例模拟

面试官追问：“如何估算一个7B模型在并发量为10、序列长度为2048时的KV Cache大小？” 回答：“KV Cache的计算公式大致为：2 * 层数 * 隐藏层维度 * 序列长度 * Batch Size * 字节数(FP16为2)。以LLaMA-7B为例，32层，维度4096。单Token的KV Cache约为 2 * 32 * 4096 * 2 = 512KB。当序列长度为2048，Batch Size为10时，KV Cache总大小约为 512KB * 2048 * 10 ≈ 10GB。加上14GB的模型权重，总显存需求至少24GB，单张24G显卡（如RTX 3090/4090）会面临OOM风险。”

## 大模型（LLMs）显存优化策略篇

### 5. 一、介绍一下 gradient accumulation 显存优化方式？

- 主标签：训练并行与系统
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）显存优化策略篇 / 未知](https://articles.zsxq.com/id_a1l60awgge6q.html)

### 基础知识补充

- 梯度累加通过多次前向和反向传播累积梯度后再更新参数。
- 能够在有限显存下等效实现大Batch Size的训练效果。
- 减少了优化器更新的频率，降低了显存峰值压力。

### 详细解答

梯度累加（Gradient Accumulation）是一种以时间换空间的显存优化技术。结论是：它允许在显存受限的设备上使用较大的等效Batch Size进行训练。原理上，受显存限制，单次前向和反向传播只能处理较小的微批次（Micro-Batch）。梯度累加在计算完微批次的梯度后，不立即调用优化器更新模型参数，而是将梯度累加起来。经过N次累加后，再执行一次参数更新，并清空梯度。这样等效的Batch Size就等于Micro-Batch Size乘以N。工程权衡上，这种方法虽然不减少模型权重和优化器状态的显存，但大幅降低了单次计算的激活值显存峰值。缺点是增加了前向和反向传播的次数，且在分布式训练中，需要注意梯度同步的时机，通常只在最后一次累加时进行通信以减少开销。

### 案例模拟

面试官追问：“梯度累加和直接使用大Batch Size训练在数学上是完全等价的吗？” 回答：“在大多数情况下是近似等价的，但并非绝对等价。主要差异在于使用了依赖于Batch统计量的操作，比如Batch Normalization。在梯度累加中，BN的均值和方差是基于Micro-Batch计算的，这可能导致统计量不准确。不过在大模型训练中，通常使用Layer Normalization或RMSNorm，这些操作独立于Batch维度，因此梯度累加与大Batch Size在数学上基本是严格等价的。”

### 6. 二、介绍一下 gradient checkpointing 显存优化方式？

- 主标签：训练并行与系统
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）显存优化策略篇 / 未知](https://articles.zsxq.com/id_a1l60awgge6q.html)

### 基础知识补充

- 梯度检查点通过丢弃部分中间激活值来大幅节省显存。
- 在反向传播时重新计算被丢弃的激活值，以时间换空间。
- 能够将激活值的显存占用从O(N)降低到O(sqrt(N))。

### 详细解答

梯度检查点（Gradient Checkpointing）是一种经典的显存优化策略。结论是：它通过牺牲约20%-30%的计算时间，换取了显著的显存节省，使得训练更深更大的模型成为可能。原理上，在标准的前向传播中，所有的中间激活值都会被保存在显存中，以便在反向传播时计算梯度，这导致显存占用随模型层数线性增长。梯度检查点技术只在特定的层（即检查点）保存激活值，丢弃其他层的激活值。当反向传播需要这些被丢弃的激活值时，系统会从最近的检查点重新执行前向计算来恢复它们。工程权衡上，这是一种典型的“以计算换显存”的方法。在训练大语言模型时，激活值往往是显存消耗的大头，开启此功能可以大幅提高单卡能容纳的Batch Size或序列长度。

### 案例模拟

面试官追问：“在什么场景下不建议开启Gradient Checkpointing？” 回答：“如果显存非常充足，完全可以容纳当前的模型和Batch Size，就不建议开启，因为重新计算会白白增加20%以上的训练时间。另外，在模型极小或者序列长度很短的情况下，激活值占用的显存比例不高，开启它的收益不明显，反而拖慢速度。但在大模型微调（如长文本SFT）中，由于序列变长导致激活值显存暴增，开启它是必选项。”
