## 大模型（LLMs）增量预训练篇

### 1. 1. 为什么要增量预训练？

- 主标签：LLM基础
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）增量预训练篇 / 未知](https://articles.zsxq.com/id_jfq8la7g20ww.html)

### 基础知识补充

- 注入垂直领域知识，弥补通用模型专业短板
- 缓解灾难性遗忘，保持通用能力同时学习新知
- 降低从头预训练成本，利用已有模型权重初始化

### 详细解答

增量预训练（Continual Pre-training）的核心目的是在通用大模型基础上，低成本地注入垂直领域知识或新语言能力。结论是：相比从头预训练，它能大幅节省算力；相比SFT，它能更深层地改变模型的内部知识表示。原理上，通用模型在预训练时可能未见过特定领域的专有词汇或逻辑，直接SFT容易导致模型产生幻觉。通过在领域无监督语料上继续自回归预测，模型能学习到领域特有的分布规律。工程权衡上，增量预训练面临“灾难性遗忘”问题，即学习新知识时遗忘旧知识，通常需要混合一定比例的通用语料（如10%-20%）来维持原有能力，同时需调整学习率等超参数以避免破坏已有表征。

### 案例模拟

面试官追问：增量预训练时如何缓解灾难性遗忘？ 回答：在实际医疗大模型项目中，我们采用了数据混合策略。在增量数据中混入约15%的通用高质量预训练数据。同时，我们使用了较小的学习率（如通用预训练的十分之一），并采用余弦退火策略。此外，还可以引入正则化惩罚项或使用LoRA等参数高效微调方法，冻结大部分原始权重，从而在注入医疗知识的同时，最大程度保留模型的通用对话和推理能力。

### 2. 2. 进行 增量预训练 需要做哪些准备工作？

- 主标签：LLM基础
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）增量预训练篇 / 未知](https://articles.zsxq.com/id_jfq8la7g20ww.html)

### 基础知识补充

- 领域语料收集与清洗，去除噪声与敏感信息
- 词表扩充与嵌入层初始化，适配领域专有词汇
- 算力评估与分布式训练框架搭建，如Megatron

### 详细解答

进行增量预训练的准备工作主要涵盖数据、模型和工程三个维度。结论是：高质量的领域数据和适配的词表是成功的关键。首先是数据准备，需收集海量领域无监督文本，进行去重、去噪、脱敏和质量过滤，并按需与通用语料按比例混合。其次是模型准备，若领域包含大量专有名词（如医学、法律或新语言），需进行词表扩充（Vocabulary Expansion），并对新增词向量进行合理初始化（如取现有子词嵌入的平均值），以提高编码效率。最后是工程准备，需评估算力需求，选择合适的分布式训练框架（如Megatron-LM或DeepSpeed），配置数据并行、张量并行和流水线并行策略，并设定合适的学习率调度器和优化器状态。

### 案例模拟

面试官追问：扩充词表后，新增的Embedding参数如何初始化？ 回答：在我们的金融大模型项目中，扩充了约8000个金融专业词汇。如果随机初始化，训练初期会产生较大梯度，破坏原有权重。我们采用启发式初始化：将新词用原分词器切分为多个子词，取这些子词Embedding的平均值作为新词的初始权重。这样能让新词在初始阶段就具备一定的语义信息，加速收敛并稳定训练过程。

### 3. 3. 增量预训练 所用 训练框架？

- 主标签：LLM基础
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）增量预训练篇 / 未知](https://articles.zsxq.com/id_jfq8la7g20ww.html)

### 基础知识补充

- DeepSpeed：微软开源，提供ZeRO系列显存优化
- Megatron-LM：英伟达开源，支持多维张量与流水并行
- HuggingFace Accelerate：轻量级封装，易于快速上手

### 详细解答

增量预训练通常采用支持大规模分布式训练的框架，主流选择包括Megatron-LM、DeepSpeed及其结合体。结论是：选择框架需根据模型规模、硬件集群和开发成本综合权衡。Megatron-LM由NVIDIA开发，核心优势在于极致的张量并行（TP）和流水线并行（PP）实现，适合在高速互联的GPU集群上训练超大模型，算力利用率极高。DeepSpeed由微软开发，其核心是ZeRO（零冗余优化器）技术，通过对优化器状态、梯度和参数进行分片，极大降低了显存占用，适合显存受限或跨节点带宽较低的场景。工程实践中，常使用Megatron-DeepSpeed结合版，或基于HuggingFace的Accelerate/Transformers进行快速开发，以兼顾训练效率与易用性。

### 案例模拟

面试官追问：你们在百亿参数模型增量预训练时选了什么框架？为什么？ 回答：我们选择了基于DeepSpeed ZeRO-3的框架。因为我们的集群节点间网络带宽有限（非全量NVLink），Megatron的张量并行需要极高的节点内通信带宽，而流水线并行在百亿规模下切分不够灵活。ZeRO-3通过参数分片将显存压力均摊到各卡，配合CPU Offload技术，让我们能在有限的A100显存下跑通13B模型的全量参数增量预训练，虽然通信开销略大，但整体吞吐量满足业务需求。

### 4. 4. 增量预训练 训练流程 是怎么样？

- 主标签：LLM基础
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 大模型（LLMs）增量预训练篇 / 未知](https://articles.zsxq.com/id_jfq8la7g20ww.html)

### 基础知识补充

- 数据预处理与Tokenization，构建定长训练样本
- 模型加载与词表扩充适配，初始化新增参数
- 分布式训练执行与断点续训，监控Loss与指标

### 详细解答

增量预训练的流程是一个从数据到模型再到评估的闭环系统。结论是：标准流程包含数据构建、环境初始化、迭代训练与动态评估四个核心步骤。首先，将清洗好的领域语料与通用语料混合，通过Tokenizer转化为Token ID，并使用数据拼接（Packing）技术组合成固定长度（如4096）的序列，以最大化计算效率。其次，加载预训练模型权重，若有词表扩充则调整Embedding和LM Head层大小。接着，配置分布式训练参数（如学习率预热、混合精度BF16、ZeRO优化），启动训练。训练过程中需实时监控Loss曲线、梯度范数等指标，定期保存Checkpoints以支持断点续训。最后，在验证集上评估困惑度（Perplexity），并进行下游任务的零样本/少样本测试。

### 案例模拟

面试官追问：训练过程中Loss突然出现尖刺（Spike）怎么处理？ 回答：在训练我们的垂类模型时遇到过Loss Spike。首先，我们会检查是否是数据质量问题，比如混入了大量重复或乱码文本。如果数据正常，通常是梯度爆炸导致。我们的处理流程是：回滚到Spike前最近的正常Checkpoint，跳过当前导致异常的数据批次（Data Skipping），或者临时降低学习率、增大梯度裁剪（Gradient Clipping）的阈值。长期来看，采用BF16代替FP16能有效缓解数值溢出导致的Spike问题。

### 5. 一、 推理过程 分哪些阶段？

- 主标签：推理优化与部署
- 来源条数：2
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 增量预训练（Pretrain）样本拼接篇 / 未知](https://articles.zsxq.com/id_8f35p8piwl4v.html)
- 来源：[LLMs_interview_notes / 大模型（LLMs）推理加速篇 / 未知](https://articles.zsxq.com/id_kgzsxgro8cee.html)

### 基础知识补充

- Prefill阶段：并行处理输入Prompt，计算KV Cache
- Decode阶段：自回归逐个生成Token，依赖历史KV
- 显存与计算特征：Prefill计算密集，Decode访存密集

### 详细解答

大语言模型的推理过程通常分为两个截然不同的阶段：Prefill（预填充）阶段和Decode（解码）阶段。结论是：这两个阶段在计算特征和性能瓶颈上存在显著差异，需要针对性优化。Prefill阶段负责处理用户输入的Prompt，模型会并行计算所有输入Token的注意力矩阵，并生成初始的Key-Value (KV) Cache。此阶段是计算密集型（Compute-bound），主要受限于GPU的矩阵运算能力。Decode阶段则是自回归生成过程，模型根据历史KV Cache和当前生成的Token预测下一个Token。此阶段每次只能生成一个Token，无法充分利用GPU的并行计算单元，属于访存密集型（Memory-bound），主要受限于显存带宽。工程上常通过PagedAttention、连续批处理等技术优化这两个阶段。

### 案例模拟

面试官追问：针对Prefill和Decode阶段的不同瓶颈，有哪些优化手段？ 回答：在业务部署中，针对Prefill阶段的计算瓶颈，我们主要采用FlashAttention技术来加速注意力机制的计算，并减少显存读写。针对Decode阶段的访存瓶颈，我们引入了vLLM框架，利用PagedAttention对KV Cache进行显存分页管理，减少显存碎片，从而大幅提升Batch Size。此外，我们还尝试了投机解码（Speculative Decoding），用小模型快速生成草稿Token，大模型并行验证，有效提升了解码阶段的生成速度。

### 6. 1.2 Decoding（递归推理与解码输出）阶段

- 主标签：推理优化与部署
- 来源条数：2
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 增量预训练（Pretrain）样本拼接篇 / 未知](https://articles.zsxq.com/id_8f35p8piwl4v.html)
- 来源：[LLMs_interview_notes / 大模型（LLMs）推理加速篇 / 未知](https://articles.zsxq.com/id_kgzsxgro8cee.html)

### 基础知识补充

- 自回归生成机制，逐个预测下一个Token
- KV Cache复用，避免重复计算历史Token特征
- 采样策略应用，如Temperature、Top-k、Top-p

### 详细解答

Decoding阶段是大模型推理中负责实际文本生成的环节。结论是：该阶段的核心机制是自回归生成，其性能瓶颈在于显存带宽而非计算能力。在原理上，模型将前一步生成的Token作为当前步的输入，结合Prefill阶段和之前Decode步积累的KV Cache，通过Transformer层计算得到下一个Token的概率分布。为了避免每次都重新计算所有历史Token的注意力，必须复用KV Cache，这导致显存占用随生成长度线性增长。在输出前，还会应用解码策略（如贪心搜索、Top-k、Top-p采样和温度调节）来控制生成文本的多样性和确定性。工程权衡上，由于每次只处理一个Token的矩阵向量乘法（GEMV），GPU计算单元利用率极低，因此提升并发度（Batching）是优化的关键。

### 案例模拟

面试官追问：为什么Decoding阶段GPU利用率低？如何提升？ 回答：Decoding阶段主要执行矩阵-向量乘法（GEMV），计算量小但需要频繁从显存读取庞大的模型权重和KV Cache，受限于显存带宽，导致计算单元闲置。在实际部署中，我们采用Continuous Batching（动态批处理）技术，在请求生成结束时立即插入新请求，而不是等待整个Batch完成。结合PagedAttention优化KV Cache显存，我们将并发Batch Size提升了3倍，显著提高了整体吞吐量和GPU利用率。

### 7. 二、推理性能的评价指标？

- 主标签：推理优化与部署
- 来源条数：2
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 增量预训练（Pretrain）样本拼接篇 / 未知](https://articles.zsxq.com/id_8f35p8piwl4v.html)
- 来源：[LLMs_interview_notes / 大模型（LLMs）推理加速篇 / 未知](https://articles.zsxq.com/id_kgzsxgro8cee.html)

### 基础知识补充

- 首字延迟（TTFT）：衡量系统响应速度与Prefill效率
- 生成吞吐量（Tokens/s）：衡量Decode阶段生成速度
- 并发请求数（QPS/RPS）：衡量系统整体服务承载能力

### 详细解答

评估大模型推理性能需要综合考虑延迟、吞吐量和资源利用率等多个维度的指标。结论是：首字延迟（TTFT）和每秒生成Token数（TPOT）是用户体验的核心，而系统吞吐量决定了服务成本。首字延迟（Time To First Token）指从发送请求到收到第一个Token的时间，主要反映Prefill阶段的计算效率和网络延迟。每个Token生成时间（Time Per Output Token）反映Decode阶段的速度。系统级指标包括吞吐量（Throughput，即每秒处理的请求数或总Token数），它与Batch Size密切相关。工程权衡中，延迟和吞吐量往往是矛盾的：增大Batch Size能提升整体吞吐量，但会导致单个请求的TTFT和TPOT增加。因此，实际部署时需根据业务场景（如实时对话 vs 离线批处理）寻找最佳平衡点。

### 案例模拟

面试官追问：如果业务要求极低的首字延迟，你会如何优化？ 回答：在我们的实时语音对话项目中，首字延迟极其关键。首先，我们会限制最大并发Batch Size，避免Prefill阶段排队过长。其次，采用分离架构（Prefill-Decode Disaggregation），将计算密集的Prefill和访存密集的Decode分配到不同的GPU节点上，互不干扰。此外，针对长Prompt，我们会开启FlashAttention，并尽可能利用Prompt Cache技术缓存常见系统提示词的KV Cache，从而将TTFT降低了40%以上。

### 8. 一、Pretrain阶段，为什么需要拼接拼接？

- 主标签：LLM基础
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 增量预训练（Pretrain）样本拼接篇 / 未知](https://articles.zsxq.com/id_enteq22h1nhq.html)

### 基础知识补充

- 提升硬件计算效率，避免大量Padding浪费算力
- 统一输入序列长度，满足Transformer固定维度要求
- 增加单次前向传播的信息密度，加速模型收敛

### 详细解答

在预训练阶段，数据拼接（Packing）是将多篇短文档合并成一个固定长度序列的技术。结论是：数据拼接是提升GPU算力利用率和训练吞吐量的必要工程手段。原理上，自然语言文档长度参差不齐，而Transformer模型在批处理时要求同一Batch内的序列长度一致。如果不进行拼接，就需要对短文档填充大量无意义的Padding Token。这不仅浪费了宝贵的显存，还会让GPU执行大量无效的注意力计算。通过将多篇文档用特殊的EOS（End of Sentence）Token分隔并拼接至最大上下文长度（如4096），可以确保每个输入序列都充满有效信息。工程权衡上，拼接虽然提升了效率，但可能导致跨文档的注意力污染，通常需要配合特定的Attention Mask（如Document Masking）来隔离不同文档的上下文。

### 案例模拟

面试官追问：拼接数据后，不同文档之间会产生注意力交互，这会影响模型效果吗？ 回答：确实会产生跨文档的注意力污染（Cross-document Attention Leakage）。在我们的预训练实践中，如果不加处理，模型可能会在生成时混淆不同文档的逻辑。为了解决这个问题，我们在实现中引入了Block Diagonal Attention Mask（块对角注意力掩码）或者重置位置编码（Position ID Reset）。这样在计算Attention时，当前文档的Token只能看到同一文档内的Token，既享受了Packing带来的计算效率提升，又保证了文档间的语义隔离。

### 9. 二、有哪些 拼接方式？

- 主标签：LLM基础
- 来源条数：1
- 答案生成方式：模型自动补全
- 来源：[LLMs_interview_notes / 增量预训练（Pretrain）样本拼接篇 / 未知](https://articles.zsxq.com/id_enteq22h1nhq.html)

### 基础知识补充

- 贪心拼接（Greedy Packing）：按顺序填满序列长度
- 最佳适应拼接（Best-fit Packing）：类似背包问题优化
- 跨文档掩码拼接：拼接同时修改Attention Mask隔离

### 详细解答

数据拼接（Packing）在工程实现上有多种策略，主要区别在于空间利用率和语义隔离程度。结论是：贪心拼接最简单常用，而基于算法优化的拼接能进一步减少碎片。最基础的是贪心拼接（Greedy Packing），即按顺序将文档首尾相连，直到达到最大长度（如4096），超出部分截断并放入下一个序列。这种方法实现简单，但可能在序列末尾留下少量Padding。进阶方法是最佳适应拼接（Best-fit Packing），将文档长度组合视为一维背包问题，通过算法寻找最优组合，最大化填充率，几乎消除Padding。此外，从模型结构角度，还有带文档隔离的拼接方式，即在拼接后，为每个序列生成特定的Attention Mask和重置Position ID，确保同一序列内的不同文档在自注意力计算时互不干扰，保证训练质量。

### 案例模拟

面试官追问：你们在项目中具体使用了哪种拼接策略？ 回答：在我们的百亿模型预训练中，我们采用了贪心拼接结合EOS Token分隔的策略。虽然最佳适应拼接能减少约2%的Padding，但在海量数据（TB级别）预处理时，复杂的组合算法会带来显著的CPU开销和时间延迟。我们权衡后认为，简单的贪心拼接配合高效的C++数据加载器（如Megatron的MMap机制）性价比最高。同时，我们没有使用复杂的跨文档Mask，而是依赖模型自身学习EOS Token的截断语义，实际评测中并未发现明显的跨文档幻觉问题。

### 12. 三、基于lora的llama2二次预训练 的思想是什么？

- 冻结预训练主干权重，保留原有通用特征表示
- 引入低秩矩阵旁路，学习领域增量知识分布
- 词表扩充与全量训练Embedding，实现语义对齐

### 详细解答

基于LoRA的二次预训练的核心思想是“冻结通用知识，低秩学习增量知识”。结论是：它利用了模型参数更新的低秩特性，在不破坏原有表征的前提下实现高效的知识迁移。原理上，大模型在海量数据上预训练后，其权重矩阵已经形成了一个稳健的通用语义空间。当我们需要注入中文或垂直领域知识时，不需要大幅度改变这些基础权重。LoRA的思想是在原有的线性层旁边增加一个旁路（由降维矩阵A和升维矩阵B组成），训练时只更新A和B。前向传播时，将旁路输出与原输出相加。此外，一个关键思想是“部分全量，部分LoRA”：对于扩充的新词表，其Embedding层和输出层（LM Head）必须全量参与训练，以确保新词向量能快速融入原有的高维语义空间中。

### 案例模拟

面试官追问：为什么扩充词表后，Embedding层不能用LoRA，必须全量训练？ 回答：在我们的实践中发现，LoRA本质上是对现有权重的微小扰动。扩充的新词（如大量中文汉字）在初始化时通常是随机的或基于启发式的，它们与原有语义空间的距离较远。如果只用LoRA更新Embedding，由于低秩矩阵的表达能力有限，新词很难快速、准确地移动到正确的语义位置。因此，必须解冻整个Embedding层和LM Head层进行全量参数更新，而Transformer的中间层则使用LoRA，这样既保证了新词的学习效率，又控制了整体显存。
