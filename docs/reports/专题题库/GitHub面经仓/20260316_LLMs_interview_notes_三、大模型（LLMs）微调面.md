## 专题总结

## 本专题考什么
大模型微调（SFT）是目前AI算法岗最核心的落地能力。面试官通过本专题主要考察候选人是否真正“动手炼过丹”，而非纸上谈兵。核心考点包括：
- **工程全链路能力**：SFT的完整生命周期（数据准备、模型选型、训练配置、评估）。
- **算力与显存评估**：全参数微调的底层显存计算逻辑（参数、梯度、优化器状态、激活值）。
- **分布式训练技术**：DeepSpeed/Megatron/FSDP 等框架的选型与底层逻辑（ZeRO系列、3D并行）。
- **数据工程**：SFT数据的质量控制、配比、格式与“Data is King”的理解。
- **训练Debug与调优**：Loss异常分析、超参设置、训练稳定性保障。

## 答题主线
回答微调类问题，建议遵循**“宏观流程 -> 数据为王 -> 资源与框架匹配 -> 微观Debug”**的逻辑主线：
1. **宏观流程**：先确立全局观，说明SFT不仅是`trainer.train()`，而是“数据清洗 -> 资源评估 -> 框架选型 -> 训练监控 -> 评测”的闭环。
2. **数据为王**：强调“Garbage in, garbage out”，突出在数据质量、多样性和指令构建上的细节把控。
3. **资源与框架匹配**：根据模型参数量（如7B/70B）和可用硬件，推导显存需求，进而选择合适的分布式策略（如单机多卡用FSDP/ZeRO，多机多卡结合Megatron）。
4. **微观Debug**：遇到异常（如Loss突降/飞升），展现从数据（Shuffle/分布）、超参（LR/Warmup）、系统（Checkpoint加载）三个维度排查的实战经验。

## 代表题答法

### 问题1：常见 SFT 的开发流程是如何的？
**答题要点：**
展现端到端的工程能力，分为五个阶段：
1. **数据准备**：收集开源/业务数据，进行清洗（去重、脱敏、过滤低质量）、格式化（如转为ChatML格式），并按任务类型配比。
2. **模型与环境准备**：选择合适的Base Model（如Llama3、Qwen），评估显存需求，配置分布式训练环境（DeepSpeed/FSDP）。
3. **训练配置**：设定超参数（学习率、Batch Size、Epoch、Warmup等），选择微调方式（全参数或LoRA），开启Gradient Checkpointing等显存优化技术。
4. **训练与监控**：启动训练，监控Loss曲线、梯度范数（Gradient Norm）、吞吐量（Tokens/sec）和显存占用，及时处理Loss Spike。
5. **模型评估**：结合客观评测集（MMLU, CEval等）与主观测试（人工抽检业务Case），防止灾难性遗忘。

### 问题2：训练数据要注重什么？
**答题要点：**
突出“质量 > 数量”的核心理念：
1. **数据质量**：剔除格式错误、逻辑混乱、包含有害信息的数据。高质量的人工标注数据（如几十万条）远胜于千万级粗糙爬虫数据。
2. **数据多样性**：覆盖不同的任务类型（问答、摘要、代码、推理）、不同的长度分布、不同的语气指令，防止模型过拟合单一模式。
3. **Prompt与Response的分布**：确保指令的丰富度，避免模型产生“捷径学习”（Shortcut learning）。
4. **Loss计算范围**：明确在SFT阶段，**只对Response部分计算Loss**，Prompt部分不参与梯度回传（通过设置label为-100实现）。

### 问题3：如果想要在某个模型基础上做全参数微调，究竟需要多少显存？
**答题要点：**
必须给出清晰的计算公式（以AdamW优化器、混合精度训练为例）：
1. **模型状态显存（固定大头）**：每个参数通常需要 **16 Bytes**。
   - 模型参数（bf16/fp16）：2 Bytes
   - 梯度（bf16/fp16）：2 Bytes
   - 优化器状态（Adam）：12 Bytes（fp32的动量 4B + fp32的方差 4B + fp32的参数备份 4B）。
   - *举例：7B模型全参数微调，基础显存需 7B * 16 = 112 GB。*
2. **激活值（Activations）**：与Batch Size、序列长度（Seq Length）成正比。必须提到**Gradient Checkpointing（梯度检查点）**技术，开启后可用计算时间换取显存空间，大幅降低激活值显存占用。
3. **优化方案**：引入DeepSpeed ZeRO-1/2/3，将优化器状态、梯度、参数切分到多张卡上，从而在单卡显存有限时完成训练。

### 问题4：分布式训练框架选择？
**答题要点：**
根据集群规模和模型大小进行选型：
1. **DeepSpeed**：最常用的SFT框架。依赖ZeRO系列技术，适合中小规模集群（单机多卡或少量多机）。ZeRO-2切分梯度和优化器，性价比最高；ZeRO-3切分参数，适合极大模型但通信开销大。
2. **Megatron-LM**：适合超大规模集群和超大模型（如千亿参数）的预训练/全参数微调。支持3D并行（张量并行TP + 流水线并行PP + 数据并行DP），算力利用率（MFU）极高，但开发和配置门槛高。
3. **PyTorch FSDP**：PyTorch原生的全分片数据并行，类似ZeRO-3，集成度高，代码侵入性小，目前在很多中型SFT任务中正在替代DeepSpeed。

### 问题5：大模型 SFT 过程中，为什么会出现第二个epoch的时候loss会突然下降问题？
**答题要点：**
这是非常经典的实战Debug题，需从以下几个维度排查：
1. **Data Shuffle失效**：最常见原因。如果Dataloader没有正确设置全局Shuffle，或者随机种子在每个epoch被重置为相同值，导致第二个epoch喂给模型的数据顺序与第一个epoch完全一致，模型产生了记忆效应，Loss会骤降。
2. **学习率调度器（LR Scheduler）设置问题**：如果使用了带有Warmup的策略，可能在第一个epoch结束时学习率刚好达到峰值或发生剧烈衰减（如Step LR配置错误），导致Loss突变。
3. **断点续训（Checkpoint）加载Bug**：如果是从第一个epoch的checkpoint恢复训练，可能只加载了模型权重，**没有正确加载优化器状态（Optimizer States）**或学习率调度器状态，导致动量丢失，产生类似重新Warmup的效果。

### 问题6：LLMs 训练时有哪些有用的建议？
**答题要点：**
分享高价值的实战“避坑”经验：
1. **精度选择**：强烈建议使用 **BF16** 而非 FP16，BF16的指数位更宽，能有效避免大模型训练中常见的梯度溢出（Overflow）导致的Loss NaN问题。
2. **显存优化**：永远默认开启 **Gradient Checkpointing**；如果显存依然紧张，考虑使用 FlashAttention-2 加速并省显存。
3. **训练稳定性**：监控梯度范数（Gradient Norm），设置合理的 Gradient Clipping（如1.0），防止梯度爆炸。
4. **敏捷验证**：在跑全量数据前，先用极小的数据集（如100条）跑几步，验证Loss能否收敛到接近0（Overfit测试），确保整个数据流、Loss计算（特别是Ignore Index设置）没有Bug。

为了解决这个问题通常在领域训练的过程中加入通用数据集。那么这个比例多少比较合适呢？目前还没有一个准确的答案。主要与领域数据量有关系，当数据量没有那么多时，一般领域数据与通用数据的比例在1:5到1:10之间是比较合适的。

#### 示例 2. 7 进行SFT操作的时候，基座模型选用Chat还是Base?

在进行SFT实验的时候，大模型选用Chat还是Base作为基座，需要根据SFT的数据量进行决定。如果你只拥有小于10k数据，建议你选用Chat模型作为基座进行微调；如果你拥有100k的数据，建议你在Base模型上进行微调。
