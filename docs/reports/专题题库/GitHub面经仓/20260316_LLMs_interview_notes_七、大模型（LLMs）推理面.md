## 专题总结

## 本专题考什么
- **自回归推理机制**：考察对大模型 Prefill（预填充）和 Decode（解码）两阶段底层差异的理解。
- **显存管理与瓶颈**：考察对模型权重、KV Cache 显存占用规律的掌握，以及对“显存墙（Memory Wall）”的认知。
- **硬件与算力基础**：考察对 Compute Bound（算力瓶颈）和 Memory Bound（访存瓶颈）的区分，以及 GPU/CPU 架构差异对推理性能的实际影响。
- **工程优化认知**：评估候选人是否了解主流推理框架（如 vLLM、TensorRT-LLM）的底层显存分配策略（如显存池化、PagedAttention）。

## 答题主线
- **先拆解阶段**：任何推理性能问题，必须先拆分为 **Prefill（计算密集型）** 和 **Decode（访存密集型）** 两个阶段来分析。
- **抓核心瓶颈**：明确大模型推理的痛点通常不在于“算力不够”，而在于“访存带宽不足”和“显存容量受限”。
- **结合底层工程**：用框架底层的内存管理机制（如预分配、缓存池）来解释表层的资源占用现象，展现工程深度。

## 代表题答法

### 1. 为什么大模型推理时显存涨的那么多还一直占着？
**答题要点：**
1. **为什么涨得多（KV Cache 机制）**：
   大模型推理采用自回归（Auto-regressive）模式。为了避免重复计算历史 Token 的 Attention，推理时会缓存之前所有 Token 的 Key 和 Value 张量，即 **KV Cache**。随着生成序列长度的增加，KV Cache 的显存占用会呈线性快速增长。如果是多并发（Batching）场景，KV Cache 的体积甚至会超过模型权重本身。
2. **为什么一直占着（显存池化与预分配策略）**：
   这属于底层框架的内存管理机制。无论是 PyTorch 的 `caching_allocator`，还是专门的推理框架（如 vLLM），为了避免频繁调用底层 `cudaMalloc` 和 `cudaFree` 带来的巨大时间开销和显存碎片化，都会采用**显存池（Memory Pool）**技术。
   - 框架在初始化或推理高峰时会**预先申请大块显存**。
   - 当一个推理请求结束后，这部分 KV Cache 占用的显存会在逻辑上被释放（标记为可用），但**在物理层面并不会还给操作系统/GPU**，而是留在显存池中供下一个请求复用。因此通过 `nvidia-smi` 观察，显存占用会一直居高不下。

### 2. 大模型在gpu和cpu上推理速度如何？
**答题要点：**
1. **核心结论**：GPU 的推理速度呈数量级碾压 CPU。但在大模型场景下，解释其原因必须切中 **“访存带宽（Memory Bandwidth）”** 这个核心，而不仅仅是算力。
2. **GPU 为什么快（高带宽 + 高并发）**：
   - **Prefill 阶段（Prompt 处理）**：属于 Compute Bound（算力瓶颈），主要进行大矩阵乘法。GPU 拥有海量流处理器，能极大加速这一阶段。
   - **Decode 阶段（逐字生成）**：属于 Memory Bound（访存瓶颈）。每生成一个 Token，都需要把庞大的模型权重和不断增长的 KV Cache 从显存搬运到计算单元。GPU（如 A100/H100）拥有 HBM（高带宽内存），带宽高达 2~3 TB/s，能快速完成数据搬运。
3. **CPU 为什么慢（致命的访存瓶颈）**：
   - CPU 的内存带宽（通常为 DDR4/DDR5，几十到一百多 GB/s）远低于 GPU 的 HBM。
   - 在 Decode 阶段，CPU 计算一个 Token 的时间绝大部分都浪费在**等待数据从内存加载到寄存器**的过程中。即使 CPU 的核心数再多，由于内存通道拥挤，算力也根本发挥不出来，导致生成速度（Tokens/second）极低。
