## 专题总结

## 本专题考什么
- **RAG 的核心动机与价值**：考察候选人是否真正理解大模型（LLM）的局限性，以及 RAG（检索增强生成）如何低成本、高效地解决这些痛点。
- **RAG 的完整工程链路**：考察对“数据接入-向量化-检索-生成”这一标准 Pipeline 的熟悉程度。
- **非结构化数据处理能力（核心拉分项）**：真实业务中 RAG 的效果上限取决于数据质量。重点考察对 PDF 解析、复杂版面（如表格）还原、以及文本分块（Chunking）策略的工程实践经验。

## 答题主线
- **先说痛点，再给方案**：从 LLM 的“幻觉、知识滞后、私有数据缺失”切入，顺理成章引出 RAG 的定义与外挂知识库的必要性。
- **宏观串联，微观深挖**：先用一句话概括 RAG 的三大阶段（Indexing, Retrieval, Generation），然后主动将话题引导至最体现工程深度的“数据预处理”环节。
- **强调“语义完整性”**：在回答 PDF 解析、表格识别、文本分块时，核心逻辑都要围绕“如何最大程度保留原始文档的结构与语义，从而提高检索的召回率和准确率”来展开。

## 代表题答法

### 问题1：什么是 RAG？为什么大模型已经很强了，还需要外挂（向量）知识库？
**答题要点：**
1. **RAG 定义**：RAG（Retrieval-Augmented Generation）即检索增强生成，是一种结合了信息检索技术与 LLM 生成能力的技术架构。它在生成回答前，先从外部知识库检索相关信息作为上下文。
2. **LLM 的四大痛点（为什么需要外挂库）**：
   - **幻觉问题（Hallucination）**：LLM 容易一本正经地胡说八道，RAG 通过提供外部事实依据来锚定生成内容。
   - **知识滞后**：LLM 的知识停留在训练截断时间，重新微调成本极高，外挂知识库可以实现知识的实时热更新。
   - **数据隐私与垂直领域缺失**：企业内部私有数据无法用于开源模型训练，通过本地向量库可以安全地利用私有资产。
   - **可解释性与溯源**：RAG 能够提供引用来源（Reference），增加回答的业务可信度。

### 问题2：请简述基于 LLM + 向量库的文档对话完整思路（Pipeline）是怎样的？
**答题要点：**
标准 RAG 流程分为三个核心阶段：
1. **数据索引阶段（Indexing）**：
   - **解析与清洗**：将文档（PDF/Word/HTML）提取为纯文本。
   - **分块（Chunking）**：将长文本切分为适合模型处理的文本块。
   - **向量化（Embedding）**：调用 Embedding 模型将文本块转化为高维向量。
   - **入库**：将向量及对应的元数据（Metadata）存入向量数据库（如 Milvus, FAISS）。
2. **检索阶段（Retrieval）**：
   - 用户输入 Query，使用相同的 Embedding 模型将其向量化。
   - 在向量库中进行相似度计算（如余弦相似度），召回 Top-K 个最相关的文本块。
3. **生成阶段（Generation）**：
   - 将用户的 Query 与召回的 Top-K 文本块组装成 Prompt 模板（如：“请根据以下参考信息回答问题：[参考信息] + [Query]”）。
   - 将 Prompt 喂给 LLM，生成最终回答。

### 问题3：在 RAG 中，为什么需要对 PDF 进行专门解析？为什么表格识别任务尤为重要？
**答题要点：**
1. **PDF 解析的必要性**：PDF 是一种基于视觉版面的格式，内部数据是绝对坐标上的字符集合，缺乏逻辑结构（如段落、标题、阅读顺序）。直接暴力提取会导致段落错乱、乱码，严重破坏文本语义，直接导致后续检索失败。
2. **表格识别的痛点与价值**：
   - **痛点**：常规的文本提取会将表格按行或列强行拼接成纯文本，丢失了表头与单元格的二维映射关系（如“张三-25岁”变成了“张三 李四 25 26”）。
   - **价值**：表格往往包含最核心的业务数据。必须通过专门的表格识别（Table Recognition/OCR）技术，将表格还原为 Markdown、HTML 或 JSON 格式，才能保留行列逻辑，确保 LLM 能够正确理解和推理表格数据。

### 问题4：为什么需要对文本进行分块（Chunking）？常见的文本分块方法有哪些？
**答题要点：**
1. **为什么需要分块**：
   - **模型限制**：Embedding 模型和 LLM 都有 Context Window（上下文窗口）长度限制，无法一次性处理整本书。
   - **检索精度（信噪比）**：Chunk 太大，包含无关信息多，向量表达会被稀释，导致检索不准；Chunk 太小，会丢失上下文语境（Context Loss）。
2. **常见的分块方法（由浅入深）**：
   - **固定长度分块（Fixed-size Chunking）**：按固定字符数（如 500 token）切分，通常设置一定的重叠度（Overlap，如 50 token）以防止句子被生硬截断。优点是简单快速，缺点是容易破坏语义完整性。
   - **基于规则/标点分块**：利用 NLTK 或 SpaCy 等工具，按句子（句号）或段落（换行符）进行切分，保证单句语义完整。
   - **递归字符分块（Recursive Character Chunking）**：LangChain 默认方法。按 `["\n\n", "\n", " ", ""]` 的优先级递归切分，尽量保持段落和句子的完整。
   - **基于文档结构分块（Markdown/HTML Header）**：根据文档的各级标题（H1, H2, H3）进行切分，最符合人类写作逻辑。
   - **语义分块（Semantic Chunking）**：计算相邻句子的 Embedding 相似度，当相似度低于某个阈值时（说明话题发生切换），在此处进行切分。这是目前较前沿的优化方向。
