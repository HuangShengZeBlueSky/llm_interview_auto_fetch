## 专题总结

## 本专题考什么
- **显存估算能力**：考察候选人是否具备对大模型参数量、数据类型（FP32/FP16/INT8）与物理显存占用之间换算的直觉与计算能力。
- **训练/推理显存结构**：考察是否清楚模型在训练和推理时，显存分别被哪些部分占用（模型权重、梯度、优化器状态、激活值、KV Cache等）。
- **显存优化策略**：考察对主流显存优化技术（时间换空间、通信换空间）的底层原理理解，如 Gradient Accumulation、Gradient Checkpointing、ZeRO 系列等。

## 答题主线
- **第一步：定基准（算参数）**：牢记“1B参数在FP16/BF16精度下占用约2GB显存”的黄金法则，以此为基点推导所有问题。
- **第二步：拆结构（找瓶颈）**：明确当前场景是推理还是训练。推理看“权重 + KV Cache”；全参训练看“权重(2) + 梯度(2) + 优化器状态(12) = 16 bytes/参数”及激活值；微调看具体方案（如LoRA）。
- **第三步：给方案（做优化）**：针对显存瓶颈对症下药。Batch Size太大导致OOM用梯度累加；激活值太大用梯度检查点；模型状态太大用ZeRO/FSDP；单卡装不下用量化（QLoRA）或模型并行。

## 代表题答法

### 问题1：大模型大概有多大，模型文件有多大?
**答题要点：**
模型大小和文件大小直接取决于**参数量**和**存储精度**。
1. **计算公式**：显存占用/文件大小 = 参数量 × 每个参数占用的字节数。
2. **精度对应关系**：
   - FP32（单精度）：4 Bytes / 参数
   - FP16/BF16（半精度）：2 Bytes / 参数（目前大模型最常用的默认精度）
   - INT8（量化）：1 Byte / 参数
   - INT4（量化）：0.5 Byte / 参数
3. **举例说明**：以 LLaMA-7B 为例，70亿参数。如果以 FP16 精度保存，模型文件大小约为 $7 \times 10^9 \times 2$ Bytes $\approx 14$ GB。加载到显存中作为纯静态权重，也需要约 14GB 显存。

### 问题2：能否用4 * v100 32G训练vicuna 65b？
**答题要点：**
**结论：绝对不能进行常规训练，即使是推理或极限微调也极其困难。**
1. **算总显存**：4张 V100 32G，总物理显存为 $4 \times 32 = 128$ GB。
2. **算模型静态占用**：Vicuna 65B 在常规 FP16 精度下，仅加载模型权重就需要 $65 \times 2 = 130$ GB 显存。**130GB > 128GB，连模型权重都无法完全加载入显存。**
3. **算训练占用**：如果是全参数训练（Adam优化器），每个参数需要 16 Bytes（权重2+梯度2+优化器状态12），总共需要 $65 \times 16 = 1040$ GB 显存，远超 128GB。
4. **补充极限方案（展现深度）**：除非使用 QLoRA（4-bit量化加载基础模型，占用约 32.5GB）配合 CPU Offload 技术，将优化器状态和梯度卸载到内存中，理论上可以跑通，但 V100 不支持高效的 INT4 计算，且跨 PCIe 通信极慢，毫无实际工程价值。

### 问题3：介绍一下 gradient accumulation 显存优化方式？
**答题要点：**
**核心思想：时间换空间，解决大 Batch Size 导致的显存溢出（OOM）问题。**
1. **痛点**：训练时 Batch Size 越大，前向传播产生的激活值（Activations）占用的显存就越大。受限于单卡显存，往往只能设置很小的 Batch Size，导致模型收敛不稳定。
2. **原理**：将一个大的 Global Batch 拆分成多个小的 Micro Batch。
   - 每次取一个 Micro Batch 进行前向（Forward）和反向（Backward）传播，计算出梯度。
   - **关键点**：算出的梯度不立即用于更新权重，而是将其累加（Accumulate）起来。
   - 循环执行 $N$ 次后，累加的梯度相当于一个大 Batch Size 的梯度，此时再执行一次优化器步进（Optimizer Step）更新权重，并清空梯度（Zero Grad）。
3. **效果**：显存占用降低到单次 Micro Batch 的水平，但实现了等效大 Batch Size 的训练效果。

### 问题4：介绍一下 gradient checkpointing 显存优化方式？
**答题要点：**
**核心思想：计算换空间（也叫激活重计算），解决深层网络中激活值占用显存过大的问题。**
1. **痛点**：在标准反向传播中，为了计算梯度，必须将前向传播产生的所有中间层激活值（Activations）保留在显存中。对于层数极深的大模型，激活值占用的显存往往远超模型权重本身。
2. **原理**：
   - **前向传播时**：不保存所有的中间激活值，而是只在网络中每隔几层设置一个“检查点（Checkpoint）”，仅保存这些检查点处的激活值。
   - **反向传播时**：当需要用到被丢弃的中间激活值来计算梯度时，从距离它最近的那个“检查点”开始，**重新执行一次前向传播**来临时计算出这些缺失的激活值。
3. **效果**：可以将激活值的显存占用从 $O(N)$ 降到 $O(\sqrt{N})$（$N$为网络层数），大幅节省显存，代价是增加了约 20%~30% 的额外前向计算时间。
