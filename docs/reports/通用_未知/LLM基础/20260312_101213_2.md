# 题目1：数据集是怎样做扩充的？

**1. 题目描述**：在训练大模型（如SFT阶段）时，如何对有限的数据集进行扩充和增强。

**2. 前置基础知识**：
解答此题需了解传统NLP的数据增强（Data Augmentation）以及大模型时代特有的“合成数据”（Synthetic Data）生成范式。

**3. 核心解答**：
数据集扩充通常分为传统方法和基于LLM的生成方法：
*   **传统数据增强**：包括回译（中翻英再翻中）、同义词替换、随机插入/删除（EDA技术）。这些方法成本低，但容易破坏语义，目前在大模型微调中占比较小。
*   **基于LLM的合成数据（主流）**：
    *   **Self-Instruct**：利用少量高质量的种子Prompt，让强大的LLM（如GPT-4）自动生成更多类似的任务指令和回复。
    *   **Evol-Instruct（指令进化）**：在现有指令的基础上，通过让LLM增加约束条件、深化逻辑、增加多步推理等方式，将简单指令“进化”为复杂指令，提升数据难度。
    *   **角色扮演/多Agent对话构造**：设定不同的Persona（系统提示词），让两个LLM互相进行多轮对话，从而低成本收集多轮对话数据。
*   **开源数据集混合与改写**：将现有的开源数据集（如ShareGPT, FLAN）进行清洗、格式统一，或使用LLM对其进行改写（如改变语气、提取摘要），以适应特定领域的风格。

**4. 引申思考/面试追问**：
*   *追问*：使用大模型生成合成数据时，怎么保证数据的质量，防止模型学到幻觉？
*   *思路*：需要建立数据清洗管线。可以通过训练一个Reward Model进行打分过滤；使用规则（长度、特定关键词）过滤；或者让另一个LLM作为裁判（LLM-as-a-Judge）评估一致性和事实性；最后辅以人工抽检。

# 题目2：为什么要SFT后还要使用DPO进行训练？

**1. 题目描述**：解释在监督微调（SFT）之后，为什么还需要引入直接偏好优化（DPO）阶段。

**2. 前置基础知识**：
必须掌握SFT（监督微调）的局限性，以及人类偏好对齐（Alignment）的核心目的。

**3. 核心解答**：
*   **SFT的局限性（Behavior Cloning）**：SFT本质上是行为克隆，它通过Next Token Prediction让模型学会“怎么说话”（如掌握对话格式、特定领域的语气）。但SFT只能让模型模仿训练集，**无法区分“好回答”和“更好/更安全的回答”**，且存在暴露偏差（Exposure Bias）。
*   **DPO的作用（偏好对齐）**：DPO（Direct Preference Optimization）引入了对比学习的思想。通过提供成对的样本（Chosen正样本和Rejected负样本），DPO不仅告诉模型“应该输出什么”，还明确告诉模型“不应该输出什么”。
*   **最终收益**：经过DPO训练，模型能显著减少幻觉，拒绝回答有害问题，提升逻辑严密性，使其价值观和输出偏好真正对齐人类的需求，而这是单纯靠SFT很难做到的。

**4. 引申思考/面试追问**：
*   *追问*：DPO和传统的RLHF（基于PPO算法）相比，有什么优缺点？
*   *思路*：优点是DPO极其轻量、稳定，不需要训练单独的Reward Model，也不需要复杂的强化学习PPO算法，直接在语言模型上优化交叉熵即可；缺点是DPO对偏好数据的质量极其敏感，且在处理超长文本或极度复杂的推理对齐时，泛化能力可能不如PPO。

# 题目3：对于难样本是怎么解决的？

**1. 题目描述**：在模型训练过程中，如何处理模型难以学习或容易预测错误的“难样本”（Hard Samples）。

**2. 前置基础知识**：
需了解深度学习中的长尾分布问题、对比学习中的Hard Negative（难负样本）以及损失函数的权重调节。

**3. 核心解答**：
解决难样本问题通常从数据、训练策略和损失函数三个维度入手：
*   **数据层面**：
    *   **过采样与数据重写**：对难样本进行过采样（增加出现频率），或使用LLM对难样本的Prompt进行改写、拆解，提供更详尽的Chain-of-Thought（思维链）过程，帮助模型理解。
    *   **挖掘难负样本**：在DPO或对比学习中，刻意构造“看起来很像正确答案但存在微小事实错误”的样本作为Rejected，逼迫模型学习细粒度的鉴别能力。
*   **训练策略层面（课程学习 Curriculum Learning）**：
    *   采用“先易后难”的训练策略，先用简单样本让模型收敛到一个较好的空间，再逐渐增加难样本的比例，防止模型一开始就陷入局部最优或梯度爆炸。
*   **损失函数层面**：
    *   引入类似Focal Loss的机制，降低高置信度（容易）样本的权重，放大低置信度（难）样本在Loss中的占比，强迫模型关注难样本。

**4. 引申思考/面试追问**：
*   *追问*：在构建DPO的偏好数据时，如何高效地自动构造难负样本（Hard Rejected）？
*   *思路*：可以对正确答案（Chosen）进行微小篡改，如修改关键实体、反转逻辑词（把“是”改成“否”）；或者调高LLM的Temperature参数进行多次采样，选取其中流畅但事实错误的回答作为难负样本。

# 题目4：如何实现Function call的功能？

**1. 题目描述**：大模型如何实现调用外部工具（Function Call / Tool Use）的能力。

**2. 前置基础知识**：
需掌握Prompt Engineering的结构化输出（JSON）、Agent的基础工作流。

**3. 核心解答**：
实现Function Call通常包含数据微调和推理工程两个完整的闭环：
*   **1. 训练/微调阶段**：需要构造包含工具调用的SFT数据。数据格式通常包括：System Prompt（包含可用API的描述、参数类型、JSON Schema格式） -> 用户提问 -> 模型输出特定格式的工具调用指令（如 `&lt;tool_call&gt; {"name": "get_weather", "args": {"city": "北京"}}`）。通过SFT让模型学会根据需求输出正确的JSON。
*   **2. 推理阶段（工作流）**：
    *   **触发**：将可用工具列表注入System Prompt，输入用户Query。
    *   **解析**：模型生成文本，后台代码正则拦截或解析出模型要求调用的函数名及参数。
    *   **执行**：在本地环境或服务器实际执行该API/代码，获取结果（如天气数据）。
    *   **回传**：将执行结果拼接成特定的角色（如 `role: tool` 或 `role: observation`）再次作为上下文输入给模型，模型据此生成最终的自然语言回答。

**4. 引申思考/面试追问**：
*   *追问*：如果模型在Function Call时出现了“幻觉”，虚构了不存在的API参数，该怎么解决？
*   *思路*：在工程上，执行API前加一层参数校验（Schema Validation），如果参数错误，将错误信息（Error Message）作为Observation传回给模型，让模型进行Self-Correction（自我修正）；在模型端，需要增加负样本SFT数据，教导模型在信息不足时提问，而不是瞎编参数。

# 题目5：为什么RAG的效果用正确率来表示？

**1. 题目描述**：解释在检索增强生成（RAG）系统中，为什么倾向于使用“正确率”等指标而非传统生成指标。

**2. 前置基础知识**：
需了解RAG的核心诉求（解决幻觉、提供事实），以及传统NLG指标（BLEU, ROUGE）的局限性。

**3. 核心解答**：
*   **RAG的核心定位**：RAG本质上解决的是“知识密集型”和“事实问答”任务。用户使用RAG就是为了获取准确的外部知识，因此**事实正确性（Factuality）**是第一位的。
*   **传统生成指标的失效**：传统的BLEU、ROUGE指标只关注模型输出与参考答案的“字面N-gram重合度”。在LLM时代，模型可以用完全不同的词汇和句式表达相同的意思（此时BLEU很低，但答案是对的），或者只改动了一个关键否定词（此时BLEU很高，但答案是完全错误的）。
*   **正确率的业务价值**：使用正确率（通常通过人工标注，或使用LLM-as-a-Judge评估答案是否准确回答了问题且忠实于检索到的文档）能够直接反映RAG系统在实际业务中解决问题的能力，是评估RAG是否产生幻觉的最直观标准。

**4. 引申思考/面试追问**：
*   *追问*：除了端到端的正确率，业界还有哪些专门评估RAG的框架和维度？
*   *思路*：可以提到RAGAS或TruLens框架。核心评估RAG的三元组：**上下文相关性**（检索召回是否准确）、**忠实度/无幻觉率**（回答是否完全基于检索内容，没有夹带私货）、**答案相关性**（回答是否直接解决了用户的Query）。

# 题目6：RAG的文档是怎么划分的？基于什么规则划分？

**1. 题目描述**：RAG系统中长文档的切分（Chunking）策略和规则是什么。

**2. 前置基础知识**：
文本分块（Chunking）的必要性（受限于Embedding模型输入长度和LLM上下文窗口），以及语义完整性。

**3. 核心解答**：
文档划分通常遵循从粗到细、从规则到语义的几种策略：
*   **基于固定长度与重叠（Fixed-size with overlap）**：最基础的规则。设定固定的Token数（如512 tokens），并保留一定比例的重叠（如50 tokens）以防止关键句子被从中间截断。
*   **基于标点与自然段（Rule-based）**：优先按照自然段落（`\n\n`）切分，其次按照句子结束符（句号、问号）切分。这种规则能最大程度保证基础语法和单句语义的完整。
*   **基于文档结构（Structural）**：对于Markdown、HTML或PDF，利用解析器识别H1、H2标题、表格、代码块进行切分，确保同一个层级或同一个表格的数据被划分在一个Chunk内。
*   **基于语义切分（Semantic Chunking）**：利用较小的Embedding模型计算相邻句子的余弦相似度，如果相似度低于某个阈值，说明话题发生转换，则在此处进行切断。

**4. 引申思考/面试追问**：
*   *追问*：Chunk Size设置过大或过小，分别对RAG有什么影响？
*   *思路*：过小会导致语义碎片化，丢失上下文，召回的都是无意义的短句；过大会导致Chunk内部包含大量噪声，降低Embedding向量的表征精准度，且在Prompt拼接时容易超出LLM上下文限制。

# 题目7：RAG召回冗余的数据怎么解决的？

**1. 题目描述**：如何处理RAG检索阶段召回了大量高度相似、信息冗余的文档片段。

**2. 前置基础知识**：
向量检索的Top-K机制，最大边际相关性（MMR），重排（Reranking）。

**3. 核心解答**：
解决召回冗余数据主要在“检索后”和“生成前”进行处理：
*   **使用MMR（最大边际相关性）算法**：在检索时，不仅考虑文档与Query的相似度，还惩罚候选文档与已选文档之间的相似度。强制召回既与问题相关，又互相之间具有多样性的文档，从源头减少冗余。
*   **引入Reranker（重排模型）**：使用Cross-Encoder模型对召回的Top-K文档进行重新打分排序。Reranker能更精准地判断文档是否真正包含答案，从而可以安全地截断后面的冗余文档（如Top-20截断为Top-5）。
*   **LLM信息压缩/摘要**：在将召回文档喂给主LLM之前，先使用一个小参数LLM或摘要提示词，对召回的多个文档进行信息去重和提取，提炼出核心事实后再输入给主LLM。
*   **入库前去重**：在文档Embedding入库前，使用MinHash或SimHash等算法对语料进行文本级去重，防止同一份内容被切分成多个相同的Chunk入库。

**4. 引申思考/面试追问**：
*   *追问*：MMR算法中的 $\lambda$ 参数是做什么的？怎么调？
*   *思路*：$\lambda$ 用于平衡相关性和多样性。公式为 $Score = \lambda \cdot Sim(Q, D) - (1-\lambda) \cdot Sim(D, D_{selected})$。$\lambda$ 越接近1，越看重相关性（可能冗余）；越接近0，越看重多样性。通常设为0.5-0.7左右。

# 题目8：有哪些可以优化RAG的方法？

**1. 题目描述**：列举提升RAG系统整体效果的高阶优化策略。

**2. 前置基础知识**：
RAG全链路流程：Query理解 -> 检索 -> 排序 -> 生成。

**3. 核心解答**：
可以从RAG的四个核心阶段进行全链路优化：
*   **1. Query优化**：
    *   **Query Rewrite**：用LLM将用户的口语化、指代不清的问题重写为标准检索词。
    *   **HyDE（假设性文档嵌入）**：让LLM先凭空生成一个假设性答案，用该答案的向量去检索真实文档，解决Query和Document特征空间不一致的问题。
*   **2. 索引与数据优化**：
    *   **多级索引（Parent-Child Retriever）**：切分细粒度Chunk用于检索，但召回时返回其所属的粗粒度父文档，兼顾检索精度与上下文完整性。
    *   **元数据过滤（Metadata Filtering）**：为Chunk打上时间、分类等标签，检索前先做硬过滤。
*   **3. 检索与排序优化**：
    *   **混合检索（Hybrid Search）**：结合BM25（关键词匹配）和Dense Embedding（语义向量），优势互补。
    *   **引入BGE-Reranker**：必选项，大幅提升最终Top-K的准确率。
*   **4. 生成优化**：
    *   **Prompt工程**：加入“如果你不知道，请回答不知道，不要编造”的防幻觉指令。
    *   **微调生成模型（RAFT）**：特定领域下，使用特定域的数据微调LLM，使其更懂得如何阅读和引用召回的文档。

**4. 引申思考/面试追问**：
*   *追问*：HyDE方法有什么明显的缺点？
*   *思路*：HyDE极度依赖LLM的Zero-shot能力。如果用户询问的是极其垂直且冷门的领域，LLM生成的假设答案可能充满严重的幻觉，导致生成的向量完全偏离正确方向，反而拉低检索准确率。此外，多了一次LLM生成，增加了响应延迟。

# 题目9：Decode-only和encoder-only有什么区别？

**1. 题目描述**：对比Transformer架构中Decoder-only和Encoder-only的核心差异。

**2. 前置基础知识**：
Transformer架构，Self-Attention机制，掩码（Mask）机制。

**3. 核心解答**：
两者的核心区别体现在**注意力机制**、**训练目标**和**适用任务**上：
*   **注意力机制（核心）**：
    *   **Encoder-only（如BERT）**：采用**双向注意力（Bidirectional Attention）**。在计算某个Token的表征时，可以看到它前面和后面的所有Token，没有Mask遮挡（除了Padding Mask）。
    *   **Decoder-only（如GPT）**：采用**单向/因果注意力（Causal Attention）**。引入了下三角Mask矩阵，计算当前Token时，只能看到它本身及它之前的Token，严禁“偷看”未来的Token。
*   **训练目标**：
    *   **Encoder-only**：通常使用掩码语言建模（MLM，如挖空填词）或下一句预测（NSP）。
    *   **Decoder-only**：使用自回归的下一词预测（Next Token Prediction, NTP）。
*   **适用任务**：
    *   **Encoder-only**：擅长自然语言理解（NLU）任务，如文本分类、命名实体识别（NER）、情感分析。
    *   **Decoder-only**：擅长自然语言生成（NLG）任务，如文本续写、对话问答、机器翻译。

**4. 引申思考/面试追问**：
*   *追问*：既然Encoder能双向看，理解能力更强，为什么不能用来做生成任务？
*   *思路*：因为生成任务本质上是自回归的（逐步生成）。如果用Encoder，每次生成一个新词，都需要把前面所有的词重新输入模型计算一遍全局注意力，计算复杂度极高。而Decoder有因果Mask，天生契合自回归，且可以利用KV Cache复用历史计算，极大提升生成效率。

# 题目10：为什么现在的生成大模型都使用Decoder-only？

**1. 题目描述**：解释当前主流大语言模型（如GPT, LLaMA, Qwen）全部采用Decoder-only架构的深层原因。

**2. 前置基础知识**：
Scaling Law（缩放定律），Zero-shot泛化，KV Cache加速。

**3. 核心解答**：
业界抛弃Encoder-Decoder（如T5）全面转向Decoder-only，主要基于以下原因：
*   **1. 任务的统一性与Zero-shot能力**：Next Token Prediction（NTP）是一个极其优美且通用的任务。通过Prompt Engineering，所有的NLP理解任务（分类、抽取）都可以被重新统一为生成任务。Decoder-only在NTP任务上展现出了极强的Zero-shot/Few-shot泛化能力。
*   **2. Scaling Law表现更优**：研究（如OpenAI的论文）表明，随着模型参数量和数据量的指数级增加，Decoder-only架构的Loss下降最为稳定且没有天花板。而Encoder-Decoder架构在参数扩大时，性能提升的边际收益递减。
*   **3. 工程效率与KV Cache**：在推理生成时，Decoder-only只需要缓存前面Token的K和V向量（KV Cache），计算极其高效。而Encoder-Decoder在生成时，Decoder需要不断与Encoder进行Cross-Attention交互，显存访存带宽压力更大，工程优化更复杂。
*   **4. 隐式表征学习**：单向注意力强迫模型在预测下一个词时，必须把所有的历史信息压缩并理解透彻。这种“戴着镣铐跳舞”的机制，反而逼迫模型学到了更深层的世界知识和逻辑表征。

**4. 引申思考/面试追问**：
*   *追问*：Encoder-Decoder架构目前在什么领域还有不可替代的优势？
*   *思路*：在输入和输出边界非常明确的序列到序列（Seq2Seq）任务中依然有优势，比如专精的机器翻译、语音识别（ASR，如Whisper）、以及某些需要强依赖全局上下文理解的文本摘要任务。

# 题目11：图网络学习用户特征向量（场景题）

**1. 题目描述**：给出一个无标签的User-Link二分图，要求用类似BERT的自监督方法（不使用监督GCN）学习用户的特征向量。

**2. 前置基础知识**：
图表示学习（Graph Embedding），随机游走（Random Walk），Word2Vec/BERT的掩码机制。

**3. 核心解答**：
这道题考察的是将图结构转化为序列数据，并利用大语言模型（Transformer/BERT）进行自监督学习的能力。可以采用类似 **DeepWalk/Node2Vec + BERT (GraphBERT)** 的思路：
*   **步骤1：序列生成（Random Walk）**。在User-Link二分图上进行随机游走。从某个User节点出发，随机点击与其相连的Link，再从该Link随机跳到点击过它的其他User，如此反复。这样就能生成大量由 `User_A -> Link_1 -> User_B -> Link_2...` 交替组成的节点序列。这种序列隐式地捕捉了图中节点的共现关系和拓扑结构。
*   **步骤2：自监督任务构造（类似BERT）**。将生成的节点序列看作是一段“文本”（User和Link的ID就是Token）。使用BERT的掩码语言模型（MLM）机制，随机Mask掉序列中的部分User节点或Link节点。
*   **步骤3：模型训练与向量提取**。让Transformer模型通过上下文预测被Mask掉的节点ID。训练收敛后，模型底层Embedding层中对应各个User ID的向量，或者将某个User输入模型后输出的Hidden State，就是融合了图拓扑结构的User特征向量。

**4. 引申思考/面试追问**：
*   *追问*：在随机游走时，如何控制模型学到的是用户的局部相似性还是全局结构相似性？
*   *思路*：可以引入Node2Vec中的 `p` (返回参数) 和 `q` (进出参数)。调小 `p` 倾向于深度优先搜索（DFS），捕捉同构性（结构角色相似）；调小 `q` 倾向于广度优先搜索（BFS），捕捉同质性（局部相邻的节点相似）。

# 题目12：高效微调方法与LoRA原理

**1. 题目描述**：列举高效微调（PEFT）方法，详细讲解LoRA的原理及参数设置。

**2. 前置基础知识**：
全量微调的显存瓶颈，矩阵的低秩分解（Low-Rank Adaptation），反向传播。

**3. 核心解答**：
*   **常见PEFT方法**：Prompt Tuning, Prefix Tuning, P-Tuning v1/v2, LoRA, QLoRA。
*   **LoRA原理**：
    *   全量微调需要更新整个权重矩阵 $W$，显存开销极大。LoRA的假设是：模型在特定下游任务上的权重更新矩阵 $\Delta W$ 具有“极低的内在秩”。
    *   因此，LoRA在微调时**冻结原预训练模型权重 $W$**，在旁路引入两个可训练的低秩矩阵 $A$ 和 $B$。其中 $A \in \mathbb{R}^{d \times r}, B \in \mathbb{R}^{r \times k}$，秩 $r \ll d, k$。
    *   前向传播时，输出变为：$h = Wx + \frac{\alpha}{r} ABx$。
    *   训练时只更新 $A$ 和 $B$ 的参数，大幅减少了优化器状态和梯度的显存占用。
*   **LoRA参数设置**：
    *   `r` (Rank/秩)：控制更新矩阵的表达能力。通常设为 8, 16, 32 或 64。简单任务用低秩，复杂任务（如代码、数学）用高秩。
    *   `alpha` (缩放系数)：用于控制LoRA旁路输出的权重。通常设为 `r` 的1倍或2倍（如 r=8, alpha=16）。
    *   `target_modules`：应用LoRA的层。早期只作用于Attention的Q和V，现在主流做法是全线性层应用（Q, K, V, O, Up, Down, Gate），效果最好。
    *   `dropout`：防止过拟合，通常设为 0.05 或 0.1。

**4. 引申思考/面试追问**：
*   *追问*：LoRA微调后的模型，在推理时会比原模型慢吗？
*   *思路*：不会。在部署推理时，可以通过重参数化技术，将训练好的 $AB$ 矩阵直接加回到原权重矩阵中，即 $W_{new} = W + \frac{\alpha}{r} AB$。由于矩阵加法，网络结构没有发生任何改变，因此推理延迟和原模型完全一样。

# 题目13：DPO训练原因与数据集构建

**1. 题目描述**：为什么使用DPO进行训练（原因），以及DPO的数据集是如何构建的。

**2. 前置基础知识**：
*原因部分参考题目2*。此处重点解答数据集构建：偏好数据三元组 $(Prompt, Chosen, Rejected)$。

**3. 核心解答**：
*   **使用DPO的原因**：直接在语言模型上优化人类偏好，无需训练复杂的Reward Model和强化学习PPO算法，训练极其稳定，且能有效缓解幻觉、提升安全性。
*   **数据集构建方法**（核心是构造高质量的 $(x, y_w, y_l)$ 三元组）：
    *   **人工标注（高成本）**：给定一个Prompt，由不同模型生成多个回答，人类标注员根据有用性、安全性等维度进行排序，选出Chosen和Rejected。
    *   **LLM-as-a-Judge（目前主流）**：使用强大的模型（如GPT-4）作为裁判。输入Prompt和两个较弱/待微调模型的回答，让GPT-4根据严格的Prompt准则打分，区分出胜者和败者。
    *   **利用现有开源数据**：如Anthropic HH-RLHF，UltraFeedback等现成的偏好数据集。
    *   **自我对齐（Self-Play/Self-Reward）**：使用待微调模型自身，在不同Temperature下采样多个回答，利用启发式规则（如代码是否编译通过、长度是否合适）或自身打分来划分正负样本。

**4. 引申思考/面试追问**：
*   *追问*：如果在构建DPO数据集时，Chosen样本普遍比Rejected样本长很多，会导致什么问题？
*   *思路*：会导致模型产生“长度偏见（Length Bias）/ 冗长惩罚”。DPO模型会走捷径，认为“只要输出足够长就是好回答”，从而在推理时生成大量废话。解决方法是在构造数据时，控制Chosen和Rejected的长度分布尽量一致，或者在Loss函数中加入长度惩罚项。

# 题目14：RLHF中reward模型一般怎么设置？

**1. 题目描述**：在进行RLHF（基于人类反馈的强化学习）训练时，Reward模型（奖励模型）通常有哪些设置方式。

**2. 前置基础知识**：
强化学习中的奖励函数（Reward Function），PPO算法的基础逻辑。

**3. 核心解答**：
在RLHF中，Reward的设置主要分为“基于规则”和“基于模型”两类，实际中常混合使用：
*   **1. 基于规则的Reward (Rule-based)**：
    *   **适用场景**：结果具有客观确定性的任务。
    *   **设置方法**：例如代码生成（编译通过给+1，失败给-1）、数学解题（提取最终答案与标准答案对比）、格式输出（JSON解析成功给正分）。
    *   **优点**：无需训练，绝对客观，不存在Reward Hacking。
*   **2. 基于模型的Reward (Model-based RM)**：
    *   **适用场景**：开放域对话、写作、价值观对齐等难以用规则量化的场景。
    *   **设置方法**：通常使用SFT模型作为基座，将其最后的lm_head替换为一个输出标量（Scalar）的线性层。使用偏好数据集（Chosen/Rejected），通过Pairwise Ranking Loss（使得Chosen的得分高于Rejected的得分）进行训练。
    *   **缺点**：容易出现Reward Hacking（模型学会了讨好RM的特定模式，如无脑输出长句，但实际质量并不高）。
*   **3. 混合Reward (Hybrid)**：
    *   实际工业界PPO训练时，最终的Reward = RM模型打分 + 规则惩罚（如长度超限惩罚、重复生成惩罚、特定违禁词强扣分）。

**4. 引申思考/面试追问**：
*   *追问*：什么是Reward Hacking？在训练RM时如何缓解？
*   *思路*：Reward Hacking是指Policy模型找到了RM的漏洞，通过生成某些特定模式（如堆砌华丽辞藻、极长文本）来骗取高分，但实际并不符合人类偏好。缓解方法包括：引入长度惩罚；在PPO阶段计算KL散度惩罚，限制Policy模型偏离初始SFT模型太远；定期用新生成的数据重新训练更新RM。

# 题目15：llama和qwen的模型结构异同点？

**1. 题目描述**：对比开源模型LLaMA和Qwen的模型结构差异与共同点。

**2. 前置基础知识**：
现代Transformer Decoder-only架构的常见组件。

**3. 核心解答**：
两者都属于当今最优秀的开源Decoder-only架构，整体骨架非常相似，但在细节工程选择上有所不同。
*   **相同点（现代大模型标配）**：
    *   **位置编码**：都废弃了绝对位置编码，采用了**RoPE（旋转位置编码）**，有利于长度外推。
    *   **归一化层**：都采用了**RMSNorm**，且都应用了Pre-Norm结构（在Attention和MLP之前进行归一化）。
    *   **激活函数**：都使用了**SwiGLU**（门控线性单元的变体）取代了传统的ReLU/GELU。
    *   **注意力机制**：在较新的版本中（Llama2/3, Qwen1.5/2），都全面采用了**GQA（分组查询注意力）**来优化推理时的KV Cache。
*   **异同点**：
    *   **词表大小（Vocab Size）**：Qwen的词表通常极大（如152k），这使得它对中文和多语言的压缩率极高，单个Token能表达更多汉字；LLaMA原生词表较小（Llama2 32k，Llama3扩充至128k）。
    *   **Tie Embeddings（权重共享）**：Qwen通常采用Tie Word Embeddings技术，即输入的Embedding层和输出的lm_head层共享权重，以减少参数量；而LLaMA这两层权重是不共享的。
    *   **Bias（偏置项）**：LLaMA为了极简，通常去除了所有线性层的Bias；而Qwen在QKV的投影层中保留了Bias，官方认为这有助于提升模型的某些特定能力（如外推性）。

**4. 引申思考/面试追问**：
*   *追问*：Qwen的大词表相比LLaMA的小词表，在训练和推理时有什么优劣？
*   *思路*：优点是文本压缩率高，同样的上下文窗口能塞进更多中文内容，推理生成中文的速度相对更快。缺点是Embedding层和lm_head层的参数量巨大，占据了模型相当大的显存比例，且由于词表大，长尾Token在训练时可能更新不充分（所以Qwen需要Tie Embeddings来缓解）。

# 题目16：Llama3的RMSnorm、GQA和SwiGLU各自的作用分别是什么？

**1. 题目描述**：解释LLaMA3架构中RMSNorm、GQA、SwiGLU三个核心组件的作用，以及为什么替换最初Transformer中的组件。

**2. 前置基础知识**：
原始Transformer的LayerNorm、Multi-Head Attention (MHA) 和 ReLU激活函数。

**3. 核心解答**：
*   **1. RMSNorm (均方根归一化)**：
    *   **作用/区别**：取代了原始的LayerNorm。LayerNorm需要计算均值和方差（减去均值，除以方差）；RMSNorm去掉了计算均值的步骤（平移操作），只计算平方和的均方根进行缩放。
    *   **为什么用**：研究表明，归一化中的“平移不变性”对模型表现贡献极小，而“缩放不变性”才是关键。RMSNorm去除了均值计算，大幅提升了计算速度（提速约10%-50%），且模型效果不降反升。
*   **2. GQA (Grouped-Query Attention 分组查询注意力)**：
    *   **作用/区别**：取代了原始的MHA（多头注意力）。MHA每个Query头都有独立的K和V头；GQA将Query头分组，每组内的多个Query共享一对K和V头（是MHA和单头MQA的折中方案）。
    *   **为什么用**：在推理生成（Decode）阶段，MHA需要缓存所有KV头，导致极大的显存占用和内存带宽瓶颈（Memory-bound）。GQA大幅减少了KV Cache的大小，极大提升了推理速度和并发量，同时保持了接近MHA的模型表达能力。
*   **3. SwiGLU (Swish-Gated Linear Unit)**：
    *   **作用/区别**：取代了原始MLP层中的ReLU/GELU。它是由Swish激活函数和GLU（门控线性单元）组合而成，包含两个线性变换矩阵，其中一个经过Swish激活后作为“门控”来控制另一个矩阵的信息流。
    *   **为什么用**：引入了乘法门控机制，使得模型能够更细粒度地控制哪些信息应该被激活。实验证明，在同等参数量和计算量下，SwiGLU能带来更低的困惑度（Perplexity），显著提升了模型的非线性表达能力。

**4. 引申思考/面试追问**：
*   *追问*：GQA在训练阶段能提升速度吗？
*   *思路*：基本不能。在训练阶段，所有的Token是并行计算的，不存在KV Cache的逐字累积问题，计算主要受限于矩阵乘法的算力（Compute-bound）。GQA主要优化的是**推理（Decode）阶段**的显存占用和访存带宽。

# 题目17：LLM找出异常user（场景题）

**1. 题目描述**：给定一份User行为特征表格（行代表User，列代表特征），如何使用LLM找出其中的异常User。

**2. 前置基础知识**：
Tabular Data to Text（表格转文本），Agent/Code Interpreter（代码解释器），异常检测算法。

**3. 核心解答**：
LLM本质上是语言模型，直接处理高维、大规模的稠密数值表格能力极差（容易产生幻觉且受限于上下文窗口）。解决此问题应根据数据量和特征类型分层处理：
*   **方案一：LLM作为代码解释器（Agent/Code Interpreter）——最推荐、最可靠的工业级做法**
    *   **理解**：不让LLM直接看数据找异常，而是让LLM写代码找异常。
    *   **做法**：将表格的Schema（列名、数据类型、少量Sample）输入给LLM，Prompt要求：“你是一个数据科学家，请编写Python代码（如使用Pandas，结合孤立森林Isolation Forest、DBSCAN或3-Sigma规则等异常检测算法）来找出异常User，并输出其ID”。LLM生成代码后，在本地沙盒执行代码处理全量CSV，返回结果。
*   **方案二：Prompt直接推理（Few-shot）——适用于小数据、强语义特征**
    *   **理解**：如果表格只有几十行，且特征具有强语义（如“发帖内容”、“登录IP地”）。
    *   **做法**：将表格序列化为自然语言（如“User 1，今天发帖100次，登录IP为海外...”）。在Prompt中明确定义什么是“异常”（例如提供几个异常用户的Few-shot示例），让LLM逐行阅读并输出判断结果及推理过程（CoT）。
*   **方案三：LLM特征提取 + 传统分类器**
    *   **理解**：用LLM处理表格中的非结构化部分。
    *   **做法**：如果特征列包含复杂的文本行为日志，先用LLM将这些文本总结为离散的标签（如“高风险行为”）或抽取为Embedding向量，然后将其与数值特征拼接，输入到传统的机器学习模型（如XGBoost）中进行异常检测。

**4. 引申思考/面试追问**：
*   *追问*：如果强制使用方案二（让LLM直接看数据），但表格有10万行，远远超过了LLM的上下文，怎么解决？
*   *思路*：采用Map-Reduce思想。Map阶段：将数据分批（如每批100行）并行喂给LLM，让其初步筛选出疑似异常的User；Reduce阶段：将所有批次筛选出的疑似异常User汇总，再次输入给LLM进行最终的交叉比对和确认。

# 题目18：n个数的列表，找出和为target的组合（代码题）

**1. 题目描述**：算法题：给定一个包含n个数字的列表和一个目标值target，找出列表中所有和为target的组合。

**2. 前置基础知识**：
回溯算法（Backtracking），深度优先搜索（DFS），剪枝优化（Pruning）。

**3. 核心解答**：
这是一道经典的“组合总和”（Combination Sum）问题。标准解法是使用回溯法。
*面试时需与面试官确认细节：数字是否可以重复使用？列表中是否有重复数字？这里以**数字不可重复使用，且需去重（即LeetCode 40题 Combination Sum II）**为例：*

```python
def combinationSum(nums, target):
    res = []
    # 1. 必须先排序，为了后续剪枝和去重
    nums.sort()
    
    def dfs(start_index, current_sum, path):
        # 终止条件：找到目标和
        if current_sum == target:
            res.append(path[:]) # 注意要浅拷贝
            return
        
        # 遍历可选数字
        for i in range(start_index, len(nums)):
            # 剪枝1：如果当前和加上数字已经大于target，后面的数字更大，直接跳出循环
            if current_sum + nums[i] > target:
                break
                
            # 剪枝2：树层去重。如果同一层级遇到相同的数字，跳过，防止产生重复组合
            if i > start_index and nums[i] == nums[i-1]:
                continue
                
            # 做出选择
            path.append(nums[i])
            # 递归向下。因为每个数字只能用一次，所以传 i + 1
            dfs(i + 1, current_sum + nums[i], path)
            # 撤销选择（回溯）
            path.pop()

    dfs(0, 0, [])
    return res

# 测试
nums = [10, 1, 2, 7, 6, 1, 5]
target = 8
print(combinationSum(nums, target))
# 输出: [[1, 1, 6], [1, 2, 5], [1, 7], [2, 6]]
```

**4. 引申思考/面试追问**：
*   *追问*：如果题目改为“列表中的数字可以无限制重复被选取”（即LeetCode 39题），代码需要改哪里？
*   *思路*：只需改动两个地方：1. 去掉树层去重的逻辑（`if i > start_index and nums[i] == nums[i-1]`）；2. 递归调用时，传入的 `start_index` 依然是 `i` 而不是 `i + 1`，表示当前数字在下一层递归中还可以继续被选择。
