# 复试项目深挖：RAG 优化路线与多智能体新 Idea

这份材料直接围绕我简历中的 3 个项目展开：

- `RAG-Triple-Engine`：讲企业问答系统的当前方案、架构升级与 P0-P3 演进路线。
- `AutoPE`：讲多智能体闭环优化、策略迭代与落地扩展。
- `llm-multiagent-benchmark`：讲统一评测、实验设计、稳定性和泛化能力。

目标不是把项目拆成 3 个孤立点，而是把它们讲成一条完整主线：

`知识获取 -> 检索增强 -> 多智能体协作 -> 自动评测与持续优化`

---

## 1. 项目总主线

如果老师让我用一句话概括这几段经历，我会这样讲：

我做的事情本质上是在解决“大模型如何从外部知识中获得可靠信息、如何在复杂任务中形成协作、以及如何被系统地评估”这三个连续问题。`RAG-Triple-Engine` 解决的是“知识怎么拿得准”；`AutoPE` 解决的是“多个角色如何围绕目标形成闭环优化”；`llm-multiagent-benchmark` 解决的是“这种协作到底有没有提升，能不能稳定泛化”。所以这三个项目不是割裂的，前者提供知识支撑，中间者负责策略执行与反思迭代，后者负责量化评测与科学比较，最终可以组合成一个更完整的 `Agentic RAG` 系统。

面试里 30 秒口述版：

> 我的项目主线可以概括成三步。第一步，我先做了一个三重引擎的 RAG 系统，解决企业文档检索和引用回答问题；第二步，我在 AutoPE 里把任务从单轮问答推进到多角色协作和策略自优化；第三步，我又做了多智能体 benchmark，把这种协作式系统放到统一框架里评测它的稳定性和泛化能力。沿着这条主线继续升级，就会自然走到更完整的 Agentic RAG 和面向真实场景的多智能体系统。

## 2. RAG 当前方案复盘

### 2.1 当前基线

`RAG-Triple-Engine` 的当前基线可以概括为下面这条链路：

```text
OCR / 文本清洗
  -> 文档切分
  -> 语义检索 + 关键词检索 + 实体摘要检索
  -> 重排
  -> 引用式回答
```

这套方案的核心目标不是追求极其复杂的智能体能力，而是先把企业文档问答里最常见的 3 个问题处理好：

- 文档类型杂，既有纯文本，也有图片、扫描件、表格截图。
- 单一检索方式不稳定，纯 dense 容易漏掉精确术语，纯关键词又难以理解语义。
- 业务问答不能只给答案，还要给出处，降低幻觉风险。

### 2.2 当前方案的优点

我认为当前方案最有价值的地方有 4 个：

- 检索互补性强：语义检索擅长理解近义表达，关键词检索擅长命中专有名词和编号，实体摘要检索擅长快速补背景。
- 对非结构化资料更友好：支持图片和 OCR 场景，能把扫描件、图文混排资料也纳入知识库。
- 工程链路完整：从清洗、切分、向量化、索引、重排到回答生成是完整闭环，不只是一个 demo。
- 回答可核验：输出答案时带来源片段和文档编号，能把“模型觉得”变成“证据支持”。

### 2.3 当前方案的短板

如果老师继续追问“你觉得现有方案还不够的地方在哪”，我会主动指出下面这些短板：

- 缺少显式的查询理解层。当前更像是“拿问题直接检索”，没有先判断它是事实型、流程型、比较型还是多跳型问题。
- chunk 策略还偏静态。不同文档类型共用一套切分规则，容易出现上下文被截断或信息重复召回。
- 元数据利用不足。文档时间、部门、权限、版本、主题等信息没有很好参与过滤和排序。
- 重排层还不够强。若只靠粗粒度 rerank，很难处理多文档、多证据拼接场景。
- 没有把失败样本闭环起来。现在更多是一次性问答，而不是“答错后知道错在哪、下一次如何变好”。

### 2.4 当前方案适合怎样讲

复试里，我不会把这个项目讲成“我已经做完了最先进的 RAG”，而会讲成：

> 我先做了一个很扎实的企业问答基线系统，核心思想是用三路检索降低单路召回的脆弱性，并通过引用回答提升可信度。在这个基础上，我很清楚下一步应该如何从检索、排序、知识结构化和 Agent 协作四个层面继续演进。

## 3. RAG 目标架构与 P0-P3 路线

### 3.1 目标架构

我希望把当前的三重引擎，升级成下面这个更完整的 `Agentic RAG` 架构：

```text
用户问题
  -> Query Understanding / Router
  -> Multi-Route Recall
       -> Dense Recall
       -> BM25 / Keyword Recall
       -> Entity / Graph Recall
       -> Metadata Filter
  -> Fusion + Rerank
  -> Evidence Aggregation
  -> Answer Generation with Citations
  -> Confidence Estimation / Fallback
  -> Feedback Logging / Memory Update
```

这个架构和当前版本最大的区别在于：

- 不再把检索看成一个黑盒，而是拆成“理解问题、选择召回路径、融合证据、给出置信度、沉淀反馈”五个显式阶段。
- 后续如果要引入多智能体，也可以很自然地把 planner、retriever、verifier 挂进去，而不是推倒重来。

### 3.2 P0：低风险提效

#### 核心改动

- 重新设计 chunk 策略：按标题、段落、表格、问答结构分块，而不是只按固定 token 长度切。
- 引入元数据过滤：把文档时间、来源、主题、版本号纳入召回前过滤。
- 升级为 `BM25 + dense` 融合召回，避免语义检索漏掉关键词。
- 增加 cross-encoder rerank，对 top-k 结果做精排。
- 增加回答引用、检索缓存和热门问题缓存。

#### 收益指标

- 离线指标：`Recall@10`、`MRR`、`nDCG@10`、引用命中率。
- 系统指标：`P95 latency`、缓存命中率、平均 token 成本、空召回率。

#### 工程代价

- 工程成本低到中等，主要是数据处理和排序链路升级。
- 不需要改变主系统范式，适合先在当前项目里快速落地。
- 对线上稳定性最友好，收益通常也最容易证明。

#### 复试表达话术

> 如果让我先做一轮最稳妥的升级，我会先做 P0。因为这一级不追求炫技，而是先把检索系统最基础但最关键的问题补齐：分块更合理、召回更互补、排序更精准、回答更可核验。它的特点是风险低、收益直观、最适合先把正确率和时延做稳。

### 3.3 P1：质量优化

#### 核心改动

- 增加 query rewrite，把用户问题改写成更适合检索的形式。
- 支持多轮上下文改写，解决“这个方案”和“上一个版本”之类的指代问题。
- 引入时间衰减和文档新鲜度机制，让高时效文档优先进入排序。
- 在融合层做去冗余，减少相似 chunk 重复挤占上下文。
- 根据问题类型动态调整多路召回配比，比如事实型更偏关键词，解释型更偏语义。

#### 收益指标

- 离线指标：多轮问题准确率、`Answer Groundedness`、去冗余后有效证据覆盖率。
- 系统指标：新文档生效时延、问题改写触发率、上下文压缩率、失败回退率。

#### 工程代价

- 工程成本中等，需要增加 query understanding 层和反馈埋点。
- 对数据和日志质量要求更高，需要有问题类型和失败样本的统计。
- 收益比 P0 更依赖场景设计，但一旦做出来，体验提升会很明显。

#### 复试表达话术

> P1 的核心不是再堆一个新模块，而是让系统开始“理解问题”而不是“看到问题就检索”。这一步做完之后，RAG 才真正从静态检索库变成面向用户意图的检索系统，尤其对多轮对话、版本追踪和强时效场景帮助很大。

### 3.4 P2：复杂知识增强

#### 核心改动

- 引入 GraphRAG 或实体关系图，把文档中的实体、关系、事件抽取成图结构。
- 支持多跳检索，先命中核心实体，再沿关系边扩展证据。
- 引入领域术语词表和混合索引，解决行业黑话、缩写、同义词问题。
- 增加用户画像个性化检索，让不同角色优先看到不同证据。
- 对长文档做层次化摘要和父子 chunk 检索，减少长上下文噪声。

#### 收益指标

- 离线指标：多跳问答准确率、实体覆盖率、长文档问答正确率、个性化命中率。
- 系统指标：索引构建耗时、图谱更新时延、存储开销、复杂查询平均时延。

#### 工程代价

- 工程成本高，需要维护新的数据结构和更新机制。
- 图谱构建、实体消歧、关系更新都会引入额外复杂度。
- 适合在明确存在复杂知识关联场景时再上，不适合一开始就全量铺开。

#### 复试表达话术

> P2 的本质是把“文档块”提升成“结构化知识对象”。当问题开始涉及实体关系、跨文档链路、流程依赖和多跳推理时，单纯的 chunk 召回会明显吃力，这时候 GraphRAG 和层次化知识组织就会体现价值。

### 3.5 P3：Agentic RAG

#### 核心改动

- 引入 `planner agent`：先判断问题复杂度，并拆成若干检索子任务。
- 引入 `retriever agent`：根据子任务选择不同召回通道和工具。
- 引入 `verifier agent`：对答案和证据一致性做校验，必要时拒答或回退。
- 支持失败回退策略：置信度低时返回“需要更多信息”或只给证据不下结论。
- 做成本和时延自适应：简单问题走轻链路，复杂问题才触发多 agent 协作。

#### 收益指标

- 离线指标：复杂任务成功率、校验后幻觉率下降幅度、多步问题最终正确率。
- 系统指标：平均链路长度、verifier 触发率、复杂问题 `P95 latency`、单问成本。

#### 工程代价

- 工程成本最高，不只是多加几个 prompt，而是要重新设计调度、观测和失败处理。
- 如果没有前面 P0-P2 的检索和证据基础，P3 很容易变成“多智能体一起胡说”。
- 适合在高价值复杂任务上做分层启用，而不是默认全量开启。

#### 复试表达话术

> P3 不是为了把系统包装成多智能体而上多智能体，而是在检索已经比较稳的前提下，把复杂问题拆解、证据核验和失败回退显式化。也就是说，我理解的 Agentic RAG 前提不是 agent 数量，而是证据链和可控性。

### 3.6 一句话总结 P0-P3

- `P0`：先把基础检索做扎实。
- `P1`：让系统开始理解用户意图和上下文。
- `P2`：让系统能处理复杂知识结构和多跳问题。
- `P3`：让系统具备任务分解、证据核验和自适应决策能力。

如果老师问“为什么不直接上 P3”，我会回答：

> 因为多智能体不是替代检索基础能力的捷径。没有稳定召回、排序、证据聚合和反馈机制，P3 往往只会把系统复杂度放大，而不是把效果放大。

## 4. 多智能体现有项目复盘

### 4.1 AutoPE 已有能力

`AutoPE` 的核心是围绕销售对话构建一个自动迭代闭环：

```text
销售 agent
  -> 顾客 agent
  -> 质检 / 评分 agent
  -> 策略总结 / 更新 agent
  -> 下一轮对话
```

我认为这个项目已经体现出 4 个很强的点：

- 它不是简单地让两个模型对话，而是做成了“对话 -> 评估 -> 总结 -> 策略更新”的闭环系统。
- 它把销售流程拆成 4 个阶段，便于定位问题出在哪一段，而不是只能看最终是否成交。
- 它引入买家建模和双层画像，把用户差异显式化，而不是假设所有客户都一样。
- 它有量化指标和自动报告，说明项目并不是停留在案例展示层面。

### 4.2 AutoPE 的不足

如果我自己给这个项目找下一阶段升级点，我会重点提下面这些：

- 现在更多依赖 prompt 内部策略，没有外部知识库支撑，容易出现产品知识不准或说服依据单薄。
- 策略更新更多是经验归纳，还没有形成显式可复用的策略记忆库。
- 缺少合规和风险审查角色，真实销售场景中这是非常关键的一环。
- 评测场景虽然能看到收益，但还不够标准化，迁移到新产品、新客户群体时泛化能力怎么证明，还可以继续加强。

### 4.3 llm-multiagent-benchmark 的补强价值

`llm-multiagent-benchmark` 虽然不在销售场景，但它对 AutoPE 后续升级有很强的补充意义：

- 它提供了统一的状态表征和跨环境实验框架，说明我不是只会做一个业务 demo，也在思考如何统一抽象。
- 它支持不同环境、不同模型、不同 prompt 的对照实验，能为 AutoPE 的迭代提供更标准的实验方法。
- 它强调稳定性、泛化能力和排行榜式比较，这刚好是多智能体系统走向科研化和平台化的关键一步。

### 4.4 适合怎样讲

复试里，这两个项目我会连着讲：

> AutoPE 让我做到了“多智能体闭环优化”，benchmark 让我补上了“统一评测与泛化分析”。前者更贴近真实场景，后者更偏方法论和实验平台，两者结合之后，就可以从一个会跑的多角色系统，走到一个可评估、可比较、可迁移的平台型项目。

## 5. 2 个强 Idea

## 5.1 Idea A：AutoPE 2.0：检索增强的销售策略自进化平台

### 问题定义

当前销售对话系统最大的瓶颈，不是“能不能生成一句听起来像销售的话术”，而是：

- 话术是否有证据支撑。
- 是否能根据客户类型调用不同知识。
- 是否能在多轮对话后沉淀出真正可迁移的策略。
- 是否能控制合规风险和错误承诺。

所以我希望把 AutoPE 从“纯 prompt 驱动的对话迭代系统”，升级成“带外部知识、长期记忆和风险审查的多智能体自进化平台”。

### 为什么现有项目还不够

现有 AutoPE 已经能做策略迭代，但还存在 4 个限制：

- 没有检索层，策略更多来自模型内部先验，不够 grounded。
- 没有策略记忆库，好的话术和失败案例难以系统复用。
- 缺少合规 reviewer，难以满足真实销售和客服场景需求。
- 复盘主要是“总结得失”，还没有做反事实推演：如果换一种话术会怎样。

### 新增模块与完整架构

```text
用户画像库 / 产品知识库 / 竞品库 / 合规规则库
  -> Planner Agent
  -> Retriever Agent
  -> Seller Agent
  -> Buyer Simulator Agent
  -> Compliance Reviewer Agent
  -> Reflection Agent
  -> Strategy Memory Builder
  -> 下一轮策略版本
```

每个模块的职责是：

- `Planner Agent`：识别当前处于销售漏斗哪个阶段，决定要检索什么知识、调用什么策略。
- `Retriever Agent`：从产品知识、竞品对比、历史成功话术、行业规则中召回证据。
- `Seller Agent`：基于当前策略和证据执行对话。
- `Buyer Simulator Agent`：继续保留双层画像和感性/理性决策链，模拟用户真实反应。
- `Compliance Reviewer Agent`：检查是否出现夸大承诺、违规表述和风险话术。
- `Reflection Agent`：分析失败点，给出阶段性策略更新建议。
- `Strategy Memory Builder`：把高质量策略、触发条件、失败案例沉淀成可检索记忆单元。

### 创新点

我认为这个方向的创新点有 4 个：

- 把 `RAG` 引入销售策略优化，不再只生成话术，而是让话术有外部知识和证据基础。
- 把“对话后总结”升级为“策略记忆库构建”，让系统具备长期可复用的经验沉淀。
- 加入合规 reviewer，让系统从“会卖”进一步变成“卖得稳、卖得安全”。
- 加入反事实复盘，即同一场景下比较不同策略版本，研究策略更新真正带来的因果增益。

### 实验设计与评价指标

我会把实验分成 4 组：

- 单 agent 基线。
- 现有 AutoPE。
- `AutoPE + 检索增强`。
- `AutoPE 2.0 全模块`。

核心指标分两类：

- 任务指标：成交率、平均对话轮次、阶段推进率、用户满意度、复购意愿。
- 安全指标：违规率、错误承诺率、知识引用率、事实性错误率。

还会做 3 类消融实验：

- 去掉检索层，看 grounded knowledge 对转化率和事实性有多大贡献。
- 去掉合规 reviewer，看安全性是否明显下降。
- 去掉策略记忆库，看跨 persona 迁移时性能是否更不稳定。

### 科研价值

这个方向的科研价值在于，它把多智能体系统从“协作完成任务”推进到“基于外部知识的策略自进化”。研究问题可以自然落在下面几类：

- 检索增强是否能显著改善多智能体策略优化的 sample efficiency。
- 合规 reviewer 是否会改变策略更新方向，形成安全与转化之间的 trade-off。
- 反事实复盘是否能比普通反思更稳定地提升跨 persona 泛化能力。

### 工程落地价值

它的工程价值也很明确：

- 可用于销售培训和新手话术推荐。
- 可用于客服、咨询、保险、电商导购等高对话密度场景。
- 可在正式上线前做策略 A/B 测试和风险审查。

### 90 秒复试讲法

> 我会把 AutoPE 的下一步定义为一个检索增强的销售策略自进化平台。现在的 AutoPE 已经具备多角色闭环优化能力，但它更多依赖模型内部知识。下一步我想把产品知识库、竞品信息、历史成功案例和合规规则都接入进来，让 seller agent 不是单纯生成话术，而是基于证据说话。同时，我会把好的策略和失败案例沉淀成策略记忆库，再加一个合规 reviewer 和反事实复盘模块，这样系统既能持续优化转化率，也能控制风险。这个方向的价值在于把 RAG、多智能体和自动优化真正融合起来，既有科研问题，也有很强的工程落地场景。

## 5.2 Idea B：SalesBench-RAG：面向销售场景的多智能体评测与迁移基准

### 问题定义

目前很多多智能体 benchmark 都偏通用决策环境，比如博弈、导航、协作控制等。这些 benchmark 很适合研究决策和协作，但离真实业务任务还有一层距离。对于销售、客服、咨询这类任务，更关键的问题其实是：

- 角色是否理解业务知识。
- 多 agent 协作是否真的优于单 agent。
- 系统是否能在不同产品、不同用户、不同话术风格之间迁移。
- 安全、合规、成本、时延这些工业指标如何一起评估。

所以我想把 `llm-multiagent-benchmark` 的方法论迁移到销售场景，做一个面向真实知识密集任务的 benchmark。

### 为什么现有项目还不够

虽然我已经做了多智能体 benchmark，但当前版本主要是 MPE 和足球环境：

- 环境偏通用决策，不强调外部知识 grounding。
- 任务指标主要是奖励、胜率、协作成功率，与业务目标之间还隔着一层。
- 缺乏针对角色画像、知识版本变化和场景迁移的系统评测。

### 新增模块与完整架构

```text
场景库生成器
  -> Persona 库
  -> 产品与知识包
  -> 任务难度梯度
  -> Agent Role Library
  -> Interaction Simulator
  -> Evaluator Suite
  -> Transfer / Robustness Arena
  -> Leaderboard
```

这里的关键模块包括：

- `场景库生成器`：构造不同业务场景，比如保险、教育咨询、SaaS 销售、客服回访。
- `Persona 库`：按照价格敏感、风险规避、理性分析、情绪化等维度生成客户画像。
- `产品与知识包`：控制知识是否充分、是否过时、是否冲突，用于测试 RAG 能力。
- `Agent Role Library`：支持单 agent、seller-buyer、seller-reviewer、planner-retriever-seller 等多种配置。
- `Evaluator Suite`：同时衡量转化、事实性、安全性、协作效率、成本和时延。
- `Transfer Arena`：测试从旧产品迁移到新产品、从单一 persona 迁移到复合 persona 的鲁棒性。

### 创新点

这个 benchmark 的创新点可以总结为 4 个：

- 它把 `多智能体评测` 和 `知识增强` 结合起来，不再只测动作决策，也测知识调用能力。
- 它把通用 benchmark 中的协作/对抗思想，迁移到了更贴近产业任务的销售与客服场景。
- 它同时评测效果、安全、成本和时延，避免只看单一成功率。
- 它支持迁移实验和难度梯度，可以系统研究“策略是在记住场景，还是学会了方法”。

### 实验设计与评价指标

我会重点做 4 条对照线：

- `Closed-book single agent` 对比 `RAG single agent`。
- `RAG single agent` 对比 `multi-agent without verifier`。
- `multi-agent without verifier` 对比 `multi-agent with verifier / reviewer`。
- `in-domain` 对比 `cross-domain transfer`。

核心评价指标包括：

- 任务效果：成交率、目标完成率、阶段推进率、客户满意度。
- 事实性：引用率、知识一致性、过时知识误用率。
- 协作性：角色分工清晰度、重复劳动率、无效交互率。
- 系统性：平均时延、单轮成本、失败回退率。
- 稳定性：跨 persona 方差、跨产品方差、随机种子敏感度。

### 科研价值

这个方向比较适合回答“你的项目除了能用，还有什么研究意义”：

- 可以研究知识 grounding 如何影响多智能体协作效率。
- 可以研究 verifier/reviewer 是否能显著降低协作幻觉。
- 可以研究不同角色拆分方式对泛化性能的影响。
- 可以研究 benchmark 如何从游戏环境扩展到真实任务环境。

### 工程落地价值

它的工程用途也很明确：

- 上线前可用于模型选择和策略选择。
- 可以变成一个内部评测平台，而不只是学术 benchmark。
- 能帮助企业在新产品、新话术、新模型切换时做风险预评估。

### 90 秒复试讲法

> 我已经做过一个偏通用环境的多智能体 benchmark，但我觉得下一步更有价值的是把这套评测方法迁移到真实业务场景。我想做一个 SalesBench-RAG，把销售、客服、咨询这类知识密集任务抽象成标准场景库，同时引入 persona、知识包和任务难度梯度，去评测单 agent、RAG、以及多智能体协作系统在效果、安全、时延和泛化上的差异。这样一来，AutoPE 就不只是一个能跑的系统，而是有一套统一 benchmark 可以证明它什么时候有效、为什么有效、在什么条件下会失效。

## 6. 高频追问问答

### Q1：为什么你的 RAG 要做三路检索，而不是只做 dense retrieval？

回答框架：

因为企业文档里既有语义表达，也有大量精确术语、编号、产品名和组织名。dense retrieval 擅长找语义相近内容，但对精确匹配并不稳定；关键词检索刚好补这部分；实体摘要则是在文档很多、背景复杂时快速补充上下文。三路检索不是简单堆模块，而是降低单一路径失效的风险。

### Q2：你会怎么设计 chunk，为什么 chunk 对 RAG 这么关键？

回答框架：

chunk 的本质是在“语义完整性”和“召回粒度”之间做平衡。切太大，会带来噪声和上下文浪费；切太小，会破坏证据链。我的做法是优先按结构切，比如标题、段落、表格、问答单元，再根据 token 长度做二次切分。如果进入 P1/P2，还会引入父子 chunk 和层次化摘要。

### Q3：如果召回结果很多，但大部分是冗余的，你怎么解决？

回答框架：

我会从两层处理。第一层是召回后去冗余，比如按相似度聚类或按文档 ID 合并，避免同一段话换个表述重复进入上下文。第二层是 rerank，把“最有用的少量证据”放到最前面。对复杂问题，还可以在 evidence aggregation 阶段先做证据压缩再交给生成模型。

### Q4：为什么你认为 Agentic RAG 一定要等 P0-P2 比较稳后再做？

回答框架：

因为 agent 不能替代基础检索能力。如果召回本身不准、证据不可核验、置信度也没有，那多智能体只会把错误扩散得更复杂。我的理解是，Agentic RAG 的价值在于任务分解和证据核验，而不是用更多 prompt 掩盖底层检索问题。

### Q5：AutoPE 为什么要做成多智能体，而不是一个强一点的单 agent？

回答框架：

因为销售对话里天然存在不同功能角色：执行、评估、反思、策略更新。把这些功能全塞进一个 agent，容易导致目标耦合过重，既要说话又要自评还要总结，行为不稳定。拆成多角色后，可以把“执行”和“评估”解耦，系统也更容易观察和优化。

### Q6：你怎么证明 AutoPE 的提升不是 prompt 偶然碰巧？

回答框架：

我会强调三点。第一，固定 persona 集合和任务配置，多次重复实验，报告均值和方差。第二，做版本对照和消融，比如去掉 reviewer、去掉反思模块、去掉知识检索层，看收益是否仍然存在。第三，增加跨 persona、跨产品、跨场景迁移测试，验证提升不是只对某个模板有效。

### Q7：多智能体系统最容易出什么问题？你会如何控制？

回答框架：

最常见的问题有 3 类：一是角色间互相强化错误，形成协同幻觉；二是链路过长导致时延和成本过高；三是角色边界不清，造成重复劳动。我会通过 verifier/reviewer、层级调度、轻重链路分流和清晰的角色职责来控制这些问题。

### Q8：为什么你的第二个 Idea 要做 benchmark，而不是继续只做业务系统？

回答框架：

因为没有 benchmark，系统优化很容易陷入“这次看起来不错，但不知道为什么”。业务系统能证明可用性，但 benchmark 才能证明方法本身有没有普适价值。尤其是多智能体系统，很多改动会影响稳定性、成本和泛化能力，没有统一评测就很难做科学比较。

### Q9：如果老师问你怎么定义 RAG 的“效果”，你会怎么答？

回答框架：

我不会只说一个正确率，而会拆成 3 层。第一层是检索层指标，比如 Recall@k、MRR、nDCG；第二层是回答层指标，比如 groundedness、引用准确率、事实性；第三层是系统层指标，比如时延、成本、空召回率和用户满意度。只有把这三层一起看，RAG 的效果才是完整的。

### Q10：如果把 AutoPE 2.0 真正落地，你最先做哪一步？

回答框架：

我会先做最小闭环版本：给现有 AutoPE 接一个产品知识库和历史成功话术库，再加一个轻量 reviewer。原因是这一步最容易验证“检索增强是否能提升对话质量和事实性”，同时不会一下子把系统复杂度拉得过高。先把 grounded knowledge 和 reviewer 跑通，再去做策略记忆库和反事实复盘，会更稳。

## 收尾总结

如果复试最后让我总结一句，我会这样讲：

> 我现在的项目已经分别覆盖了 RAG、多智能体闭环和统一评测三个层面。下一步我最想做的不是再单独堆一个新功能，而是把这三条线真正合起来，做一个有知识支撑、有角色协作、有量化评测、还能持续自我优化的 Agentic RAG 系统。我觉得这条路线既有研究问题，也有很强的工程落地价值。
