# 题目1：了解 KV cache 吗，GQA 的思想是什么

**题目描述**：解释大模型推理中的 KV cache 机制，并阐述 GQA (Grouped-Query Attention) 的核心思想。

**前置基础知识**：
*   **自回归生成 (Autoregressive Generation)**：LLM 生成文本是逐字（token）生成的，第 $t$ 步的生成依赖于前 $t-1$ 步的输出。
*   **标准多头注意力 (MHA, Multi-Head Attention)**：在标准 Transformer 中，每个注意力头（Head）都有自己独立的 Query (Q)、Key (K)、Value (V) 投影权重。

**核心解答**：
1.  **KV Cache**：
    *   **是什么**：在自回归推理时，每次生成新 token 都需要和之前所有的历史 token 计算 Attention。为了避免每生成一个新词就把历史词的 K 和 V 重新计算一遍，推理引擎会将历史 token 的 K 和 V 张量缓存到显存中。
    *   **为什么需要**：本质是“空间换时间”，极大地降低了计算量（FLOPs），将生成阶段的计算复杂度从 $O(N^2)$ 降到了 $O(N)$，是 LLM 推理加速的基石。但代价是随着序列变长，显存占用会线性激增。
2.  **GQA (Grouped-Query Attention) 的思想**：
    *   **演进背景**：由于 MHA 的 KV Cache 占用显存过大，业界曾提出 MQA（Multi-Query Attention），即所有 Q 头共享唯一的一个 K 头和 V 头，这把 KV Cache 压缩到了极致，但会导致模型表达能力下降。
    *   **GQA 核心思想**：GQA 是 MHA 和 MQA 的折中方案。它将所有的 Q 头划分为 $G$ 个组（Group），每个组内的 Q 头共享同一个 K 头和 V 头。
    *   **优势**：在保持与 MHA 几乎相当的模型性能和表达能力的同时，大幅度降低了 KV Cache 的显存占用（约为 MHA 的 $1/G$），从而支持更大的 Batch Size 和更长的上下文窗口。主流大模型（如 Llama 2/3, Qwen 等）均采用了 GQA 架构。

**引申思考/面试追问**：
1.  **追问**：当引入 RoPE（旋转位置编码）时，KV Cache 里的 K 是编码前还是编码后的？
    *   **思路**：是编码后的。因为 RoPE 属于绝对位置编码的形式实现相对位置编码，当前 token 的 Q 乘以历史 token 的 K 时，K 必须已经注入了它所在位置的绝对位置信息，所以缓存的是经过 RoPE 处理后的 K。
2.  **追问**：KV Cache 变大后容易产生显存碎片，工业界是怎么解决的？
    *   **思路**：引出 vLLM 的 PagedAttention 技术，借鉴操作系统的虚拟内存分页思想，将 KV Cache 划分为固定大小的 Block，非连续存储，极大减少了显存碎片。

# 题目2：讲一下 FlashAttention 的思想和做法

**题目描述**：阐述 FlashAttention 算法的核心思想及其实现机制。

**前置基础知识**：
*   **内存墙 (Memory Wall)**：现代 GPU 的计算能力（TFLOPs）远快于显存带宽（HBM Bandwidth）。Transformer 的 Attention 计算中，大量时间耗费在将数据从 HBM 读到 SRAM，再从 SRAM 写回 HBM，而非真正的矩阵计算上。
*   **标准 Attention 计算**：公式为 $Softmax(QK^T/\sqrt{d})V$。中间会产生一个 $N \times N$（$N$ 为序列长度）的 Attention Score 矩阵，这个庞大的中间矩阵频繁的读写是性能和显存的瓶颈。

**核心解答**：
*   **核心思想**：FlashAttention 是一种**IO 感知（IO-Aware）**且**精确（Exact）**的注意力算法。它的核心思想是最大程度减少 GPU 显存（HBM）与片上内存（SRAM）之间的数据搬运次数（Memory Access Cost, MAC），从而打破内存墙，实现训练/推理加速并大幅降低显存占用。
*   **具体做法**：
    1.  **分块计算 (Tiling)**：将输入的 Q、K、V 矩阵切分成小块（Blocks），使得这些小块可以完全装入速度极快的 SRAM 中。然后在 SRAM 中完成局部的矩阵乘法。
    2.  **在线 Softmax (Online Softmax)**：标准 Softmax 需要拿到整行元素才能算分母。FlashAttention 引入了数学等价的 Online Softmax 技巧，通过维护每个分块的局部最大值（Max）和指数和（Sum），在不需要全局数据的情况下，分块迭代地计算 Softmax，避免了中间 $N \times N$ 矩阵的写回。
    3.  **重计算 (Recomputation)**：在反向传播时，为了节省显存，FlashAttention 不保存前向传播时庞大的 $N \times N$ 注意力矩阵。相反，它只保存前向计算出来的 Softmax 归一化统计量（Max 和 Sum），在反向时利用这些统计量在 SRAM 中快速重新计算注意力分数，以此求导。重计算增加了一点计算量，但省下了海量 IO 时间，整体速度依然更快。

**引申思考/面试追问**：
1.  **追问**：FlashAttention v2 相比 v1 做了哪些核心优化？
    *   **思路**：v2 优化了并行化策略（在 sequence length 维度也进行了并行），减少了非矩阵乘法（如 rescale 操作）的计算，提升了硬件利用率（利用 Tensor Cores），并且原生支持了 GQA 和 MQA。
2.  **追问**：如果上下文长度极长（如 100K+），单卡的 SRAM 装不下分块怎么办？
    *   **思路**：引出 RingAttention 或 FlashAttention-3/分布式 Attention。通过将长序列切分到多张 GPU 卡上，卡间通过 Ring 通信传递 KV 块，实现无限长上下文的 Attention 计算。

# 题目3：了解过哪些模型训练推理优化的方法（DeepSpeed，数据并行，张量并行，流水线并行）

**题目描述**：简述大模型分布式训练和优化的常用并行策略及 DeepSpeed 原理。

**前置基础知识**：
*   **显存占用来源**：大模型训练时，显存不仅被模型参数（Parameters）占据，更大头在于优化器状态（Optimizer States，如 Adam 的动量和方差）、梯度（Gradients）以及前向激活值（Activations）。

**核心解答**：
为了解决单卡显存不足和训练速度慢的问题，业界采用了 3D 并行和显存优化技术：
1.  **数据并行 (DP / DDP)**：
    *   **原理**：每张 GPU 卡上都有一份完整的模型副本。将一个大的 Batch 切分成多个小 Micro-batch 分发给不同卡。各卡独立前向和反向传播，最后通过 All-Reduce 通信同步梯度，更新参数。
    *   **局限**：无法解决单卡装不下整个模型（及优化器状态）的问题。
2.  **张量并行 (TP, Tensor Parallelism)**：
    *   **原理**：以 Megatron-LM 为代表。将模型层内的矩阵乘法（如 Attention 的 QKV 权重矩阵、MLP 的线性层）按行或列切分到不同的 GPU 上。各卡计算局部结果后，通过 All-Reduce 或 All-Gather 汇总。
    *   **特点**：通信极其频繁，通常只在同一个物理节点内（利用高带宽的 NVLink）使用。
3.  **流水线并行 (PP, Pipeline Parallelism)**：
    *   **原理**：将模型的不同层（Layers）按顺序切分到不同的 GPU 上。例如 GPU 0 计算 1-10 层，输出传给 GPU 1 计算 11-20 层。
    *   **优化**：为了解决 GPU 闲置等待（Bubble 气泡）问题，通常将数据切分为多个 Micro-batch，采用 1F1B（One Forward One Backward）等调度策略让各卡流水线运转起来。
4.  **DeepSpeed (ZeRO 优化器)**：
    *   **原理**：针对数据并行中“每张卡都存一份完整状态”造成的严重显存冗余，ZeRO（Zero Redundancy Optimizer）通过切分状态来省显存。
    *   **ZeRO-1**：只切分优化器状态（Optimizer States）。
    *   **ZeRO-2**：切分优化器状态 + 梯度（Gradients）。
    *   **ZeRO-3**：切分优化器状态 + 梯度 + 模型参数（Parameters）。每张卡只存 $1/N$ 的数据，在前向或反向计算需要某层参数时，通过 All-Gather 临时从其他卡拉取，算完立刻丢弃（释放显存）。

**引申思考/面试追问**：
1.  **追问**：在一个拥有多台机器、每台机器 8 张卡的集群里，你会如何分配 TP, PP, DP 的维度？
    *   **思路**：通常单机内 8 卡走 TP（因为 NVLink 带宽极高，能 hold 住 TP 频繁的通信）；跨机器走 PP（通信量较小，只传激活值）；如果还有多余的机器，再在外层套 DP/ZeRO（扩展 Batch Size）。
2.  **追问**：ZeRO-3 和 TP 都能切分参数，它们有什么本质区别？
    *   **思路**：TP 是数学计算逻辑上的切分，单卡只计算一部分矩阵乘法；ZeRO-3 是数据存储上的切分，计算时依然要还原出完整的权重进行完整的计算。

# 题目4：场景题：如果要提升大模型的多轮对话能力，你有哪些可以改进的点

**题目描述**：从算法和工程角度，给出提升 LLM 多轮对话能力的优化方案。

**前置基础知识**：
*   多轮对话的痛点通常在于：**遗忘前文**（超出上下文窗口）、**角色混淆**（分不清 User 和 Assistant）、**指令遵循衰减**（聊了几轮后忘记了 System Prompt 的初始设定）。

**核心解答**：
提升多轮对话能力是一个系统工程，可以从数据、训练和推理三个维度进行改进：

1.  **数据构造维度（核心）**：
    *   **高质量多轮数据**：构建包含复杂逻辑连贯、多轮指代消解（如“那个东西”、“他”）、话题转移的高质量多轮对话语料。
    *   **System Prompt 强化**：在构建数据时，设计多样化的 System Prompt，并在长对话的后续轮次中，构造模型依然严格遵循设定的回复，强化模型对系统指令的长期记忆。
    *   **困难负样本**：针对容易出错的场景（如复读机、幻觉、角色扮演崩塌），构造负样本对，用于后续的偏好对齐。
2.  **模型训练维度（SFT & RLHF）**：
    *   **Loss Masking**：在多轮 SFT 训练拼接上下文时，必须对所有的 User 输入和 System Prompt 进行 Mask 操作，只对 Assistant 生成的 token 计算 Loss。如果不 Mask，模型会学到预测用户的话，导致推理时自言自语。
    *   **上下文长度扩展**：如果多轮对话极长，需要使用 RoPE 缩放（如 YaRN, NTK-Aware）扩展窗口，并使用长文本数据进行微调（Long-context FT）。
    *   **多轮偏好对齐 (DPO/RLHF)**：在奖励模型（Reward Model）或 DPO 训练中，专门针对多轮一致性打分。对于那些能正确引用前文细节、没有前后矛盾的回复给予高奖励。
3.  **推理与工程维度**：
    *   **动态上下文截断/摘要**：当对话轮数超过模型 Max Window 时，采用滑动窗口策略丢弃最老的消息；或引入一个小模型，在后台自动把前 $N$ 轮对话总结成一小段 Summary，拼接到当前 Prompt 前面。
    *   **System Prompt 注入**：在推理时，可以在每隔几轮的 User 输入前，隐式地再次拼入一次 System Prompt，强制唤醒模型的设定记忆。

**引申思考/面试追问**：
1.  **追问**：在 SFT 阶段，多轮对话的数据在输入模型前具体是如何拼接的？
    *   **思路**：需要使用特定的 Chat Template。例如使用 `<|im_start|>user\n...<|im_end|>\n<|im_start|>assistant\n...<|im_end|>`。必须确保特殊 token（如 EOS、角色标识符）的正确插入，并在计算 Loss 时精准对齐 `<|im_start|>assistant` 到 `<|im_end|>` 之间的区间。
2.  **追问**：多轮对话中，KV Cache 会越来越大，怎么优化推理性能？
    *   **思路**：除了 PagedAttention，还可以考虑使用 RadixAttention（如 SGLang），在多轮对话中，前几轮的 KV Cache 可以被命中复用，只需计算最新一轮 User 输入的增量 Attention。

# 题目5：代码题：56. 合并区间

**题目描述**：给定一个区间的集合，合并所有重叠的区间。（LeetCode 56）

**前置基础知识**：
*   **排序算法**：对二维数组按特定列进行排序。
*   **贪心思想**：先按左端点排序，保证处理顺序的局部有序性，然后依次判断相邻区间是否可合并。

**核心解答**：
*   **解题思路**：
    1.  如果区间列表为空，直接返回空。
    2.  首先将所有的区间**按照左端点进行升序排序**。
    3.  初始化一个结果数组 `merged`，并将排序后的第一个区间放入其中。
    4.  从第二个区间开始遍历：
        *   令当前区间为 `curr`，`merged` 中最后一个区间为 `last`。
        *   **判断重叠**：如果 `curr` 的左端点 $\le$ `last` 的右端点，说明有重叠。此时合并它们，即更新 `last` 的右端点为 $max(last.right, curr.right)$。
        *   **无重叠**：如果 `curr` 的左端点 $>$ `last` 的右端点，说明没有重叠，直接将 `curr` 追加到 `merged` 数组中。
*   **复杂度分析**：
    *   时间复杂度：$O(N \log N)$，其中 $N$ 为区间的数量。排序占主导地位，后续只需一次线性遍历 $O(N)$。
    *   空间复杂度：$O(\log N)$，排序所需要的栈空间（如果不考虑存储结果的数组）。

*   **Python 代码实现**：
```python
def merge(intervals):
    if not intervals:
        return []
    
    # 按照区间的左端点进行升序排序
    intervals.sort(key=lambda x: x[0])
    
    merged = [intervals[0]]
    
    for i in range(1, len(intervals)):
        current_interval = intervals[i]
        last_merged_interval = merged[-1]
        
        # 如果当前区间的左端点 <= 结果集中最后一个区间的右端点，说明有重叠
        if current_interval[0] <= last_merged_interval[1]:
            # 合并区间，更新右端点为两者的最大值
            last_merged_interval[1] = max(last_merged_interval[1], current_interval[1])
        else:
            # 没有重叠，直接加入结果集
            merged.append(current_interval)
            
    return merged
```

**引申思考/面试追问**：
1.  **追问**：如果区间的数量非常庞大，比如几十亿个，无法全部加载进内存，该怎么处理？
    *   **思路**：使用外部排序（External Sorting）。将大文件切分成多个小文件，在内存中分别排序后写回磁盘，然后使用多路归并（结合最小堆）逐个读取合并；或者如果区间端点范围有限，可以考虑使用线段树或差分数组的思想。
2.  **追问**：如果是求两个区间列表的交集呢？（LeetCode 986）
    *   **思路**：使用双指针法。指针 `i` 和 `j` 分别遍历两个列表，交集的左端点是 `max(A[i].left, B[j].left)`，右端点是 `min(A[i].right, B[j].right)`。如果左端点 $\le$ 右端点，则存在交集。随后移动右端点较小的那个指针。
