# 题目1：大模型的灾难性遗忘问题怎么解决

**1. 题目描述**
探讨在对大模型进行微调（SFT）或持续预训练时，如何缓解模型因为学习新知识而导致原本具备的旧知识或通用能力大幅下降的“灾难性遗忘”现象。

**2. 前置基础知识**
*   **灾难性遗忘（Catastrophic Forgetting）**：神经网络在学习新任务的数据分布时，权重参数发生剧烈更新，覆盖了之前学习到的特征表征，导致在旧任务上性能骤降。
*   **稳定性与可塑性困境（Stability-Plasticity Dilemma）**：模型既需要保持稳定以记住旧知识，又需要具备可塑性以学习新知识。

**3. 核心解答**
解决大模型灾难性遗忘通常从数据、参数更新机制和损失函数三个维度入手：

*   **数据层面：数据混合/重放（Data Replay / Mixing）**
    这是工业界最直接、最有效的方法。在微调新领域数据时，按一定比例（通常为5%~20%）混入通用领域的预训练数据或旧任务的SFT数据。这样在反向传播时，梯度不仅指向新任务的最优解，也能兼顾旧任务的优化方向。
*   **架构层面：参数高效微调（PEFT，如LoRA）**
    冻结预训练模型的主干参数，只在旁路引入极少量的可训练参数（如LoRA的低秩矩阵）。因为包含了模型绝大部分通用知识的主干网络权重未发生改变，从而在物理层面上极大地保护了旧知识。在多任务场景下，还可以使用多LoRA路由机制（如LoRA Hub、MoE-LoRA）避免不同任务间的参数干扰。
*   **算法层面：正则化方法（Regularization）**
    *   **EWC (Elastic Weight Consolidation)**：通过计算旧任务的Fisher信息矩阵来评估每个参数对旧任务的重要性。在更新时，对那些对旧任务极其重要的参数施加较大的L2惩罚，限制其改变幅度；对不重要的参数则允许大幅调整。
    *   **Weight Decay**：在损失函数中加入对初始预训练权重的偏离惩罚，防止参数跑得太远。
*   **算法层面：知识蒸馏（Knowledge Distillation）**
    将原始未微调的大模型作为Teacher模型，正在微调的模型作为Student模型。在损失函数中增加一项KL散度，强制Student模型在输出分布（Logits）上尽量对齐Teacher模型，从而在学习新特征的同时“模仿”并保留旧特征。
*   **后处理层面：模型合并（Model Merging / Task Vector）**
    将微调后的模型权重与原始基础模型的权重进行线性插值（例如：$\theta_{final} = \alpha \theta_{finetune} + (1-\alpha) \theta_{base}$）。这种做法可以在保留新领域能力的同时，找回很大一部分通用能力。

**4. 引申思考/面试追问**
*   **追问：如果由于隐私合规原因，完全无法获取旧数据，只能用新数据微调，怎么解决遗忘？**
    *   *思路*：此时无法使用Data Replay。可以采用纯正则化方法（如EWC）；或者采用生成式重放（Generative Replay），即用旧模型（Teacher）生成一批伪数据作为通用数据混入训练；或者严格使用PEFT方法并保持较小的学习率。
*   **追问：LoRA一定能完全避免灾难性遗忘吗？**
    *   *思路*：不一定。虽然主干被冻结，但如果LoRA模块的秩（Rank）设置过大、学习率过高，或者训练轮数过多，LoRA模块输出的激活值会变得非常大，从而显著改变整个Transformer层的输出分布，依然会导致在旧任务上的表现变差。

# 题目2：了解 RAG 吗，GraphRAG 的做法

**1. 题目描述**
简述RAG（检索增强生成）的基本概念，并详细阐述基于知识图谱的检索增强生成（GraphRAG）的实现原理和流程。

**2. 前置基础知识**
*   **RAG**：为缓解大模型幻觉和知识滞后，在生成回答前，先从外部知识库检索出与Query相关的文本片段，作为上下文Prompt输入给大模型。
*   **知识图谱（KG）**：以实体（节点）和关系（边）构成的图结构数据。
*   **传统向量RAG的痛点**：基于Dense Embedding的相似度检索容易丢失全局上下文，擅长回答细节性问题（“点”），但极难回答需要跨文档、多跳推理的宏观总结性问题（“面”）。

**3. 核心解答**
**RAG的基本流程**分为三步：Indexing（文档切块与向量化）、Retrieval（Query向量化并计算相似度召回Top-K文档）、Generation（拼接文档与Query交由LLM生成）。

**GraphRAG的做法（以微软开源的GraphRAG架构为例）**：
GraphRAG 的核心思想是将非结构化文本转化为层次化的知识图谱，并利用图谱的社区结构来回答全局性和复杂性问题。

*   **阶段一：图谱构建与索引 (Indexing)**
    1.  **实体与关系抽取**：将私有文档切块，利用LLM（通过Prompt工程）从每个Chunk中提取实体（Entity）以及实体间的关系（Relation），构建出底层的知识图谱。
    2.  **社区发现（Community Detection）**：将构建好的图谱进行聚类（通常使用Leiden等图聚类算法）。将联系紧密的节点划分到同一个“社区”中，并可以形成从细粒度到粗粒度的多层次（Hierarchical）社区结构。
    3.  **社区摘要生成（Community Summaries）**：利用LLM，为每一个社区生成结构化的摘要报告，总结该社区内的核心人物、事件、主题和关系。
*   **阶段二：检索与生成 (Querying)**
    1.  **全局查询（Global Search - 针对宏观问题）**：当用户提问类似“总结这批文档的核心脉络”时，系统采用 Map-Reduce 架构。**Map阶段**：将用户的Query和同一层级的所有“社区摘要”并行发给LLM，生成多个中间答案并打分；**Reduce阶段**：将得分高的中间答案汇总，再次交给LLM生成最终的全局回答。
    2.  **局部查询（Local Search - 针对实体/多跳问题）**：提取用户Query中的实体，在图谱中定位该节点，并向外扩展提取其K跳邻居节点、相关连的原始文本Chunk以及所属的社区摘要，拼接成丰富的结构化上下文，交由LLM生成回答。

**4. 引申思考/面试追问**
*   **追问：相比于传统向量RAG，GraphRAG的构建成本极高，主要贵在哪里？如何优化？**
    *   *思路*：主要贵在“索引阶段”需要极其庞大的LLM API调用开销（每个Chunk都要过LLM抽实体，每个社区都要过LLM写摘要）。优化思路包括：使用更小、更便宜的开源模型（如GLM-4-Flash、Llama-3-8B）专门微调一个信息抽取模型替代GPT-4；或者结合传统NLP的NER（命名实体识别）技术进行混合抽取。
*   **追问：如何将现有的向量检索与GraphRAG结合？**
    *   *思路*：混合检索（Hybrid RAG）。Query进来后，一方面去向量库召回相关Chunk，另一方面去图谱中召回子图和社区摘要。在Prompt拼接时，将结构化知识（图谱路径）和非结构化知识（文本片段）共同作为上下文。

# 题目3：场景题：目前有一些专业领域知识（如医疗知识或者法律条文），准备做个智能助手，如何搭建一个 RAG 链路

**1. 题目描述**
设计一个针对专业垂直领域（医疗或法律）的RAG系统架构，涵盖从数据接入、处理、检索到最终生成的全流程。

**2. 前置基础知识**
*   **垂直领域特性**：医疗/法律文本具有极强的专业性（充满术语、缩写）、严谨的层级结构（如法律的编章节目条），且容错率极低（幻觉可能导致严重后果）。
*   **高级RAG架构**：包含Query改写、精细化Chunking、混合检索、重排（Rerank）等组件。

**3. 核心解答**
搭建专业领域的RAG链路，需重点在“数据清洗”、“检索精度”和“风控”上下功夫，具体链路如下：

*   **第一步：离线数据处理与索引构建 (Offline)**
    *   **多模态解析**：法律卷宗或医疗报告常包含PDF、表格和印章。需使用专业的文档解析工具（如Marker、Nougat）提取纯文本并还原表格Markdown结构。
    *   **语义与规则分块（Chunking）**：**绝对不能简单按字数截断**。法律场景必须采用“基于规则的分块（Rule-based Chunking）”，以“条文”为最小粒度，并保留其“第X法-第X章-第X条”的层级前缀；医疗场景可按“疾病定义-症状-治疗方案”切分。
    *   **元数据提取（Metadata）**：为每个Chunk注入丰富的Metadata，如法律效力级别、颁布时间、医疗科室、适用人群等。
    *   **领域模型微调**：通用Embedding模型对“心肌梗死”和“心绞痛”的区分度可能不够。需利用领域内的QA对，通过对比学习（Contrastive Learning）微调Embedding模型（如BGE）和Reranker模型。
*   **第二步：在线检索链路 (Online)**
    *   **Query理解与改写**：用户的提问往往口语化（如“肚子右下边疼咋办”），需先过一个轻量级LLM进行Query改写（改写为“右下腹疼痛 阑尾炎 诊断”），并提取出实体用于后续过滤。
    *   **混合检索（Hybrid Search）**：采用 **向量检索（Dense，捕捉语义相似度） + 关键词检索（BM25/ES，捕捉特定法条编号、罕见药名等硬匹配）**。结合Metadata过滤（如只查“2023年以后”的“刑法”），大幅缩小搜索空间。
    *   **重排（Reranking）**：将召回的Top 50文档片段，输入给Cross-Encoder架构的Reranker模型进行细粒度的语义交叉打分，截取最相关的Top 5。
*   **第三步：生成与风控 (Generation & Guardrails)**
    *   **Prompt工程**：设定严格的系统提示词，例如：“你是一个严谨的执业律师。请**仅**根据以下参考资料回答问题。如果资料中没有答案，请明确回答‘资料未提及’，绝不能编造。”要求输出附带引用标记（Citation）。
    *   **后处理风控**：加入护栏（Guardrails），对生成的回答进行敏感词过滤，并在医疗/法律建议末尾强制添加免责声明（“本回答仅供参考，不作为最终诊断/判决依据”）。

**4. 引申思考/面试追问**
*   **追问：法律条文中经常出现“处罚标准见本法第42条”，但第42条可能在另一个Chunk里，这种跨Chunk引用怎么解决？**
    *   *思路*：1. 在Chunking阶段进行“文档内链接解析”，把引用的条文内容直接拉取过来拼接到当前Chunk末尾；2. 使用 Parent-Child Chunking（父子分块），检索时匹配细粒度的子块，但喂给大模型的是包含上下文的较大父块；3. 引入GraphRAG构建法条间的引用图谱。

# 题目4：开放题：用过豆包吗，觉得目前豆包还存在什么问题

**1. 题目描述**
评估应聘者对当前主流大模型产品（字节跳动“豆包”）的使用深度、产品洞察力以及对其背后大模型技术瓶颈的理解。

**2. 前置基础知识**
*   **豆包（Doubao）**：字节跳动推出的C端大模型AI助手，优势在于极快的响应速度、优秀的拟人化语音交互、以及极高的性价比（API价格极低）。
*   **当前大模型通用痛点**：复杂推理（System 2 thinking）、长上下文迷失（Lost in the middle）、幻觉问题、多模态深度理解。

**3. 核心解答**
*(注：本题为主观分析题，需结合业界共识从技术和产品两个维度客观作答)*

虽然豆包在响应速度、语音交互体验和日常对话的拟人性上做得非常出色，但在深入使用中，依然暴露出当前大模型普遍存在的一些瓶颈问题：

*   **技术层面（模型底层能力）：**
    1.  **复杂逻辑推理与深度思考能力仍有差距**：在处理长代码工程级生成、复杂的数理逻辑证明，或需要多步严格推导（System 2）的任务时，相比于GPT-4o或Claude 3.5 Sonnet，豆包更容易出现逻辑断裂、遗忘前提条件或陷入死循环。
    2.  **长文本上下文的“中间迷失”**：虽然宣称支持超长上下文窗口，但在进行“大海捞针”式的复杂长文档信息抽取时，对于文档中间部分的细节信息，依然存在遗漏或理解偏差的现象。
    3.  **顽固的幻觉问题**：在面对极其垂直冷门的领域知识，或者要求提供具体的参考文献、数据出处时，依然存在“一本正经胡说八道”的幻觉现象，其内置的联网检索（RAG）对齐度还有待提升。
*   **产品与体验层面：**
    1.  **多模态理解的深度不够**：目前的视觉模型在处理普通图片问答时表现尚可，但在解析复杂的学术架构图、密集的财务报表或带有复杂空间关系的图像时，细粒度OCR和逻辑解析能力较弱。
    2.  **长期记忆与个性化不足**：虽然提供了智能体（Agent）创建功能，但跨越多个周期的长期会话记忆（Long-term Memory）仍不够连贯，无法像人类助手一样随着时间推移精准刻画用户的深度Profile。
    3.  **复杂任务的规划执行（Tool Calling）**：在作为Copilot自主调用多个工具完成复杂任务（如“帮我搜索行业数据，分析后画个图表并生成PPT”）时，任务拆解的准确率和遇到错误时的自我纠正（Self-Correction）能力偏弱，往往需要人类频繁介入。

**4. 引申思考/面试追问**
*   **追问：如果你是豆包的算法工程师，你会如何优化它的复杂推理能力？**
    *   *思路*：1. 在RLHF阶段引入PRM（过程奖励模型），对复杂的推理步骤进行细粒度的步骤级奖励，而不仅仅是结果级奖励；2. 扩大高质量的CoT（思维链）微调数据比例；3. 探索类似OpenAI o1模型的Test-time Compute策略，在推理阶段引入MCTS（蒙特卡洛树搜索）让模型多“思考”一会再输出。

# 题目5：代码：416.分割等和子集

**1. 题目描述**
给定一个只包含正整数的非空数组，判断是否可以从数组中选出一些数字，使得这些数字的和等于整个数组元素和的一半。（LeetCode 416）

**2. 前置基础知识**
*   **0-1 背包问题**：动态规划经典模型。给定背包容量，每个物品只能选择放入或不放入（0或1），求能否恰好装满或价值最大化。
*   **问题转化**：本题要求分割为两个和相等的子集。由于数组总和为 `sum`，问题等价于：**能否从数组中挑选若干个元素，使得它们的和恰好等于 `sum / 2`**。这完美契合了容量为 `sum / 2` 的 0-1 背包问题。

**3. 核心解答**
*   **边界条件判断**：
    1.  计算数组总和 `total_sum`。如果 `total_sum` 是奇数，绝对不可能平分为两个整数子集，直接返回 `false`。
    2.  计算目标和 `target = total_sum / 2`。
    3.  如果数组中最大的元素大于 `target`，则无论怎么分都无法满足，直接返回 `false`。
*   **动态规划状态定义**：
    定义一维布尔数组 `dp[j]`，表示：在遍历当前元素时，是否能够凑出总和恰好为 `j` 的子集。
*   **状态转移方程**：
    对于数组中的当前元素 `num`，针对背包容量 `j`（`j >= num`），我们有两种选择：
    1.  不选 `num`：能否凑出 `j` 取决于之前的状态 `dp[j]`。
    2.  选 `num`：能否凑出 `j` 取决于之前的状态能否凑出 `j - num`，即 `dp[j - num]`。
    综合起来：`dp[j] = dp[j] || dp[j - num]`。
*   **遍历顺序（关键）**：
    外层循环正序遍历数组元素 `num`；**内层循环必须倒序遍历**背包容量 `j`（从 `target` 遍历到 `num`）。因为如果正序遍历，`dp[j - num]` 可能会在当前轮次被提前更新，导致同一个元素被重复使用多次（变成了完全背包问题）。

**代码实现 (Python)**：
```python
def canPartition(nums: list[int]) -> bool:
    total_sum = sum(nums)
    
    # 如果总和是奇数，不能平分
    if total_sum % 2 != 0:
        return False
        
    target = total_sum // 2
    
    # 优化：如果最大值已经大于 target，必定无法平分
    if max(nums) > target:
        return False
    
    # dp[j] 表示是否能凑出和为 j 的子集
    dp = [False] * (target + 1)
    # 凑出和为 0 总是可行的（一个都不选）
    dp[0] = True 
    
    # 外层循环遍历物品（数组元素）
    for num in nums:
        # 内层循环倒序遍历背包容量，防止同一元素被重复使用
        for j in range(target, num - 1, -1):
            dp[j] = dp[j] or dp[j - num]
            
        # 提前剪枝：如果已经能凑出 target，直接返回 True
        if dp[target]:
            return True
            
    return dp[target]
```

**复杂度分析**：
*   **时间复杂度**：$O(n \times target)$，其中 $n$ 是数组长度，$target$ 是数组元素总和的一半。
*   **空间复杂度**：$O(target)$，利用滚动数组优化，仅使用了一维 `dp` 数组。

**4. 引申思考/面试追问**
*   **追问：如果题目改为“将数组分成K个相等的子集”（LeetCode 698），还能用 0-1 背包做吗？**
    *   *思路*：不能直接用 0-1 背包了。这变成了 NP-Hard 问题，通常需要使用 **回溯法（Backtracking） + 状态压缩 DP / 剪枝**。需要维护一个长度为 K 的数组记录每个子集当前的总和，或者使用位运算（Bitmask）表示哪些元素已经被使用，在搜索过程中进行强力剪枝（如排序后从大到小尝试）。
